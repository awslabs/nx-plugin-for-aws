## Task 1: Implement the Game API

We will implement the following APIs in this section:

1. `saveGame` - create or update a game.
2. `queryGames` - return a paginated list of previously saved games.
3. `queryInventory` - return a paginated list of items in a player's inventory.
4. `queryActions` - return the conversation history for a given game.

:::note[Where each piece of state lives]
- `games` and `inventory` are in **DynamoDB** (via ElectroDB) — the Game API lambdas and the Inventory MCP server read/write them.
- `actions` (chat transcript) is in **S3** — the Story Agent's `S3SessionManager` writes one JSON blob per turn, and `queryActions` in the Game API reads them back so the UI can replay history on page reload. There's no `saveAction` procedure; the agent is the only writer.
:::

### Entity modelling

This is the ER diagram for our application.

```d2
direction: right

game: Game {
  shape: sql_table
  playerName: "string  PK"
  genre: string
  lastUpdated: string
}

item: Item {
  shape: sql_table
  playerName: "string  PK"
  itemName: "string  SK"
  emoji: "string  (optional)"
  lastUpdated: string
  quantity: number
}

game -> item: 1..*
```

We will model our data with [ElectroDB](https://electrodb.dev/en/core-concepts/introduction/) on top of the `DungeonDb` table we created with the `ts#dynamodb` generator in <Link path="get_started/tutorials/dungeon-game/1">Module 1</Link>. That generator already added `electrodb` and `@aws-sdk/client-dynamodb` to the workspace, so the only extra dependency we need is the S3 client used by the Game API's `queryActions` procedure to read the conversation turns the agent has written:

:::tip[Runtime Configuration]
The `DungeonDb` construct registers the deployed table name in <Link path="guides/runtime-config">Runtime Configuration</Link> under the `dynamodb` namespace, stored in AWS AppConfig. The generated `resolveTableName()` helper retrieves it at runtime using Lambda Powertools, rather than needing it passed as an environment variable. A benefit of this approach is that as the number of tables or other infrastructure identifiers grows, you only need a single `RUNTIME_CONFIG_APP_ID` environment variable when running against deployed infrastructure.
:::

## Task 4: Test the Game API locally

There's no need to deploy to AWS to try out our API — the `serve-local` target runs the Game API against [DynamoDB Local](https://docs.aws.amazon.com/amazondynamodb/latest/developerguide/DynamoDBLocal.html). Because we connected the Game API to the `DungeonDb` project in Module 1, this target also starts DynamoDB Local automatically.

Then build the codebase:

Select **Yes, sync the changes and run the tasks** to continue. You'll see this prompt whenever project references change.

### Start the local server

Start the Game API locally with the `serve-local` target. Make sure your container engine (Docker or Finch) is running first, since this also boots DynamoDB Local:

<NxCommands commands={["serve-local game-api"]} />

The `serve-local` target sets `SERVE_LOCAL=true`, so `resolveTableName()` returns the local table name and `getDynamoDBClient()` connects to DynamoDB Local instead of AWS — no deployment and no `RUNTIME_CONFIG_APP_ID` required.

<Aside type="tip" title="First run">
The first run pulls the DynamoDB Local container image, which may take a moment. Leave the server running in this terminal and open a second terminal for the `curl` commands below.
</Aside>

### Test the API

Once your server is up and running, query the (empty) list of games:

```bash
curl -X GET 'http://localhost:2022/games.query?input=%7B%7D'
```

:::note[URL Encoding]
The `%7B%7D` we pass to test the API is a URI encoded empty JSON object (`{}`).
:::

You will see an empty list:

```json
{"result":{"data":{"items":[],"cursor":null}}}
```

Now save a game:

```bash
curl -X POST 'http://localhost:2022/games.save' \
  -H 'Content-Type: application/json' \
  -d '{"playerName":"Alice","genre":"zombie"}'
```

The save returns the persisted game (with the `lastUpdated` timestamp the entity sets for you):

```json
{"result":{"data":{"playerName":"Alice","genre":"zombie","lastUpdated":"..."}}}
```

Query again to confirm it's persisted in DynamoDB Local:

```bash
curl -X GET 'http://localhost:2022/games.query?input=%7B%7D'
```

This response now includes the saved game:

```json
{"result":{"data":{"items":[{"playerName":"Alice","genre":"zombie","lastUpdated":"..."}],"cursor":null}}}
```

You can stop the local server (`Ctrl+C`) once you're done.

## Task 5: Test the Inventory MCP server locally

We can try out the MCP server's tools with the [MCP Inspector](https://github.com/modelcontextprotocol/inspector) using the generated `mcp-server-inspect` target. The inspector connects to the server over STDIO, so we run DynamoDB Local separately and tell the server to use it.

In one terminal, start DynamoDB Local:

<NxCommands commands={["serve-local dungeon-db"]} />

In a second terminal, start the inspector:

<NxCommands commands={["mcp-server-inspect inventory"]} />

This starts the MCP Inspector at `http://localhost:6274`. In the inspector UI, before connecting, add an environment variable `SERVE_LOCAL` with the value `true` so the server's tools talk to DynamoDB Local, then click **Connect**.

<Aside type="tip" title="Why set SERVE_LOCAL?">
Unlike `serve-local`, the `mcp-server-inspect` target runs the plain STDIO server, so it doesn't set `SERVE_LOCAL` for you. Setting it in the inspector's environment makes `getDynamoDBClient()` and `resolveTableName()` connect to the local table rather than AWS.
</Aside>

Switch to the **Tools** tab, click **List Tools**, and try `add-to-inventory` (e.g. `playerName: Alice`, `itemName: Rusty Sword`, `emoji: ⚔️`) followed by `list-inventory-items` to see it persisted to DynamoDB Local. Stop both terminals (`Ctrl+C`) when you're done.

<Aside type="tip" title="Prefer the HTTP transport?">
If you'd rather drive the server over HTTP (e.g. for scripting), the `mcp-server-serve-local` target runs the Streamable HTTP transport with `SERVE_LOCAL=true` set for you, and starts DynamoDB Local automatically:

<NxCommands commands={["run @dungeon-adventure/inventory:mcp-server-serve-local"]} />

Then initialize a session against `http://localhost:8000/mcp`:
```bash
curl -X POST http://localhost:8000/mcp \
  -H 'Content-Type: application/json' \
  -H 'Accept: application/json, text/event-stream' \
  -d '{"jsonrpc":"2.0","id":1,"method":"initialize","params":{"protocolVersion":"2025-03-26","capabilities":{},"clientInfo":{"name":"curl","version":"1.0.0"}}}'
```
</Aside>

Congratulations, you have built and tested your first tRPC API and MCP server against a local DynamoDB table!  🎉🎉🎉
