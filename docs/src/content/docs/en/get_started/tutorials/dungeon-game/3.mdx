## Task 1: Implement the Story Agent

The Story Agent is a [Strands](https://strandsagents.com/) agent exposed via the [AG-UI](https://ag-ui.com/) protocol. AgentCore routes each `runtimeSessionId` to a dedicated microVM, so a single long-lived `Agent` per container serves every turn for that session. When AgentCore-managed session storage is attached, the generator-produced `agent.py` persists conversation history to the mounted `SESSION_STORAGE_PATH` via `FileSessionManager`, so an adventure can survive a stop/resume.

We will update the agent's system prompt to play a dungeon adventure, and remove the sample tool.

The system prompt tells the agent that the first message will be of the form `I am <player>, playing in the <genre> genre.` — our UI sends exactly this as the opening user message so the agent has everything it needs to start the story. Because the AG-UI library retains each thread's conversation history automatically, we don't need to thread player name or genre through on subsequent turns.

The connection generator already wired up the Inventory MCP client in <Link path="get_started/tutorials/dungeon-game/1">Module 1</Link>, so the agent has access to the inventory tools.

:::note[Where's `main.py`?]
The AG-UI protocol's FastAPI wrapper handles request/response translation for us. The `main.py` that the generator produced already creates the wrapper around `get_agent(...)` — we don't need to touch it.
:::

### Create a Cognito test user

Before testing the deployed agent (and before we play the game via the UI in the next module), create a Cognito test user. We disabled self-signup earlier, so this step is required regardless of whether you test locally or against the deployed runtime:

```bash
# From the CDK outputs
export POOL_ID="<UserPoolId from CDK outputs>"
export CLIENT_ID="<UserPoolClientId from CDK outputs>"
export REGION="<your-region>"

# Disable MFA to simplify user creation
aws cognito-idp set-user-pool-mfa-config \
  --mfa-configuration OFF \
  --user-pool-id $POOL_ID \
  --region $REGION

# Create a test user
aws cognito-idp admin-create-user \
  --user-pool-id $POOL_ID \
  --username "test" \
  --temporary-password "TempPass123-" \
  --region $REGION \
  --message-action SUPPRESS > /dev/null

# Set a permanent password (replace with something more secure!)
aws cognito-idp admin-set-user-password \
  --user-pool-id $POOL_ID \
  --username "test" \
  --password "PermanentPass123-" \
  --region $REGION \
  --permanent > /dev/null
```

### Test your Agent

You can test your Agent by either:
<ul>
<li>Starting a local instance of the Agent server and chatting with it interactively using the `agent-chat` target, or</li>
<li>Calling the deployed API using curl with a JWT token.</li>
</ul>

<Tabs>
  <TabItem label="Local">
  The `agent-chat` target starts the agent locally (together with its dependencies, via `agent-serve-local`) and drops you into an interactive chat prompt. Run:

    <NxCommands commands={["agent-chat story"]} />

    :::note[`agent-serve` vs `agent-serve-local`]
    If you want to chat with the agent against your **deployed** MCP server instead of a local one, start the server directly with `agent-serve` and pass `RUNTIME_CONFIG_APP_ID` / `AWS_REGION`:

    <NxCommands highlights={['xxxx', 'region']} env={{RUNTIME_CONFIG_APP_ID:"xxxx", AWS_REGION: '<region>'}} commands={["agent-serve story"]} />

    then in another shell run `agent-chat-cli agui http://localhost:8081/invocations`.
    :::

    When prompted, try starting the game with something like:

    ```
    I am Aria, playing in the medieval genre.
    ```
  </TabItem>
  <TabItem label="Deployed">
    Exchange the test user's credentials for a Cognito access token:

    ```bash
    # Authenticate the test user and capture their access token
    export BEARER_TOKEN=$(aws cognito-idp initiate-auth \
      --client-id "$CLIENT_ID" \
      --auth-flow USER_PASSWORD_AUTH \
      --auth-parameters USERNAME='test',PASSWORD='PermanentPass123-' \
      --region $REGION \
      --query "AuthenticationResult.AccessToken" \
      --output text)
    ```

    Invoke the deployed agent using the Bedrock AgentCore runtime URL. The AG-UI protocol expects a `RunAgentInput` payload — a minimal one that sends the opening user message looks like:

    ```bash
    # Set the Story Agent ARN from CDK outputs
    export AGENT_ARN="<StoryAgentArn from CDK outputs>"

    # URL-encode the ARN
    export ENCODED_ARN=$(echo $AGENT_ARN | sed 's/:/%3A/g' | sed 's/\//%2F/g')

    # Construct the invocation URL
    export AGENT_URL="https://bedrock-agentcore.$REGION.amazonaws.com/runtimes/$ENCODED_ARN/invocations?qualifier=DEFAULT"

    # Invoke the agent
    curl -N -X POST "$AGENT_URL" \
      -H "authorization: Bearer $BEARER_TOKEN" \
      -H "Content-Type: application/json" \
      -H "Accept: text/event-stream" \
      -H "X-Amzn-Bedrock-AgentCore-Runtime-Session-Id: abcdefghijklmnopqrstuvwxyz-123456789" \
      -d '{
        "threadId": "abcdefghijklmnopqrstuvwxyz-123456789",
        "runId": "run-1",
        "messages": [{"id": "m1", "role": "user", "content": "I am UnnamedHero, playing in the superhero genre."}],
        "tools": [], "context": [], "forwardedProps": {}, "state": {}
      }'
    ```

    :::note[URL Encoding]
    The URL encoding is necessary to properly format the ARN for the Bedrock AgentCore API endpoint.
    :::
  </TabItem>
</Tabs>

If the command runs successfully, you should start to see AG-UI Server-Sent Events streaming back — `TEXT_MESSAGE_CONTENT` events carry the story text chunk-by-chunk.

Congratulations. You have built and deployed your first Strands Agent on Bedrock AgentCore Runtime!  🎉🎉🎉
