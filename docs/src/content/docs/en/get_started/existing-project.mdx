Not every project starts with `pnpm create @aws/nx-workspace`. If you already have an [Nx workspace](https://nx.dev) — or a monorepo you can add Nx to — you can adopt `@aws/nx-plugin` incrementally without recreating your project.

<Aside type="caution" title="Before you start">
Save and commit all in-progress work first, so the changes made at each step can be reviewed — and reverted — at any point.
</Aside>

## Prerequisites

<Snippet name="prerequisites" />

You'll also need **an Nx workspace**. If your project isn't an Nx workspace yet, add Nx first — see [Adding Nx to a non-Nx project](#adding-nx-to-a-non-nx-project) below. If you already use Nx, skip straight to [Adding the plugin](#adding-the-plugin).

## Adding Nx to a non-Nx project

If your project doesn't use Nx yet, add it first with [`nx init`](https://nx.dev/recipes/adopting-nx/adding-to-existing-project). This is a step Nx itself owns; the plugin builds on top of a working Nx workspace. It works on a plain package, an npm/pnpm/yarn workspace, a Turborepo or a Lerna monorepo:

<NxInitCommand />

`nx init` is Nx's own tool for adopting Nx — it detects your tooling, writes `nx.json`, and wires your existing scripts into Nx. Follow the prompts. When running non-interactively (CI or an AI agent), pass `--interactive=false`; if Nx detects tool plugins for your stack (e.g. `@nx/vite`) it will also ask for an explicit choice — pass `--plugins=skip` to defer (plugins can be added later with `nx add`) or `--plugins=all` to accept them. Once it completes and your workspace builds under Nx, [add the plugin](#adding-the-plugin) as below.

### Non-Node projects (Python, Go, Java, Rust, …)

Nx and the plugin are distributed as npm packages, so a project with no Node.js tooling needs a minimal root `package.json` before `nx init` can do anything useful:

```json
{
  "name": "my-project",
  "private": true,
  "type": "module"
}
```

Create that file, then run `nx init` and [add the plugin](#adding-the-plugin) as usual. Your existing language tooling is untouched — Nx projects generated by the plugin (including <Link path="guides/py-project">Python projects</Link>) live alongside your existing code, and you can wire your existing build into Nx incrementally.

### Single-package projects

The plugin is designed for monorepos — it generates each project into its own directory under `packages/`. If you're starting from a **single-package project** (one `package.json` at the root with your source directly beneath it, no workspaces), move your existing package into `packages/` before adopting the plugin so your project sits alongside the ones the plugin generates:

<Steps>

1. Create a `packages/<your-package>/` directory and move your source, `package.json` and `tsconfig.json` into it.
2. Create a new root `package.json` that acts as the workspace manifest rather than a project itself, for example `{ "name": "<your-workspace>", "private": true }`.
3. Declare the workspace: for `pnpm`, add a `pnpm-workspace.yaml` with `packages: ['packages/*']`; for `npm`/`yarn`/`bun`, add `"workspaces": ["packages/*"]` to the new root `package.json`.
4. Run `nx init` (if you haven't), then [add the plugin](#adding-the-plugin).

</Steps>

Keeping the root as a workspace manifest (rather than a project) is what lets `nx run-many --target build --all` build every project without a task referring to itself — see [the recursion note](#recursive-task-invocation-detected) below.

<Aside type="note" title="Package manager & layout">
The plugin's projects are generated under a `packages/` directory. `pnpm` reads workspace globs from `pnpm-workspace.yaml`; other package managers read the `workspaces` field of the root `package.json` — `init` updates whichever your workspace uses. If your workspace uses a different layout (such as `apps/` and `libs/`), the plugin's projects live under `packages/` alongside your existing structure.
</Aside>

## Adding the plugin

Use [`nx add`](https://nx.dev/nx-api/nx/documents/add), which installs the plugin at a version compatible with your Nx installation and then runs its `init` generator to configure your workspace:

<NxCommands commands={['add @aws/nx-plugin']} />

Once it completes, your workspace is ready — choose the generators you need from the <Link path="guides/index">guides</Link> and start generating projects.

### What the `init` generator configures

The `init` generator makes the deterministic changes a workspace needs to run the plugin's generators. It is **idempotent** — running it again is a no-op — and it preserves your workspace's module format: a workspace whose root `package.json` has `type: "module"` stays ESM, anything else stays CommonJS, and generated code follows suit. It does **not** rewrite an existing `tsconfig.base.json`, migrate your package manager, or restructure your existing projects.

It performs the following:

- Creates `aws-nx-plugin.config.mts` recording your chosen <Link path="guides/typescript-infrastructure">infrastructure as code provider</Link> (CDK or Terraform) and container engine. Generators read `iac.provider` from this file.
- Registers the sync generators (`@nx/js:typescript-sync` and `@aws/nx-plugin:ts#sync`) on the `compile` target in `nx.json`, so TypeScript project references stay in sync as you add projects.
- Ensures a root `tsconfig.json` and a `tsconfig.base.json` exist (Nx's TypeScript sync and the plugin's projects need them). Missing files are created; **existing ones are left untouched** so they don't disturb your other projects. A created `tsconfig.base.json` carries the compiler options the plugin's TypeScript projects rely on — if you keep your own, these are the options to align with:

  ```json
  {
    "compilerOptions": {
      "composite": true,
      "declarationMap": true,
      "emitDeclarationOnly": true,
      "importHelpers": true,
      "isolatedModules": true,
      "lib": ["es2022"],
      "module": "nodenext",
      "moduleResolution": "nodenext",
      "noEmitOnError": true,
      "noFallthroughCasesInSwitch": true,
      "noImplicitOverride": true,
      "noImplicitReturns": true,
      "noUnusedLocals": true,
      "skipLibCheck": true,
      "strict": true,
      "target": "es2022"
    }
  }
  ```

- For `pnpm` workspaces, allow-lists the build scripts the plugin's dependencies need (`@swc/core`, `esbuild`, `nx`, `sharp`) in `pnpm-workspace.yaml`.
- Adds convenience scripts (without overwriting any of yours) to the root `package.json`, plus the `nx`, `@nx/js`, `@nx/workspace`, `typescript` and Biome dev dependencies — `nx` pinned to a single version to avoid the [duplicate-nx deadlock](#nx-sync-hangs-or-plugin-worker--exited-before-the-connection-was-established) — and writes a default `biome.json` if none exists.
- Configures the <Link path="get_started/building-with-ai">MCP server</Link> for supported coding agents (unless disabled).

## Incremental adoption

You don't have to convert your whole workspace at once. A good path is:

<Steps>

1. Add the plugin, then generate a single project (for example a <Link path="guides/typescript-project">TypeScript project</Link> or an <Link path="guides/ts-api">API</Link>) and build it.
2. Add further projects as you need them — websites, APIs, infrastructure — using the <Link path="guides/index">generator guides</Link>.
3. Migrate your existing code into the workspace over time, if desired. The plugin's projects sit alongside your existing ones; they don't require you to move anything.

</Steps>

## Troubleshooting

#### `The workspace is out of sync`

TypeScript project references need syncing. `init` registers the sync generators; run:

<NxCommands commands={['sync']} />

#### `ERR_PNPM_IGNORED_BUILDS` during install

pnpm skips install scripts for packages that aren't allow-listed. `init` allow-lists the ones the plugin's tooling needs (`@swc/core`, `esbuild`, `nx`, `sharp`) in `pnpm-workspace.yaml`, but the block can fire during `nx add @aws/nx-plugin` itself — before `init` has run. Run `pnpm approve-builds` and approve the listed packages (or add them under `allowBuilds:`), then re-run the install.

#### `nx sync` hangs, or `plugin worker ... exited before the connection was established`

If two different `nx` versions are present in the same node_modules tree (run `npm ls nx` to confirm), their plugin-worker IPC deadlocks. The `init` generator pins the root `nx` devDependency to the version the plugin's own `@nx/*` packages resolve, but a later install of another `@nx/*` package at a different patch version can reintroduce the mismatch. Align every `nx` and `@nx/*` entry in your root `package.json` to a single version and reinstall.

#### `Recursive task invocation detected`

Your **root** `package.json` is registered as an Nx project (it has an `"nx"` key, which `nx init` adds for a single-package repo) *and* carries a `build` script of `nx run-many --target build`. Nx then infers a `build` target on the root that calls itself. Move your source into `packages/<your-package>/` so the root becomes a workspace manifest rather than a project — see [Single-package projects](#single-package-projects). (Nx's own standalone presets avoid this by setting `"nx": { "includedScripts": [] }` on the root package; you can do the same as a quicker alternative.)

#### TypeScript errors in your **existing** projects after adding the plugin (`TS6059`, `TS7016`, `TS5011`, `NG4006`)

The plugin's own generated projects carry the TypeScript settings they need in their per-project `tsconfig.lib.json`, so they build regardless of your `tsconfig.base.json`. These errors show up instead in a **pre-existing project of yours** that inherits a `tsconfig.base.json` `init` created (with `composite`/`emitDeclarationOnly`/`nodenext`) but isn't compatible with those settings — for example, the Angular compiler rejects `emitDeclarationOnly` (`NG4006`), or a project that sets `outDir` without a `rootDir` now needs one (`TS5011`). Add per-project `tsconfig` overrides re-declaring the conflicting options for those projects (e.g. `"rootDir": "src"`), or point the plugin's projects and your projects at separate base configs.

`init` never rewrites an existing `tsconfig.base.json`, precisely so it can't silently break the projects that inherit from it — resolving these mismatches is a decision only you can make for your codebase.

<Aside type="note" title="ESM and CommonJS">
The module format is workspace-wide, read from the root `package.json` `type` field: `type: "module"` means ESM, anything else (including no `type` field) means CommonJS. `init` preserves whichever your workspace uses, and the plugin's TypeScript generators emit matching code. If you later want to switch, change the root `type` — but that's a migration of your existing code, not something `init` does for you.
</Aside>
