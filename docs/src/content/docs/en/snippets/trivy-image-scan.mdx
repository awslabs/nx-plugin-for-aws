---
title: Container Image Scanning
---

The Docker image built for this project is scanned for vulnerabilities as part of the build using [Trivy](https://trivy.dev/), running from the [ECR-hosted Trivy image](https://gallery.ecr.aws/aquasecurity/trivy).

The scan runs after the image is built and **fails the build** if any fixable `HIGH` or `CRITICAL` severity vulnerability is found. Vulnerabilities without an available fix are ignored so that unactionable base image advisories do not block your build.

The scan uses the same container engine as your image build (`docker` or `finch`), so no additional tooling is required.

:::caution[Green Base Images]
The generated `Dockerfile` uses a base image with no known `HIGH` or `CRITICAL` vulnerabilities at time of generation, and upgrades bundled tooling (such as `npm`) to keep it that way. New vulnerabilities are disclosed over time, so a build that passes today may fail later as the vulnerability database is updated. When this happens, update to a newer base image where the vulnerability is fixed, or suppress the finding as described below.
:::

#### Suppressing Trivy Findings

There may be instances where you want to suppress a specific vulnerability, for example when no fix is yet available and you have assessed the risk as acceptable.

Add the vulnerability ID (one per line) to the `.trivyignore` file at the root of your project:

```text
// .trivyignore
# node-tar arbitrary file write - not exploitable in our usage
CVE-2024-XXXXX
```

For more details on filtering findings, refer to the [Trivy filtering documentation](https://trivy.dev/latest/docs/configuration/filtering/#by-finding-ids).
