This generator creates a new relational database project backed by [Amazon Aurora](https://aws.amazon.com/rds/aurora/) PostgreSQL. It generates the application code and infrastructure needed to provision and manage a database using AWS CDK or Terraform.

The generated infrastructure includes:

- Reusable Aurora construct or module
- Application-specific database construct or module  
- IAM-authenticated application database user handler
- Runtime configuration registration in AWS AppConfig
- Migration handler for automatic Prisma migration deployment

## Implementing your Database

### Data Modelling

The generated project uses [Prisma ORM](https://www.prisma.io/docs/orm) to define your database schema and generate a type-safe client. Model your tables, relations, and field types in `prisma/schema.prisma`.

Example `User` model:

```ts title="packages/postgres/prisma/schema.prisma"
model User {
  id          Int      @id @default(autoincrement()) 
  firstName   String
  lastName    String
}
```

For more details, see the official [Prisma data modelling guide](https://www.prisma.io/docs/orm/core-concepts/data-modeling#data-modeling-with-prisma-orm).

### Generating the Database Client

The generator automatically configures the `generate` target to create a type-safe TypeScript Prisma client whenever you build the project. The client is written to `generated/prisma` (added to `.gitignore`).

You can also manually generate the client at any time:

  </TabItem>

The runtime wrapper in `src/prisma.ts` exports:

- `DB_PACKAGE_NAME` - the key used under the `database` runtime config namespace in AWS AppConfig
- `getPrisma()` - loads database connection settings from AWS AppConfig and creates a Prisma client using IAM authentication

The client automatically:
- Retrieves database configuration from AWS AppConfig using `RUNTIME_CONFIG_APP_ID` environment variable
- Generates temporary authentication tokens via AWS RDS Signer for IAM authentication
- Manages SSL/TLS connections with certificate validation
- Handles connection pooling through persistent database connection pools

Example usage in a tRPC API:

```ts title="packages/api/src/procedures/list-user.ts"
import { getPrisma } from ':aws-nx-poc/postgres';
import { publicProcedure } from '../init.js';
import { ListUserOutputSchema } from '../schema/index.js';

export const listUser = publicProcedure
  .output(ListUserOutputSchema)
  .query(async () => {
    const prisma = await getPrisma();
    const users = await prisma.user.findMany({
      orderBy: { id: 'asc' },
    });

    return { users };
  });
```

### Creating Migrations

:::note[Run Prisma from the Package Directory]
When using the Prisma CLI, run commands from your generated database package directory (e.g., `packages/postgres`) so Prisma can find `prisma/schema.prisma` and `prisma.config.ts`.
:::

Prisma Migrate requires a [shadow database](https://www.prisma.io/docs/orm/prisma-migrate/understanding-prisma-migrate/shadow-database) to compare your schema with the current database state and generate SQL changes.

To generate migrations locally, point Prisma at a development database by exporting `DATABASE_URL`:

```bash
export DATABASE_URL="postgresql://username:password@hostname:5432/dbname"
```

## Deploying to AWS

The relational database generator creates CDK or Terraform infrastructure based on your selected `iacProvider`.

<Infrastructure>
<Fragment slot="cdk">
The CDK construct is created in `common/constructs`. Example usage:

```ts title="packages/infra/src/stacks/application-stack.ts"
import { Postgres } from ':my-scope/common-constructs';

export class ApplicationStack extends Stack {
  constructor(scope: Construct, id: string, props?: StackProps) {
    super(scope, id, props);

    const vpc = new Vpc(this, 'Vpc', {
      subnetConfiguration: [
        {
          name: 'public',
          subnetType: SubnetType.PUBLIC,
        },
        {
          name: 'private_with_egress',
          subnetType: SubnetType.PRIVATE_WITH_EGRESS,
        },
        {
          name: 'private_isolated',
          subnetType: SubnetType.PRIVATE_ISOLATED,
        },
      ],
    });

    const db = new Postgres(this, 'Db', {
      vpc,
      vpcSubnets: {
        subnetType: SubnetType.PRIVATE_ISOLATED,
      }
    });
  }
}
```

This provisions an Aurora cluster with RDS Proxy, admin credentials, application database user, runtime config registration, and migration handler. Deploy into private isolated subnets to prevent direct internet access.

The generated infrastructure creates two database users:
- **Admin user** - Created during cluster provisioning with credentials stored in AWS Secrets Manager
- **Application user** - Created via a Lambda custom resource with IAM authentication enabled and full privileges on the application database
</Fragment>
<Fragment slot="terraform">
The Terraform module is created in `common/terraform`. Example usage:

```hcl title="packages/infra/src/main.tf"
module "postgres" {
  source = "../../common/terraform/src/app/dbs/postgres"

  vpc_id     = module.vpc.vpc_id
  subnet_ids = module.vpc.private_isolated_subnet_ids

  tags = local.common_tags
}
```

This provisions an Aurora cluster with RDS Proxy, admin credentials, create-db-user Lambda, runtime config registration, migration Lambda, and container registry resources.

The generated infrastructure creates two database users:
- **Admin user** - Created during cluster provisioning with credentials stored in AWS Secrets Manager
- **Application user** - Created via a Lambda function with IAM authentication enabled and full privileges on the application database
</Fragment>
</Infrastructure>

### Connecting Lambda Functions

To use the Prisma client from Lambda functions, ensure the following requirements are met:

1. Deploy functions in the same VPC as the database
2. Allow network access to the database port
3. Grant IAM `rds-db:connect` permissions for the generated runtime user
4. Set required environment variables:
   - `NODE_EXTRA_CA_CERTS=/var/runtime/ca-cert.pem` (required for TLS validation)
   - `RUNTIME_CONFIG_APP_ID` (set automatically by the generated infrastructure)
   - `AWS_REGION` (set automatically in Lambda execution environment)

:::note[Prisma CLI vs Runtime Client]
**Prisma CLI** commands (`prisma migrate dev`, `prisma migrate deploy`) read `DATABASE_URL` from environment variables.

**Runtime client** (`src/prisma.ts`) loads connection settings from AWS AppConfig and uses IAM authentication.
:::

<Infrastructure>
<Fragment slot="cdk">

```ts title="packages/infra/src/stacks/application-stack.ts"
import { RuntimeConfig } from ':my-scope/common-constructs';

const runtimeConfig = RuntimeConfig.ensure(this);

const api = new Api(this, 'Api', {
  integrations: Api.defaultIntegrations(this)
    .withDefaultOptions({
      vpc,
      vpcSubnets: {
        subnetType: SubnetType.PRIVATE_WITH_EGRESS,
      },
      environment: {
        NODE_EXTRA_CA_CERTS: '/var/runtime/ca-cert.pem',
      },
    })
    .build(),
});

Object.entries(api.integrations).forEach(([operation, integration]) => {
  db.allowDefaultPortFrom(
    integration.handler,
    `Allow ${operation} to connect to the database`,
  );
  db.grantConnect(integration.handler);
});
```

This configures each API Lambda function to run in the same VPC, read runtime config from AppConfig, and connect to the database using IAM authentication.
</Fragment>
<Fragment slot="terraform">
When connecting Lambda functions to the database, use these module outputs:

- `appconfig_application_id` - expose to compute environment
- `cluster_port` - allow network access
- `database_runtime_user` - grant `rds-db:connect` permission

Ensure compute runs in the same VPC and private subnets, and grant AppConfig read permissions to the compute role.
</Fragment>
</Infrastructure>

### RDS Proxy

The generated infrastructure includes an RDS Proxy by default, which sits between your application and the Aurora cluster. RDS Proxy provides several benefits:

- **Connection pooling** - Maintains a pool of database connections that can be shared across application instances, reducing the overhead of establishing new connections
- **Connection resilience** - Automatically handles failovers and reconnects during Aurora instance replacements or maintenance
- **IAM authentication** - Supports IAM-based database authentication, eliminating the need to manage database credentials in your application code
- **Improved security** - Enforces TLS encryption for all connections

<Infrastructure>
<Fragment slot="cdk">

By default, RDS Proxy is enabled. The generated runtime client (`getPrisma()`) automatically connects through the proxy endpoint. You can disable it if needed:

```ts title="packages/infra/src/stacks/application-stack.ts"
const db = new Postgres(this, 'Db', {
  ...
  enableRdsProxy: false,
});
```

When RDS Proxy is disabled, your application connects directly to the Aurora cluster endpoint.
</Fragment>
<Fragment slot="terraform">

By default, RDS Proxy is enabled. The generated runtime client (`getPrisma()`) automatically connects through the proxy endpoint. You can disable it if needed:

```hcl title="packages/infra/src/main.tf"
module "postgres" {
  source = "../../common/terraform/src/app/dbs/postgres"
  ...
  enable_rds_proxy = false
}
```

When RDS Proxy is disabled, your application connects directly to the Aurora cluster endpoint. The proxy is configured to require TLS encryption and use IAM authentication.
</Fragment>
</Infrastructure>

## Customising Database Architecture

The generated infrastructure can be customised to match your workload requirements. The following examples demonstrate a few common customisation options available.

### Cluster Instances

Configure the writer and reader instances for your Aurora cluster.

<Infrastructure>
<Fragment slot="cdk">

```ts title="packages/infra/src/stacks/application-stack.ts"
const db = new Postgres(this, 'Db', {
  ...
  writer: ClusterInstance.serverlessV2('writer'),
  readers: [ClusterInstance.serverlessV2('reader')],
});
```
</Fragment>
<Fragment slot="terraform">

```hcl title="packages/infra/src/main.tf"
module "postgres" {
  source = "../../common/terraform/src/app/dbs/postgres"
  ...
  instance_count = 2  # 1 writer + 1 reader
}
```
</Fragment>
</Infrastructure>
