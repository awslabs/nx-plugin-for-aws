This generator creates a new relational database project backed by [Amazon Aurora](https://aws.amazon.com/rds/aurora/). It generates the application code and infrastructure needed to provision a database using AWS CDK or Terraform, with support for Aurora PostgreSQL and Aurora MySQL.

The generated infrastructure includes a reusable Aurora construct or module, an application-specific database construct or module, a handler that creates an IAM-authenticated application database user, runtime configuration registration in AWS AppConfig, and a migration handler that applies your Prisma migrations during deployment.

## Implementing your Database

### Data Modelling

The generated project uses [Prisma ORM](https://www.prisma.io/docs/orm) for defining your database schema and generating a type-safe client. You can model your tables, relations and field types in `prisma/schema.prisma`, then use Prisma to generate the client and create migrations for your database.

For more details on how to define models with Prisma, refer to the official guide on [data modeling with Prisma ORM](https://www.prisma.io/docs/orm/core-concepts/data-modeling#data-modeling-with-prisma-orm).

As an example, a `User` model would be represented as follows in the Prisma schema:

```ts title="packages/postgres/prisma/schema.prisma"
model User {
  id          Int      @id @default(autoincrement()) 
  firstName   String
  lastName    String
}
```

### Data Migration

When generating Prisma migration files, Prisma Migrate requires access to a shadow database. This is used during migration generation to compare your Prisma schema with the current state of the database and determine the SQL changes that need to be created.

For more details on why this is required and how Prisma uses it, refer to the Prisma documentation on [shadow databases](https://www.prisma.io/docs/orm/prisma-migrate/understanding-prisma-migrate/shadow-database).

:::note[Run Prisma from the Package Directory]
When using the Prisma CLI, run the commands from your generated database package directory, for example `packages/postgres`, so Prisma can find `prisma/schema.prisma` and `prisma.config.ts`.
:::

To generate migrations locally, you can point Prisma at a development database instance. First export `DATABASE_URL` so Prisma can connect to your database:

```bash
export DATABASE_URL="postgresql://username:password@hostname:5432/dbname"
```

Then create the migration files without applying them:


  </TabItem>
  <TabItem label="pnpm" icon="seti:pnpm">

Running `prisma migrate dev --create-only` will keep the `prisma/migrations` directory up to date by generating a new migration folder each time your schema changes require a new migration.

For deployments to AWS, you do not need to run `prisma migrate deploy` manually. The generated infrastructure creates the application database user first, then runs your migrations automatically during deployment, as long as the `prisma/migrations` directory is up to date.
:::


### Generating Database Client

The generator automatically configures the `generate` target to generate the type-safe TypeScript Prisma client whenever you build the project.

The generated client is structured as follows:

<FileTree>
  - generated
    - prisma
      - internal
      - models
      - browser.ts
      - client.ts
      - commonInputTypes.ts
      - enums.ts
      - models.ts
</FileTree>

The generated Prisma client is written to `generated/prisma`, and that folder is added to `.gitignore` in generated projects.

The runtime wrapper lives in `src/prisma.ts`. It exports:

- `DB_PACKAGE_NAME`, the key used under the `database` runtime config namespace in AWS AppConfig
- `getPrisma()`, which loads the database connection settings from AWS AppConfig and creates a Prisma client using IAM authentication

You can then use `getPrisma()` in other projects in your workspace, such as a `tRPC` API:

```ts title="packages/api/src/procedures/list-user.ts"
import { getPrisma } from ':aws-nx-poc/postgres';
import { publicProcedure } from '../init.js';
import {
  ListUserOutputSchema,
} from '../schema/index.js';

export const listUser = publicProcedure
  .output(ListUserOutputSchema)
  .query(async () => {
    const prisma = await getPrisma();
    const users = await prisma.user.findMany({
      orderBy: { id: 'asc' },
    });

    return { users };
  });
```

## Deploying your Database

### Infrastructure

The relational database generator creates CDK or Terraform infrastructure as code based on your selected `iacProvider`. You can use this to deploy your database.

<Infrastructure>
<Fragment slot="cdk">
The CDK construct for deploying your database is created in the `common/constructs` folder. You can consume this in a CDK application, for example:

```ts title="packages/infra/src/stacks/application-stack.ts"
import { Postgres } from ':my-scope/common-constructs';

export class ApplicationStack extends Stack {
  constructor(scope: Construct, id: string, props?: StackProps) {
    super(scope, id, props);

    const vpc = new Vpc(this, 'Vpc', {
      subnetConfiguration: [
        {
          name: 'public',
          subnetType: SubnetType.PUBLIC,
        },
        {
          name: 'private_with_egress',
          subnetType: SubnetType.PRIVATE_WITH_EGRESS,
        },
        {
          name: 'private_isolated',
          subnetType: SubnetType.PRIVATE_ISOLATED,
        },
      ],
    });

    const db = new Postgres(this, 'Db', {
      vpc,
      vpcSubnets: {
        subnetType: SubnetType.PRIVATE_ISOLATED,
      }
    });
  }
}
```

This provisions an Aurora cluster and an RDS Proxy in your VPC, along with the generated admin credentials secret, the application database user custom resource, runtime config registration, and the migration handler. The database should typically be deployed into private isolated subnets so that it is not directly accessible from the public internet.

If you need to customise the cluster topology, you can override Aurora cluster properties such as `writer` and `readers` when instantiating the construct.
</Fragment>
<Fragment slot="terraform">
The Terraform module for deploying your database is created in the `common/terraform` folder. You can use this in a Terraform configuration:

```hcl title="packages/infra/src/main.tf"
module "postgres" {
  source = "../../common/terraform/src/app/dbs/postgres"

  vpc_id     = module.vpc.vpc_id
  subnet_ids = module.vpc.private_isolated_subnet_ids

  tags = local.common_tags
}
```

This provisions an Aurora cluster with RDS Proxy, the generated admin credentials secret, the create-db-user Lambda function, runtime config registration in AWS AppConfig, the migration Lambda function, and the supporting resources needed to publish and run the migration container image.

If you need to customise the cluster topology or sizing, you can override module inputs such as `instance_count`, `serverless_min_capacity`, `serverless_max_capacity`, `engine_version` and `port`.
</Fragment>
</Infrastructure>

### Granting Access

To allow Lambda-based application code to use the generated Prisma client in AWS, deploy it into the same VPC as the database, allow network access to the database port, and grant IAM database connect permissions for the generated runtime user.

:::note[Prisma CLI vs Runtime Client]
Prisma CLI commands such as `prisma migrate dev` and `prisma migrate deploy` read `DATABASE_URL` from the environment variables.

The generated runtime client in `src/prisma.ts` does not read `DATABASE_URL`. It loads `database[DB_PACKAGE_NAME]` from AWS AppConfig and connects to Aurora using IAM authentication.
:::

<Infrastructure>
<Fragment slot="cdk">

```ts title="packages/infra/src/stacks/application-stack.ts"
import { RuntimeConfig } from ':my-scope/common-constructs';

const runtimeConfig = RuntimeConfig.ensure(this);

const api = new Api(this, 'Api', {
  integrations: Api.defaultIntegrations(this)
    .withDefaultOptions({
      vpc,
      vpcSubnets: {
        subnetType: SubnetType.PRIVATE_WITH_EGRESS,
      },
      environment: {
        NODE_EXTRA_CA_CERTS: '/var/runtime/ca-cert.pem',
      },
    })
    .build(),
});

Object.entries(api.integrations).forEach(([operation, integration]) => {
  db.allowDefaultPortFrom(
    integration.handler,
    `Allow ${operation} to connect to the database`,
  );
  db.grantConnect(integration.handler);
});
```

This ensures that each API Lambda function runs inside the same VPC as the database, can read the runtime configuration from AWS AppConfig, and is explicitly allowed to connect to the database using the generated IAM database user.
</Fragment>
<Fragment slot="terraform">
The generated Terraform database module exposes `appconfig_application_id`, `cluster_port` and `database_runtime_user` for custom consumers.

When wiring your own Terraform-managed Lambda or API to the generated database:

- run the compute in the same VPC and private subnets as the database
- expose the AppConfig application ID from `module.postgres.appconfig_application_id` to your compute environment
- grant AppConfig read permissions to the compute role
- grant `rds-db:connect` for the generated runtime user `module.postgres.database_runtime_user`
- allow network access to the Aurora port exposed as `module.postgres.cluster_port`

The `rds-db:connect` resource ARN also needs the Aurora cluster resource ID, so expose that from your Terraform modules when custom compute needs direct database access.
</Fragment>
</Infrastructure>

### SSL/TLS requirements

When running Lambda functions against Aurora, you must configure the runtime to loads additional CA certificates. In CDK, that can be done by adding:

```ts
const api = new Api(this, 'Api', {
  integrations: Api.defaultIntegrations(this)
    .withDefaultOptions({
      ...
      environment: {
        NODE_EXTRA_CA_CERTS: '/var/runtime/ca-cert.pem',
      },
    })
    .build(),
});
```

This is required for Lambda to validate TLS connections to Aurora RDS. For more details, see the AWS Lambda RDS documentation: https://docs.aws.amazon.com/lambda/latest/dg/services-rds.html

## Customising your Database Architecture

The generated database construct exposes the underlying Aurora configuration so you can adapt it to your workload.

### Overriding the Writer

<Infrastructure>
<Fragment slot="cdk">
In CDK, you can override the default writer instance if you want to change how the primary Aurora instance is provisioned:

```ts title="packages/infra/src/stacks/application-stack.ts"
const db = new Postgres(this, 'Db', {
  vpc,
  vpcSubnets: {
    subnetType: SubnetType.PRIVATE_ISOLATED,
  },
  writer: ClusterInstance.serverlessV2('writer'),
});
```
</Fragment>
<Fragment slot="terraform">
In Terraform, Aurora instances are managed through module inputs rather than separate `writer` and `readers` definitions. The generated module always provisions the primary instance for you, and you customise the cluster around it using inputs such as `instance_count`, `engine_version` and the serverless capacity settings:

```hcl title="packages/infra/src/main.tf"
module "postgres" {
  source = "../../common/terraform/src/app/dbs/postgres"

  vpc_id                  = module.vpc.vpc_id
  subnet_ids              = module.vpc.private_isolated_subnet_ids
  instance_count          = 1
  serverless_min_capacity = 0.5
  serverless_max_capacity = 8

  tags = local.common_tags
}
```
</Fragment>
</Infrastructure>

### Adding Readers

<Infrastructure>
<Fragment slot="cdk">
If your workload needs read scaling, you can add reader instances in CDK using the `readers` property:

```ts title="packages/infra/src/stacks/application-stack.ts"
const db = new Postgres(this, 'Db', {
  vpc,
  vpcSubnets: {
    subnetType: SubnetType.PRIVATE_ISOLATED,
  },
  readers: [ClusterInstance.serverlessV2('reader')],
});
```
</Fragment>
<Fragment slot="terraform">
In Terraform, the equivalent control is `instance_count`, which determines how many Aurora instances are created:

```hcl title="packages/infra/src/main.tf"
module "postgres" {
  source = "../../common/terraform/src/app/dbs/postgres"

  vpc_id         = module.vpc.vpc_id
  subnet_ids     = module.vpc.private_isolated_subnet_ids
  instance_count = 2

  tags = local.common_tags
}
```
</Fragment>
</Infrastructure>

### Pinning the Engine Version

If you want to use a specific Aurora engine version, you can pin it explicitly rather than relying on the default.

<Infrastructure>
<Fragment slot="cdk">

```ts title="packages/infra/src/stacks/application-stack.ts"
const db = new Postgres(this, 'Db', {
  vpc,
  vpcSubnets: {
    subnetType: SubnetType.PRIVATE_ISOLATED,
  },
  engineVersion: AuroraPostgresEngineVersion.VER_17_7,
});
```
</Fragment>
<Fragment slot="terraform">
In Terraform, use `engine_version`:

```hcl title="packages/infra/src/main.tf"
module "postgres" {
```

</Infrastructure>

### Selecting Subnets and Lifecycle Options

You can also customise where the database is deployed and how it behaves during deletion.

<Infrastructure>
<Fragment slot="cdk">
Use `vpcSubnets` to control which subnets the database is placed into, and lifecycle-related properties such as `deletionProtection` and `removalPolicy` to control deletion behaviour:

```ts title="packages/infra/src/stacks/application-stack.ts"
import { RemovalPolicy } from 'aws-cdk-lib';

const db = new Postgres(this, 'Db', {
  vpc,
  vpcSubnets: {
    subnetType: SubnetType.PRIVATE_ISOLATED,
  },
  deletionProtection: true,
  removalPolicy: RemovalPolicy.SNAPSHOT,
});
```
</Fragment>
<Fragment slot="terraform">
Use `subnet_ids` to control where the database and migration Lambda are deployed.

```hcl title="packages/infra/src/main.tf"
module "postgres" {
  source = "../../common/terraform/src/app/dbs/postgres"

  vpc_id              = module.vpc.vpc_id
  subnet_ids          = module.vpc.private_isolated_subnet_ids
  deletion_protection = true
  skip_final_snapshot = false
  port                = 5432

  tags = local.common_tags
}
```

You can use `deletion_protection` to prevent accidental deletion, `skip_final_snapshot` to control snapshot behaviour on deletion, and `port` to override the default engine port.
</Fragment>
</Infrastructure>
