The `connection` generator wires a <Link path="guides/ts-strands-agent">TypeScript Strands Agent</Link> to a <Link path="guides/ts-rdb">Relational Database</Link> project, making a Prisma client available throughout your agent's request lifecycle.

The injection strategy depends on the agent's protocol — **HTTP** (tRPC-based) or **A2A** (Agent-to-Agent).

## Prerequisites

Before using this generator, ensure you have:

1. A <Link path="guides/ts-strands-agent">`ts#strands-agent`</Link> project
2. A <Link path="guides/ts-rdb">`ts#rdb`</Link> project

## Generator Output

The files modified depend on the agent's protocol.

**HTTP Protocol** — modifies four files:

<FileTree>

- packages/my-service/src/my-agent
  - init.ts `db` property added to `Context` interface
  - index.ts `createContext` made async; Prisma client created and returned in context
  - router.ts `db` destructured from `opts.ctx` and passed to `getAgent`
  - agent.ts `getAgent` updated to accept `{ db }` as second parameter

</FileTree>

**A2A Protocol** — modifies two files:

<FileTree>

- packages/my-service/src/my-agent
  - index.ts Prisma client created before `A2AExpressServer`; passed to `getAgent`
  - agent.ts `getAgent` updated to accept `{ db }` as second parameter

</FileTree>

Additionally, the agent's `<agent-name>-serve-local` target is updated to depend on the database's `serve-local` target.

## How It Works

### HTTP Protocol

For HTTP agents, the database flows through the tRPC request context — `createContext` creates a Prisma client per request and it becomes available to every procedure via `opts.ctx`. The router then passes it explicitly into `getAgent`.

#### Context Interface

The Prisma client type is added to the tRPC `Context` in `init.ts`:

```ts title="packages/my-service/src/my-agent/init.ts" {1,5}
import { getPrisma as getMyDb } from ':my-scope/my-db';

export interface Context {
  sessionId: string;
  myDb: Awaited<ReturnType<typeof getMyDb>>;
}
```

#### Request Context

`createContext` is made async and the Prisma client is instantiated and returned:

```ts title="packages/my-service/src/my-agent/index.ts" {1,6,10}
import { getPrisma as getMyDb } from ':my-scope/my-db';

const createContext = async (
  opts: CreateHTTPContextOptions | CreateWSSContextFnOptions,
): Promise<Context> => {
  const myDb = await getMyDb();
  const sessionId = /* ... */;
  return {
    sessionId,
    myDb,
  };
};
```

#### Router and Agent

The database is passed from context into `getAgent` via the subscription handler:

```ts title="packages/my-service/src/my-agent/router.ts" {2,3}
.subscription(async function* (opts) {
  const { myDb } = opts.ctx;
  const agent = await getAgent(opts.ctx.sessionId, { myDb });
  // ...
})
```

```ts title="packages/my-service/src/my-agent/agent.ts" {3}
export const getAgent = async (
  sessionId: string,
  { myDb }: { myDb: Awaited<ReturnType<typeof getMyDb>> },
) => {
  return new Agent({ /* use myDb in tools */ });
};
```

### A2A Protocol

For A2A agents, there is no tRPC context. The database is created once at startup and passed directly to `getAgent` before the `A2AExpressServer` starts.

#### Server Startup

The Prisma client is created before the `A2AExpressServer` and passed to `getAgent`:

```ts title="packages/my-service/src/my-agent/index.ts" {1,4,6}
import { getPrisma as getMyDb } from ':my-scope/my-db';

void (async () => {
  const myDb = await getMyDb();
  const server = new A2AExpressServer({
    agent: await getAgent('default', { myDb }),
    // ...
  });
})();
```

#### Agent Definition

```ts title="packages/my-service/src/my-agent/agent.ts" {3}
export const getAgent = async (
  sessionId: string,
  { myDb }: { myDb: Awaited<ReturnType<typeof getMyDb>> },
) => {
  return new Agent({ /* use myDb in tools */ });
};
```

## Multiple Databases

Running the generator again with a different target adds the second database alongside the first. For HTTP agents, both appear in the `Context` interface and are destructured together in the router:

```ts title="packages/my-service/src/my-agent/router.ts"
const { postgresDb, mysqlDb } = opts.ctx;
const agent = await getAgent(opts.ctx.sessionId, { postgresDb, mysqlDb });
```

## Infrastructure

The generated agent construct implements `IGrantable` and `IConnectable`, so you can grant network and IAM access to the database directly on the construct.

<Infrastructure>
<Fragment slot="cdk">

```ts title="packages/infra/src/stacks/application-stack.ts"
import { MyDatabase } from ':my-scope/common-constructs';

const db = new MyDatabase(this, 'Db', { vpc, ... });
const myAgent = new MyAgent(this, 'MyAgent', { vpc, ... });

db.allowDefaultPortFrom(myAgent);
db.grantConnect(myAgent);
```

`allowDefaultPortFrom` opens the security group rule so the agent runtime can reach the database port. `grantConnect` grants IAM `rds-db:connect` permission to the agent's execution role.

</Fragment>
<Fragment slot="terraform">

Pass the database module outputs into your agent module so it can reach the database and read its runtime configuration:

```hcl title="packages/infra/src/main.tf"
module "my_database" {
  source              = "../../common/terraform/src/app/dbs/my-database"
  vpc_id              = module.vpc.vpc_id
  database_subnet_ids = module.vpc.private_isolated_subnet_ids
}

module "my_agent" {
  source = "../../common/terraform/src/app/agents/my-agent"

  appconfig_application_id     = module.my_database.appconfig_application_id
  database_cluster_resource_id = module.my_database.cluster_resource_id
  database_runtime_user        = module.my_database.database_runtime_user
  database_security_group_id   = module.my_database.security_group_id
  database_port                = module.my_database.cluster_port
}
```

Ensure the agent's execution role has `rds-db:connect` permission and that its security group can reach the database security group on the database port.

</Fragment>
</Infrastructure>

## Local Development

<NxCommands commands={["<agent-name>-serve-local <project-name>"]} />

This starts the agent and all connected databases. The `SERVE_LOCAL=true` environment variable causes each Prisma client to connect to its local Docker database instead of Aurora.
