The `connection` generator wires a <Link path="guides/trpc">tRPC API</Link> to a <Link path="guides/ts-dynamodb">DynamoDB</Link> project, configuring local development so both start together automatically.

## Prerequisites

Before using this generator, ensure you have:

1. A <Link path="guides/trpc">`ts#trpc-api`</Link> project
2. A <Link path="guides/ts-dynamodb">`ts#dynamodb`</Link> project

## Usage

### Run the Generator

<RunGenerator generator="connection" />

Select your tRPC API project as the source and your DynamoDB project as the target.

<GeneratorParameters generator="connection" />

## Generator Output

The generator updates your tRPC API's `project.json` to add a dependency from its `serve-local` target to the DynamoDB project's `serve-local` target. No source files are modified.

## Using DynamoDB in Procedures

Import `getDynamoDBClient`, `resolveTableName`, and your entity factories from the DynamoDB package and use them inside your tRPC procedures:

```ts title="packages/api/src/procedures/example.ts"
import { createExampleEntity } from ':my-scope/my-table';
import { publicProcedure } from '../init.js';

export const listExamples = publicProcedure
  .query(async () => {
    const entity = await createExampleEntity();
    const result = await entity.scan.go();
    return result.data;
  });
```

`getDynamoDBClient()` returns a cached singleton and `resolveTableName()` caches the table name after the first call, so both are efficient across warm Lambda invocations.

## Infrastructure

To allow the API Lambda functions to access the DynamoDB table, grant the necessary permissions in your infrastructure.

<Infrastructure>
<Fragment slot="cdk">

```ts title="packages/infra/src/stacks/application-stack.ts"
import { MyTable } from ':my-scope/common-constructs';

const table = new MyTable(this, 'Table');
const api = new Api(this, 'Api', {
  integrations: Api.defaultIntegrations(this).build(),
});

Object.entries(api.integrations).forEach(([, integration]) => {
  table.table.grantReadWriteData(integration.handler);
});
```

`grantReadWriteData` grants both the DynamoDB and KMS permissions required by each Lambda handler.
</Fragment>
<Fragment slot="terraform">

```hcl title="packages/infra/src/main.tf"
module "my_table" {
  source = "../../common/terraform/src/app/dynamodb/my-table"
}

resource "aws_iam_role_policy" "dynamodb_access" {
  role = module.my_api.lambda_role_name

  policy = jsonencode({
    Version = "2012-10-17"
    Statement = [
      {
        Effect = "Allow"
        Action = [
          "dynamodb:GetItem", "dynamodb:PutItem", "dynamodb:UpdateItem",
          "dynamodb:DeleteItem", "dynamodb:Query", "dynamodb:Scan",
          "dynamodb:BatchGetItem", "dynamodb:BatchWriteItem",
        ]
        Resource = [
          module.my_table.table_arn,
          "${module.my_table.table_arn}/index/*",
        ]
      },
    ]
  })
}
```
</Fragment>
</Infrastructure>

Ensure Lambda functions have the `RUNTIME_CONFIG_APP_ID` environment variable set so `resolveTableName()` can look up the table name at runtime.

## Local Development

The generator configures your tRPC API's `serve-local` target to depend on the DynamoDB project's `serve-local` target. Running:

<NxCommands commands={["serve-local <api-project-name>"]} />

automatically starts the local DynamoDB instance alongside your API. The `SERVE_LOCAL=true` environment variable is set automatically, so `getDynamoDBClient()` and `resolveTableName()` connect to the local DynamoDB Local instance instead of AWS.
