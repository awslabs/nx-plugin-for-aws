L'immagine Docker creata per questo progetto può essere scansionata alla ricerca di vulnerabilità utilizzando [Trivy](https://trivy.dev/), eseguito dall'[immagine Trivy ospitata su ECR](https://gallery.ecr.aws/aquasecurity/trivy).

Un target `trivy` viene aggiunto al progetto che scansiona l'immagine creata e **termina con codice di uscita non-zero** se viene trovata qualsiasi vulnerabilità di gravità `HIGH` o `CRITICAL`. Il `Dockerfile` generato utilizza un'immagine base senza vulnerabilità risolvibili note di queste gravità al momento della generazione e aggiorna gli strumenti inclusi (come `npm`) per mantenerla tale.

La scansione utilizza lo stesso motore di container della build dell'immagine (`docker` o `finch`), quindi non sono necessari strumenti aggiuntivi. Poiché la scansione viene rieseguita solo quando l'immagine cambia, un'immagine non modificata non viene scansionata nuovamente. Lo script root `trivy` fornito scansiona ogni immagine nel workspace:

<PackageManagerShortCommand commands={['trivy']} />

:::tip[Esegui Trivy in CI]
La scansione non è intenzionalmente collegata a `build` poiché ciò può introdurre attriti non necessari durante l'iterazione nello sviluppo, dato che il database delle vulnerabilità di Trivy viene continuamente aggiornato con nuovi CVE.

Raccomandiamo invece di eseguire il comando sopra come passaggio dedicato nella pipeline CI prima del deployment negli ambienti di produzione.
:::

:::caution[Le vulnerabilità non risolvibili vengono ignorate]
La scansione viene eseguita con `--ignore-unfixed`, quindi le vulnerabilità senza una correzione disponibile non fanno fallire la build, poiché non possono essere risolte aggiornando gli strumenti nel `Dockerfile` — non vengono riportate nell'output della scansione della build. Man mano che vengono divulgate nuove vulnerabilità, il risultato delle scansioni può cambiare nel tempo, quindi rivedi periodicamente l'immagine eseguendo `trivy image <your-image>` senza il flag e ricostruisci contro un'immagine base più recente (o aggiungi passaggi al container per applicare le patch disponibili) una volta pubblicata una correzione.
:::

#### Soppressione dei Risultati di Trivy

Potrebbero esserci casi in cui si desidera sopprimere una vulnerabilità specifica, ad esempio quando non è ancora disponibile una correzione e si è valutato il rischio come accettabile.

Aggiungi l'ID della vulnerabilità (uno per riga) al file `.trivyignore` nella radice del progetto (cioè accanto al file `project.json`):

```gitignore
# .trivyignore
# node-tar arbitrary file write - not exploitable in our usage
CVE-2024-XXXXX
```

Per maggiori dettagli sul filtraggio dei risultati, consulta la [documentazione sul filtraggio di Trivy](https://trivy.dev/latest/docs/configuration/filtering/#by-finding-ids).
