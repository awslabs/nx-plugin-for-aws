Questo generatore crea un nuovo progetto TypeScript basato su [Amazon DynamoDB](https://aws.amazon.com/dynamodb/), utilizzando [ElectroDB](https://electrodb.dev/) per la modellazione di entità type-safe. Genera il codice dell'applicazione e l'infrastruttura necessaria per il provisioning e la gestione di una tabella DynamoDB utilizzando AWS CDK o Terraform, con supporto per il design single-table e sviluppo locale integrato tramite DynamoDB Local.

## Sviluppo Locale

### Avvio di DynamoDB Locale

Il generatore configura un target `serve-local` che avvia un'istanza di [DynamoDB Local](https://docs.aws.amazon.com/amazondynamodb/latest/developerguide/DynamoDBLocal.html) e crea la tabella:

<NxCommands commands={['run <project-name>:serve-local']} />

Questo automaticamente:
1. Scarica l'immagine di DynamoDB Local (target `pull-image`)
2. Avvia un container
3. Crea una tabella locale con gli indici definiti in `dynamodb.config.json`

### Modellazione dei Dati

Il progetto generato utilizza [ElectroDB](https://electrodb.dev/) per la modellazione di entità type-safe su una singola tabella DynamoDB. Aggiungi o aggiorna i file delle entità in `src/entities/`, utilizzando l'entità di esempio generata come punto di partenza.

Esempio di definizione di entità:

```ts title="packages/my-table/src/entities/example.ts"
import { Entity } from 'electrodb';
import { getDynamoDBClient, resolveTableName } from '../client.js';

export const createExampleEntity = async () =>
  new Entity(
    {
      model: {
        entity: 'example',
        version: '1',
        service: 'MyTable',
      },
      attributes: {
        id: {
          type: 'string',
          required: true,
        },
        createdAt: {
          type: 'string',
          required: true,
          default: () => new Date().toISOString(),
          readOnly: true,
        },
        updatedAt: {
          type: 'string',
          required: true,
          default: () => new Date().toISOString(),
          watch: '*',
          set: () => new Date().toISOString(),
        },
      },
      indexes: {
        primary: {
          pk: {
            field: 'pk',
            composite: ['id'],
          },
          sk: {
            field: 'sk',
            composite: [],
          },
        },
      },
    },
    { client: getDynamoDBClient(), table: await resolveTableName() },
  );
```

Per maggiori dettagli, consulta la [documentazione delle entità ElectroDB](https://electrodb.dev/en/modeling/entities/).

### Utilizzo del Client DynamoDB

Il file generato `src/client.ts` esporta due utility chiave:

- `getDynamoDBClient()` — restituisce un singleton `DynamoDBClient` in cache. Quando `SERVE_LOCAL=true`, si connette all'istanza locale di DynamoDB Local; altrimenti crea un client AWS utilizzando la catena di credenziali predefinita.
- `resolveTableName()` — restituisce il nome della tabella DynamoDB. Quando `SERVE_LOCAL=true`, restituisce la costante del nome della tabella locale; altrimenti recupera il nome da AWS AppConfig utilizzando la variabile d'ambiente `RUNTIME_CONFIG_APP_ID` e lo memorizza in cache per le chiamate successive.

### Arresto di DynamoDB Locale

L'arresto di `serve-local` (ad esempio con `Ctrl+C`) rimuove automaticamente il container di DynamoDB Local, ma preserva il volume nominato in modo che i tuoi dati persistano tra i riavvii.

:::caution[Windows]
A causa di limitazioni nella gestione dei segnali su Windows, il container non viene rimosso automaticamente quando `serve-local` viene arrestato. Dovrai rimuoverlo manualmente:

```bash
<engine> rm -f <scope>-dynamodb
```

Sostituisci `<engine>` con il tuo motore di container (`docker` o `finch`) e `<scope>` con lo scope del tuo workspace Nx (ad esempio `proj`).
:::

## Aggiunta/Rimozione di Global Secondary Index

I GSI sono definiti in `dynamodb.config.json` nella radice del progetto. Aggiungi una voce a `globalSecondaryIndexes` per ogni GSI, seguendo la convenzione di denominazione del [design single-table](https://electrodb.dev/en/core-concepts/single-table-relationships/) per le chiavi GSI:

```json title="packages/my-table/dynamodb.config.json"
{
  "globalSecondaryIndexes": [
    {
      "indexName": "gsi1pk-gsi1sk-index",
      "partitionKey": "gsi1pk",
      "sortKey": "gsi1sk"
    }
  ]
}
```

Il campo `sortKey` è opzionale per i GSI con solo chiave hash.

Questo file di configurazione viene letto direttamente dalla radice del progetto da tutti i consumatori:
- **Sviluppo locale** — `serve-local` legge `dynamodb.config.json` e crea o aggiorna la tabella locale per corrispondere all'elenco dei GSI
- **CDK** — il costrutto legge `dynamodb.config.json` al momento della sintesi, quindi le modifiche ai GSI si riflettono al prossimo `cdk deploy`
- **Terraform** — il modulo legge `dynamodb.config.json` al momento del plan/apply

### Un GSI per Deployment

:::caution
DynamoDB [non consente la creazione o l'eliminazione di più di un GSI in un singolo aggiornamento della tabella](https://docs.aws.amazon.com/amazondynamodb/latest/developerguide/GSI.OnlineOps.html).

Ogni deployment può aggiungere o rimuovere **al massimo un GSI**. Se devi aggiungere o rimuovere più GSI, fallo uno alla volta — aggiorna `dynamodb.config.json`, effettua il deployment dello stack, quindi ripeti per la modifica successiva.
:::

## Connessione alla Tabella

In qualsiasi progetto TypeScript, importa le factory delle entità dal tuo package DynamoDB e utilizzale direttamente:

```ts
import { createExampleEntity } from ':my-scope/my-table';

const entity = await createExampleEntity();
const result = await entity.query.primary({ id: '123' }).go();
```

:::note[Configurazione runtime]
Quando viene eseguito in AWS, `resolveTableName()` recupera il nome della tabella da AWS AppConfig utilizzando la variabile d'ambiente `RUNTIME_CONFIG_APP_ID`. I progetti costruiti con questo plugin (API tRPC, API Smithy, agenti, server MCP) hanno già questa variabile configurata automaticamente. Per altri progetti TypeScript, assicurati che `RUNTIME_CONFIG_APP_ID` sia impostata nell'ambiente di runtime con l'ID dell'applicazione AppConfig fornito dalla tua infrastruttura. Per maggiori informazioni, consulta la <Link path="guides/runtime-config">guida alla Configurazione Runtime</Link>.
:::

### Protezione dalla Cancellazione

La protezione dalla cancellazione è abilitata per impostazione predefinita per prevenire la cancellazione accidentale della tabella.

#### Disabilitare la Protezione dalla Cancellazione

Disabilitala per ambienti in cui la cancellazione della tabella è prevista, come stack di sviluppo o preview di breve durata.

<Infrastructure>
<Fragment slot="cdk">

```ts title="packages/infra/src/stacks/application-stack.ts"
import { MyTable } from ':my-scope/common-constructs';

const table = new MyTable(this, 'Table', {
  deletionProtection: false,
});
```
</Fragment>
<Fragment slot="terraform">

```hcl title="packages/infra/src/main.tf"
module "my_table" {
  source                      = "../../common/terraform/src/app/dynamodb/my-table"
  deletion_protection_enabled = false
}
```
</Fragment>
</Infrastructure>

### Modalità di Fatturazione

La tabella utilizza per impostazione predefinita la fatturazione on-demand (`PAY_PER_REQUEST`). Passa alla capacità provisionata per carichi di lavoro prevedibili ad alto throughput.

import { MyTable } from ':my-scope/common-constructs';

const table = new MyTable(this, 'Table', {
  billingMode: BillingMode.PROVISIONED,
  readCapacity: 5,
  writeCapacity: 5,
