La configurazione runtime è il meccanismo utilizzato da Nx Plugin for AWS per passare valori al momento del deploy tra progetti e componenti generati in modo che possano connettersi tra loro. Ad esempio, quando generi un'API, il suo URL viene automaticamente registrato nella configurazione runtime in modo che un sito web connesso possa scoprirlo.

## Come Funziona

La configurazione runtime è organizzata in **namespace**. Ogni namespace è un raggruppamento logico di valori di configurazione correlati. Al momento del deploy, tutti i namespace vengono memorizzati in **AWS AppConfig** come Configuration Profiles.

Due namespace integrati vengono utilizzati dai construct generati:

- **`connection`** — configurazione che consente ai progetti generati di connettersi tra loro:
  - **URL delle API** — registrati automaticamente dai construct API
  - **Impostazioni Cognito** — registrate automaticamente dal construct UserIdentity
  - **ARN runtime degli agenti** — aggiunti dal generatore di connessione quando connetti un sito web React a uno Strands Agent
- **`agentcore`** — ARN runtime di AgentCore per agenti e server MCP. Registrati automaticamente dai construct agent/MCP e utilizzati per la scoperta lato server (agent → agent tramite A2A, agent → server MCP).

Il namespace `connection` viene anche distribuito come file `runtime-config.json` nel bucket S3 del tuo sito web, abilitando la scoperta lato client delle risorse backend. Il namespace `agentcore` è solo lato server (tramite AppConfig) quindi gli ARN runtime degli agenti non vengono esposti al frontend a meno che non connetti esplicitamente un sito web a un agente.

Puoi definire tutti i namespace aggiuntivi che desideri, fornendo un'alternativa conveniente alle variabili d'ambiente per passare valori al momento del deploy come i nomi delle tabelle DynamoDB alle tue funzioni Lambda o ad altre risorse di calcolo.

```d2
direction: down

iac: "Infrastructure as code\nRuntimeConfig.set(...)"

appconfig: "AWS AppConfig" {
  shape: cylinder
  connection: "connection namespace"
  agentcore: "agentcore namespace"
  custom: "custom namespaces"
}

s3: "website S3 bucket" {
  shape: cylinder
  rcj: runtime-config.json {
    shape: page
  }
}

website: React website

server: Lambda / agent

iac -> appconfig.connection
iac -> appconfig.agentcore
iac -> appconfig.custom

appconfig.connection -> s3.rcj: deployed
s3.rcj -> website: fetched at load

appconfig.connection -> server
appconfig.agentcore -> server: Powertools getAppConfig
appconfig.custom -> server
```

## Infrastruttura

### Scrittura della Configurazione

I construct generati scrivono automaticamente la configurazione rilevante nel namespace `connection`. Puoi anche scrivere i tuoi valori in qualsiasi namespace.

<Infrastructure>
<Fragment slot="cdk">
Il construct CDK `RuntimeConfig` è un singleton con scope di stage. Usa `set()` per scrivere una chiave in un namespace:

```ts title="packages/infra/src/stacks/application-stack.ts"
import { RuntimeConfig } from ':my-scope/common-constructs';

const rc = RuntimeConfig.ensure(this);

// Built-in 'connection' namespace (written automatically by generated constructs)
rc.set('connection', 'apis', {
  ...rc.get('connection').apis,
  MyApi: api.url,
});

// Custom namespaces for server-side configuration
rc.set('tables', 'users', {
  tableName: usersTable.tableName,
  tableArn: usersTable.tableArn,
});
```

Al momento di synth/deploy, `RuntimeConfig` crea un'applicazione **AWS AppConfig** contenente:
- Un **Configuration Profile** per ogni namespace
- Una **Hosted Configuration Version** con i dati JSON per ogni profilo
- Un **Deployment** istantaneo all'ambiente `default`
</Fragment>
<Fragment slot="terraform">
Terraform collega la configurazione runtime attraverso tre moduli `core/runtime-config/*`. Dichiarali nel seguente ordine nel tuo modulo root:

<Steps>

1. **`core/runtime-config/appconfig`** — dichiarato una volta, vicino all'inizio del modulo root. Crea l'applicazione AppConfig, l'ambiente, la strategia di deployment e un profilo di configurazione per namespace. I suoi output `application_id` e `application_arn` vengono passati a ogni modulo che legge la configurazione runtime a runtime (agenti, server MCP, funzioni Lambda, ecc.).

   ```hcl title="packages/infra/src/main.tf"
   module "runtime_config_appconfig" {
     source = "../../common/terraform/src/core/runtime-config/appconfig"

     application_name = "my-app-runtime-config"
   }
   ```

2. **`core/runtime-config/entry`** — un'invocazione per contributo. I moduli API, agent e MCP generati chiamano questo internamente per pubblicare i loro URL e ARN. Chiamalo direttamente per pubblicare configurazioni personalizzate.

   ```hcl title="packages/infra/src/main.tf"
   # Automatico — fatto all'interno dei moduli generati
   module "add_api_url" {
     source = "../../common/terraform/src/core/runtime-config/entry"

     namespace = "connection"
     key       = "apis"
     value     = { "MyApi" = module.my_api.api_url }
   }

   # Namespace personalizzato per la configurazione lato server
   module "add_table_config" {
     source = "../../common/terraform/src/core/runtime-config/entry"

     namespace = "tables"
     key       = "users"
     value     = {
       tableName = aws_dynamodb_table.users.name
       arn       = aws_dynamodb_table.users.arn
     }
   }
   ```

3. **`core/runtime-config/appconfig-deployment`** — dichiarato una volta, alla fine del modulo root, con `depends_on` che copre ogni modulo che contribuisce con una entry. Aggrega le entry contribuite in un JSON per namespace e crea la versione di configurazione ospitata + deployment contro l'applicazione condivisa.

   ```hcl title="packages/infra/src/main.tf"
   module "runtime_config_appconfig_deployment" {
     source = "../../common/terraform/src/core/runtime-config/appconfig-deployment"

     application_id            = module.runtime_config_appconfig.application_id
     environment_id            = module.runtime_config_appconfig.environment_id
     deployment_strategy_id    = module.runtime_config_appconfig.deployment_strategy_id
     configuration_profile_ids = module.runtime_config_appconfig.configuration_profile_ids
     namespaces                = module.runtime_config_appconfig.namespaces

     depends_on = [
       module.my_api,
       module.add_table_config,
       # ...ogni modulo che contribuisce con una entry
     ]
   }
   ```

</Steps>
</Fragment>
</Infrastructure>

### Lettura della Configurazione

I consumer lato server necessitano dell'**AppConfig Application ID** e delle autorizzazioni IAM per leggere la configurazione a runtime. I construct generati gestiscono questo automaticamente.

<Infrastructure>
<Fragment slot="cdk">
Usa `appConfigApplicationId` per ottenere l'AppConfig Application ID, e `grantReadAppConfig()` per concedere i permessi di lettura:

```ts title="packages/infra/src/stacks/application-stack.ts"
const rc = RuntimeConfig.ensure(this);

// Get the AppConfig Application ID (lazy token, resolved at synth time)
const appId = rc.appConfigApplicationId;

// Pass it as an environment variable to a Lambda function
const myFunction = new Function(this, 'MyFunction', {
  // ...
  environment: {
    RUNTIME_CONFIG_APP_ID: appId,
  },
});

// Grant the function permission to read from AppConfig
rc.grantReadAppConfig(myFunction);
```

:::note[Registrazione automatica]
I construct API e agent generati fanno questo automaticamente — devi farlo manualmente solo se crei le tue funzioni Lambda o risorse di calcolo che necessitano di leggere la configurazione runtime.
:::
</Fragment>
<Fragment slot="terraform">
Fai riferimento all'output `application_id` del modulo condiviso `runtime_config_appconfig` per ottenere l'AppConfig Application ID, e aggiungi le appropriate dichiarazioni di policy IAM:

```hcl title="packages/infra/src/main.tf"
# Pass the AppConfig Application ID as an environment variable
resource "aws_lambda_function" "my_function" {
  # ...
  environment {
    variables = {
      RUNTIME_CONFIG_APP_ID = module.runtime_config_appconfig.application_id
    }
  }
}

# Grant the function permission to read from AppConfig
resource "aws_iam_policy" "appconfig_read" {
  name = "AppConfigReadPolicy"
  policy = jsonencode({
    Version = "2012-10-17"
    Statement = [{
      Effect = "Allow"
      Action = [
        "appconfig:StartConfigurationSession",
        "appconfig:GetLatestConfiguration"
      ]
      Resource = ["${module.runtime_config_appconfig.application_arn}/*"]
    }]
  })
}
```

:::note[Registrazione automatica]
I moduli agent e MCP generati accettano gli input `appconfig_application_id` + `appconfig_application_arn` — passa gli output del modulo condiviso `runtime_config_appconfig` e collegheranno la variabile d'ambiente e i permessi IAM per te. Devi farlo manualmente solo per funzioni Lambda personalizzate o risorse di calcolo.
:::
</Fragment>
</Infrastructure>

:::tip[Sviluppo Locale]
Se utilizzi la configurazione runtime con namespace personalizzati (ad es. `tables`, `features`), devi solo definire una singola variabile d'ambiente `RUNTIME_CONFIG_APP_ID` localmente per puntare alla tua applicazione AppConfig distribuita. Questo consente alle tue API o agenti locali di recuperare la stessa configurazione dell'ambiente distribuito senza dover replicare tutte le singole variabili d'ambiente.
:::

## Accesso Lato Server tramite AppConfig

I consumer lato server come le funzioni Lambda e gli agenti possono recuperare la configurazione runtime da **AWS AppConfig** utilizzando [AWS Lambda Powertools](https://docs.aws.amazon.com/powertools/).

Tutti i construct API e agent generati sono automaticamente configurati con:
- La variabile d'ambiente `RUNTIME_CONFIG_APP_ID` (l'AppConfig Application ID)
- Permessi IAM per leggere da AppConfig

<Tabs>
  <TabItem label="TypeScript" icon="seti:typescript">

Usa `getAppConfig` da [`@aws-lambda-powertools/parameters`](https://docs.aws.amazon.com/powertools/typescript/latest/features/parameters):

```ts
import { getAppConfig } from '@aws-lambda-powertools/parameters/appconfig';

// Retrieve the 'connection' namespace as a parsed JSON object
const config = await getAppConfig('connection', {
  application: process.env.RUNTIME_CONFIG_APP_ID!,
  environment: 'default',
  transform: 'json',
});

// Access values
const apiUrl = config.apis?.MyApi;
const cognitoProps = config.cognitoProps;
```

Puoi anche recuperare namespace personalizzati:

```ts
// Retrieve a custom 'tables' namespace
const tablesConfig = await getAppConfig('tables', {
  application: process.env.RUNTIME_CONFIG_APP_ID!,
  environment: 'default',
  transform: 'json',
});

const usersTableName = tablesConfig.users?.tableName;
```

  </TabItem>
  <TabItem label="Python" icon="seti:python">

Usa `get_app_config` da [`aws_lambda_powertools.utilities.parameters`](https://docs.aws.amazon.com/powertools/python/latest/utilities/parameters/):

```python
import os
from aws_lambda_powertools.utilities import parameters

# Retrieve the 'connection' namespace as a parsed JSON object
config = parameters.get_app_config(
    name="connection",
    environment="default",
    application=os.environ["RUNTIME_CONFIG_APP_ID"],
    transform="json",
)

# Access values
api_url = config.get("apis", {}).get("MyApi")
cognito_props = config.get("cognitoProps")
```

Puoi anche recuperare namespace personalizzati:

```python
# Retrieve a custom 'tables' namespace
tables_config = parameters.get_app_config(
    name="tables",
    environment="default",
    application=os.environ["RUNTIME_CONFIG_APP_ID"],
    transform="json",
)

users_table_name = tables_config.get("users", {}).get("tableName")
```

  </TabItem>
</Tabs>
