I nuovi workspace creati con `@aws/nx-plugin` includono gli hook pre-commit di [git-secrets](https://github.com/awslabs/git-secrets) configurati di default. Questo esegue la scansione dei file in staging alla ricerca di pattern che corrispondono a credenziali AWS prima di ogni commit, prevenendo perdite accidentali di credenziali.

## Cosa è Incluso

Quando crei un nuovo workspace, viene configurato automaticamente quanto segue:

<FileTree>
  - .git-secrets/
    - git-secrets Lo script bash git-secrets (vendorizzato da awslabs/git-secrets)
    - setup.sh Registra i pattern delle credenziali AWS nella configurazione git locale
  - .husky/
    - pre-commit Esegue la scansione git-secrets prima di ogni commit
  - package.json Include husky e lo script prepare per la configurazione automatica
</FileTree>

## Come Funziona

1. Lo script `prepare` viene eseguito dopo `npm install` / `pnpm install`, che inizializza gli hook git di husky e registra i pattern delle credenziali AWS nella tua configurazione git locale.
2. Ad ogni `git commit`, l'hook pre-commit di husky invoca `git-secrets` per scansionare i file in staging.
3. Se viene rilevato un pattern corrispondente a una credenziale AWS, il commit viene bloccato con un messaggio di errore.

## Pattern Rilevati

Vengono scansionati i seguenti pattern:

- **AWS Access Key IDs** — `AKIA...`, `ASIA...` e altri prefissi di chiavi IAM seguiti da 16 caratteri alfanumerici maiuscoli
- **AWS Secret Access Keys** — stringhe base64 di 40 caratteri precedute da un'etichetta `SECRET_ACCESS_KEY`
- **AWS Account IDs** — numeri di 12 cifre preceduti da un'etichetta `ACCOUNT_ID`
- **Bedrock API Keys** — chiavi Bedrock a lunga durata (`ABSK...`) e a breve durata
- **Valori del file credentials** — valori di chiavi letterali trovati in `~/.aws/credentials`

Le chiavi di esempio ben note (ad es., `AKIAIOSFODNN7EXAMPLE`) sono automaticamente inserite nella lista consentita.

## Consentire Falsi Positivi

Se git-secrets blocca un commit che contiene un falso positivo:

```sh
# Consenti un pattern specifico
git config --add secrets.allowed 'my-pattern-here'
```

Oppure aggiungi i pattern a un file `.gitallowed` nella radice del repository (un'espressione regolare per riga).

Per un override una tantum:

```sh
git commit --no-verify
```

## Disattivazione

Per creare un workspace senza git-secrets:

```sh
pnpm create @aws/nx-workspace my-project --no-gitSecrets
```

Oppure passa `--gitSecrets=false` al generatore preset.

## Requisiti

- **Git** deve essere inizializzato nel workspace (`git init`)
- **Bash** deve essere disponibile nel PATH (viene fornito con Git per Windows su Windows)
- I pattern vengono registrati quando viene eseguito `prepare` — se cloni una copia nuova, esegui il comando di installazione del tuo package manager per registrarli nuovamente
