## Task 1: Implementare lo Story Agent

Lo Story Agent è un agente [Strands](https://strandsagents.com/) esposto tramite il protocollo [AG-UI](https://ag-ui.com/). Ogni sessione AG-UI (identificata da un `threadId`) corrisponde a un'istanza Agent di lunga durata sul server, quindi la cronologia delle conversazioni dell'agente viene mantenuta in memoria per tutta la durata della sessione.

Aggiorneremo il prompt di sistema dell'agente per giocare un'avventura nel dungeon e rimuoveremo lo strumento di esempio.

### Implementazione dell'agente

Aggiorna `packages/story/dungeon_adventure_story/agent/agent.py`:

<E2EDiff lang="python" before="dungeon-adventure/3/agent.py.old.template" after="dungeon-adventure/3/agent.py.template" />

Il prompt di sistema indica all'agente che il primo messaggio sarà della forma `I am <player>, playing in the <genre> genre.` — la nostra UI invia esattamente questo come messaggio utente di apertura, così l'agente ha tutto ciò di cui ha bisogno per iniziare la storia. Poiché la libreria AG-UI mantiene automaticamente la cronologia delle conversazioni di ogni thread, non è necessario passare il nome del giocatore o il genere nei turni successivi.

Il generatore di connessioni ha già configurato il client MCP dell'inventario nel <Link path="get_started/tutorials/dungeon-game/1">Modulo 1</Link>, quindi l'agente ha accesso agli strumenti dell'inventario.

:::note[Dov'è `main.py`?]
Il wrapper FastAPI del protocollo AG-UI gestisce la traduzione richiesta/risposta per noi. Il `main.py` prodotto dal generatore crea già il wrapper attorno a `get_agent(...)` — non è necessario modificarlo.
:::

### Creare un utente test Cognito

Prima di testare l'agente deployato (e prima di giocare tramite l'UI nel prossimo modulo), crea un utente test Cognito. Abbiamo disabilitato l'auto-registrazione in precedenza, quindi questo passaggio è necessario indipendentemente dal fatto che tu testi localmente o contro il runtime deployato:

```bash
# Dagli output CDK
export POOL_ID="<UserPoolId dagli output CDK>"
export CLIENT_ID="<UserPoolClientId dagli output CDK>"
export REGION="<tua-regione>"

# Disabilita MFA per semplificare la creazione dell'utente
aws cognito-idp set-user-pool-mfa-config \
  --mfa-configuration OFF \
  --user-pool-id $POOL_ID \
  --region $REGION

# Crea un utente test
aws cognito-idp admin-create-user \
  --user-pool-id $POOL_ID \
  --username "test" \
  --temporary-password "TempPass123-" \
  --region $REGION \
  --message-action SUPPRESS > /dev/null

# Imposta una password permanente (sostituisci con qualcosa di più sicuro!)
aws cognito-idp admin-set-user-password \
  --user-pool-id $POOL_ID \
  --username "test" \
  --password "PermanentPass123-" \
  --region $REGION \
  --permanent > /dev/null
```

### Testare il tuo Agent

Puoi testare il tuo Agent in due modi:
<ul>
<li>Avviando un'istanza locale del server Agent e chattando con esso in modo interattivo usando il target `agent-chat`, oppure</li>
<li>Chiamando l'API deployata usando curl con un token JWT.</li>
</ul>

<Tabs>
  <TabItem label="Local">
  Il target `agent-chat` avvia l'agente localmente (insieme alle sue dipendenze, tramite `agent-serve-local`) e ti porta in un prompt di chat interattivo. Esegui:

    <NxCommands commands={["agent-chat story"]} />

    :::note[`agent-serve` vs `agent-serve-local`]
    Se vuoi chattare con l'agente contro il tuo server MCP **deployato** invece di uno locale, avvia il server direttamente con `agent-serve` e passa `RUNTIME_CONFIG_APP_ID` / `AWS_REGION`:

    <NxCommands highlights={['xxxx', 'region']} env={{RUNTIME_CONFIG_APP_ID:"xxxx", AWS_REGION: '<region>'}} commands={["agent-serve story"]} />

    poi in un'altra shell esegui `agent-chat-cli agui http://localhost:8081/invocations`.
    :::

    Quando richiesto, prova ad avviare il gioco con qualcosa come:

    ```
    I am Aria, playing in the medieval genre.
    ```
  </TabItem>
  <TabItem label="Deployed">
    Scambia le credenziali dell'utente test per un access token Cognito:

    ```bash
    # Autentica l'utente test e cattura il suo access token
    export BEARER_TOKEN=$(aws cognito-idp initiate-auth \
      --client-id "$CLIENT_ID" \
      --auth-flow USER_PASSWORD_AUTH \
      --auth-parameters USERNAME='test',PASSWORD='PermanentPass123-' \
      --region $REGION \
      --query "AuthenticationResult.AccessToken" \
      --output text)
    ```

    Invoca l'agente deployato usando l'URL Bedrock AgentCore runtime. Il protocollo AG-UI si aspetta un payload `RunAgentInput` — uno minimale che invia il messaggio utente di apertura è simile a questo:

    ```bash
    # Imposta Story Agent ARN dagli output CDK
    export AGENT_ARN="<StoryAgentArn dagli output CDK>"

    # Codifica URL dell'ARN
    export ENCODED_ARN=$(echo $AGENT_ARN | sed 's/:/%3A/g' | sed 's/\//%2F/g')

    # Costruisci l'URL di invocazione
    export AGENT_URL="https://bedrock-agentcore.$REGION.amazonaws.com/runtimes/$ENCODED_ARN/invocations?qualifier=DEFAULT"

    # Invoca l'agente
    curl -N -X POST "$AGENT_URL" \
      -H "authorization: Bearer $BEARER_TOKEN" \
      -H "Content-Type: application/json" \
      -H "Accept: text/event-stream" \
      -H "X-Amzn-Bedrock-AgentCore-Runtime-Session-Id: abcdefghijklmnopqrstuvwxyz-123456789" \
      -d '{
        "threadId": "abcdefghijklmnopqrstuvwxyz-123456789",
        "runId": "run-1",
        "messages": [{"id": "m1", "role": "user", "content": "I am UnnamedHero, playing in the superhero genre."}],
        "tools": [], "context": [], "forwardedProps": {}, "state": {}
      }'
    ```

    :::note[Codifica URL]
    La codifica URL è necessaria per formattare correttamente l'ARN per l'endpoint API di Bedrock AgentCore.
    :::
  </TabItem>
</Tabs>

Se il comando viene eseguito con successo, dovresti iniziare a vedere gli eventi Server-Sent di AG-UI in streaming — gli eventi `TEXT_MESSAGE_CONTENT` trasportano il testo della storia pezzo per pezzo.
