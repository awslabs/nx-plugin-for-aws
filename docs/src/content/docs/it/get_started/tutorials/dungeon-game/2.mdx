## Task 1: Implementare la Game API

Implementeremo le seguenti API in questa sezione:

1. `saveGame` - crea o aggiorna un gioco.
2. `queryGames` - restituisce una lista paginata di giochi salvati precedentemente.
3. `queryInventory` - restituisce una lista paginata degli oggetti nell'inventario di un giocatore.
4. `queryActions` - restituisce la cronologia delle conversazioni per un dato gioco.

:::note[Dove risiede ogni pezzo di stato]
- `games` e `inventory` sono in **DynamoDB** (tramite ElectroDB) — le lambda della Game API e il server MCP dell'Inventario li leggono/scrivono.
- `actions` (trascrizione della chat) è in **S3** — l'`S3SessionManager` dello Story Agent scrive un blob JSON per turno, e `queryActions` nella Game API li legge in modo che l'UI possa riprodurre la cronologia al ricaricamento della pagina. Non esiste una procedura `saveAction`; l'agente è l'unico scrittore.
:::

Nota qui che poiché ogni procedura è gestita da una singola funzione lambda, possiamo seguire il principio del minimo privilegio e assegnare solo i permessi di lettura/scrittura necessari in base all'implementazione della procedura.

:::tip[Configurazione Runtime]
Il costrutto `DungeonDb` registra il nome della tabella deployata nella <Link path="guides/runtime-config">Configurazione Runtime</Link> sotto il namespace `dynamodb`, memorizzato in AWS AppConfig. L'helper `resolveTableName()` generato lo recupera a runtime utilizzando Lambda Powertools, invece di doverlo passare come variabile d'ambiente. Un vantaggio di questo approccio è che man mano che il numero di tabelle o altri identificatori di infrastruttura cresce, hai bisogno solo di una singola variabile d'ambiente `RUNTIME_CONFIG_APP_ID` quando esegui contro l'infrastruttura deployata.
:::

Poiché abbiamo collegato la Game API al progetto `DungeonDb` nel Modulo 1, questo target avvia anche DynamoDB Local automaticamente.

### Avviare il server locale

Avvia la Game API localmente con il target `serve-local`. Assicurati che il tuo motore di container (Docker o Finch) sia in esecuzione prima, poiché questo avvia anche DynamoDB Local:

<NxCommands commands={["serve-local game-api"]} />

Il target `serve-local` imposta `SERVE_LOCAL=true`, quindi `resolveTableName()` restituisce il nome della tabella locale e `getDynamoDBClient()` si connette a DynamoDB Local invece che ad AWS — nessun deployment e nessun `RUNTIME_CONFIG_APP_ID` richiesto.

<Aside type="tip" title="Prima esecuzione">
La prima esecuzione scarica l'immagine del container di DynamoDB Local, il che potrebbe richiedere un momento. Lascia il server in esecuzione in questo terminale e apri un secondo terminale per i comandi `curl` qui sotto.
</Aside>

### Testare l'API

Una volta avviato il server, interroga la lista (vuota) dei giochi:

```bash
curl -X GET 'http://localhost:2022/games.query?input=%7B%7D'
```

:::note[Codifica URL]
Il `%7B%7D` che passiamo per testare l'API è un oggetto JSON vuoto (`{}`) codificato in URI.
:::

Vedrai una lista vuota:

```json
{"result":{"data":{"items":[],"cursor":null}}}
```

Ora salva un gioco:

```bash
curl -X POST 'http://localhost:2022/games.save' \
  -H 'Content-Type: application/json' \
  -d '{"playerName":"Alice","genre":"zombie"}'
```

Il salvataggio restituisce il gioco persistito (con il timestamp `lastUpdated` che l'entità imposta per te):

```json
{"result":{"data":{"playerName":"Alice","genre":"zombie","lastUpdated":"..."}}}
```

Interroga di nuovo per confermare che è stato persistito in DynamoDB Local:

```bash
curl -X GET 'http://localhost:2022/games.query?input=%7B%7D'
```

Questa risposta ora include il gioco salvato:

```json
{"result":{"data":{"items":[{"playerName":"Alice","genre":"zombie","lastUpdated":"..."}],"cursor":null}}}
```

Puoi fermare il server locale (`Ctrl+C`) una volta terminato.

## Task 5: Testare il server MCP dell'Inventario localmente

Possiamo provare gli strumenti del server MCP con l'[MCP Inspector](https://github.com/modelcontextprotocol/inspector) utilizzando il target `mcp-server-inspect` generato. L'inspector si connette al server tramite STDIO, quindi eseguiamo DynamoDB Local separatamente e diciamo al server di utilizzarlo.

In un terminale, avvia DynamoDB Local:

<NxCommands commands={["serve-local dungeon-db"]} />

In un secondo terminale, avvia l'inspector:

<NxCommands commands={["mcp-server-inspect inventory"]} />

Questo avvia l'MCP Inspector su `http://localhost:6274`. Nell'interfaccia dell'inspector, prima di connetterti, aggiungi una variabile d'ambiente `SERVE_LOCAL` con il valore `true` in modo che gli strumenti del server comunichino con DynamoDB Local, quindi fai clic su **Connect**.

<Aside type="tip" title="Perché impostare SERVE_LOCAL?">
A differenza di `serve-local`, il target `mcp-server-inspect` esegue il server STDIO semplice, quindi non imposta `SERVE_LOCAL` per te. Impostarlo nell'ambiente dell'inspector fa sì che `getDynamoDBClient()` e `resolveTableName()` si connettano alla tabella locale invece che ad AWS.
</Aside>

Passa alla scheda **Tools**, fai clic su **List Tools**, e prova `add-to-inventory` (ad es. `playerName: Alice`, `itemName: Rusty Sword`, `emoji: ⚔️`) seguito da `list-inventory-items` per vederlo persistito in DynamoDB Local. Ferma entrambi i terminali (`Ctrl+C`) quando hai finito.

<Aside type="tip" title="Preferisci il trasporto HTTP?">
Se preferisci gestire il server tramite HTTP (ad es. per scripting), il target `mcp-server-serve-local` esegue il trasporto HTTP Streamable con `SERVE_LOCAL=true` impostato per te, e avvia DynamoDB Local automaticamente:

<NxCommands commands={["run @dungeon-adventure/inventory:mcp-server-serve-local"]} />

Quindi inizializza una sessione contro `http://localhost:8000/mcp`:
```bash
curl -X POST http://localhost:8000/mcp \
  -H 'Content-Type: application/json' \
  -H 'Accept: application/json, text/event-stream' \
  -d '{"jsonrpc":"2.0","id":1,"method":"initialize","params":{"protocolVersion":"2025-03-26","capabilities":{},"clientInfo":{"name":"curl","version":"1.0.0"}}}'
```
</Aside>
