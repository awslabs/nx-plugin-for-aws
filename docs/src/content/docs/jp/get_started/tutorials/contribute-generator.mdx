このチュートリアルでは、実践的な例を通じてNx Plugin for AWSにジェネレータをコントリビュートする方法をガイドします。また、ジェネレータの考え方やテスト方法についての有用なリファレンスとしても活用できます。ジェネレータの作業に関するトップヒントについては、一般的なガイダンスのセクションにスキップすることもできます。

`@aws/nx-plugin`向けの新しいジェネレータを作成して、tRPC APIの新しいプロシージャを生成できるようにしましょう。

### プラグインのチェックアウト

まずプラグインをクローンします:

```bash
git clone git@github.com:awslabs/nx-plugin-for-aws.git
```

依存関係のインストールとビルド:

```bash
cd nx-plugin-for-aws
pnpm i
pnpm nx run-many --target build --all
```

### 空のジェネレータ作成

新しいジェネレータを`packages/nx-plugin/src/trpc/procedure`に作成します。

新しいジェネレータのスキャフォールディング用に専用ジェネレータを用意しているので、以下のように実行できます:

<RunGenerator generator="ts#nx-generator" requiredParameters={{ pluginProject: '@aws/nx-plugin', name: 'ts#trpc-api#procedure', directory: 'trpc/procedure', description: 'Adds a procedure to a tRPC API' }} />

以下のファイルが自動生成されます:

### ジェネレータの実装

tRPC APIにプロシージャを追加するには2つの作業が必要です:

1. 新規プロシージャ用TypeScriptファイルの作成
2. ルーターへのプロシージャ登録

#### 新規プロシージャの作成

`generateFiles`ユーティリティを使用してEJSテンプレートをレンダリングします。テンプレートは`packages/nx-plugin/src/trpc/procedure/files/procedures/__procedureNameKebabCase__.ts.template`に作成します:

```ts title="files/procedures/__procedureNameKebabCase__.ts.template"
import { publicProcedure } from '../init.js';
import { z } from 'zod';

export const <%- procedureNameCamelCase %> = publicProcedure
  .input(z.object({
    // TODO: 入力定義
  }))
  .output(z.object({
    // TODO: 出力定義
  }))
  .<%- procedureType %>(async ({ input, ctx }) => {
    // TODO: 実装
    return {};
  });
```

:::tip[テンプレート置換]
`generateFiles`はファイル/ディレクトリ名の`__<変数>__`を置換し、`.template`拡張子を除去します。テンプレート内ではEJS構文`<% ... %>`で変数を参照します。
:::

テンプレートで使用する3つの変数:

- `procedureNameCamelCase`
- `procedureNameKebabCase` 
- `procedureType`

これらを`generateFiles`に渡す必要があります。ユーザーが選択したプロジェクトのソースルートはプロジェクト設定から取得します。

ジェネレータを更新:

```ts title="procedure/generator.ts" {8-19}
import {
  generateFiles,
  joinPathFragments,
  readProjectConfiguration,
  Tree,
} from '@nx/devkit';
import { TrpcProcedureSchema } from './schema';
import { formatFilesInSubtree } from '../../utils/format';
import camelCase from 'lodash.camelcase';
import kebabCase from 'lodash.kebabcase';

export const trpcProcedureGenerator = async (
  tree: Tree,
  options: TrpcProcedureSchema,
) => {
  const projectConfig = readProjectConfiguration(tree, options.project);

  const procedureNameCamelCase = camelCase(options.procedure);
  const procedureNameKebabCase = kebabCase(options.procedure);

  generateFiles(
    tree,
    joinPathFragments(__dirname, 'files'),
    projectConfig.sourceRoot,
    {
      procedureNameCamelCase,
      procedureNameKebabCase,
      procedureType: options.type,
    },
  );

  await formatFilesInSubtree(tree);
};

export default trpcProcedureGenerator;
```

:::tip[ファイルフォーマット]
`formatFilesInSubtree`を呼び出すことで、生成/変更したファイルをユーザーのPrettier設定でフォーマットします。
:::

#### ルーターへのプロシージャ登録

次に、ジェネレータで新しいプロシージャをルーターに接続します。これはユーザーのソースコードの読み取りと更新を意味します！

[GritQL](https://docs.grit.io/)を使用して、ソースコードを宣言的に検索・変換します。`addDestructuredImport`ヘルパーは名前付きインポートを追加し、`applyGritQL`はGritQLパターンを適用してルーターのオブジェクトリテラルにプロシージャを追加します。

```ts title="procedure/generator.ts" {6, 23-36}
import {
  generateFiles,
  joinPathFragments,
  readProjectConfiguration,
  Tree,
} from '@nx/devkit';
import { TrpcProcedureSchema } from './schema';
import { formatFilesInSubtree } from '../../utils/format';
import camelCase from 'lodash.camelcase';
import kebabCase from 'lodash.kebabcase';
import { addDestructuredImport, applyGritQL } from '../../utils/ast';

export const trpcProcedureGenerator = async (
  tree: Tree,
  options: TrpcProcedureSchema,
) => {
  const projectConfig = readProjectConfiguration(tree, options.project);

  const procedureNameCamelCase = camelCase(options.procedure);
  const procedureNameKebabCase = kebabCase(options.procedure);

  generateFiles(
    tree,
    joinPathFragments(__dirname, 'files'),
    projectConfig.sourceRoot,
    {
      procedureNameCamelCase,
      procedureNameKebabCase,
      procedureType: options.type,
    },
  );

  const routerPath = joinPathFragments(projectConfig.sourceRoot, 'router.ts');

  await addDestructuredImport(
    tree,
    routerPath,
    [procedureNameCamelCase],
    `./procedures/${procedureNameKebabCase}.js`,
  );

  await applyGritQL(
    tree,
    routerPath,
    `\`router({ $props })\` => \`router({ $props, ${procedureNameCamelCase} })\` where { $props <: not contains \`${procedureNameCamelCase}\` }`,
  );

  await formatFilesInSubtree(tree);
};

export default trpcProcedureGenerator;
```

:::tip[GritQLパターン]
上記のコードスニペットで、`applyGritQL`は[GritQL](https://docs.grit.io/)パターンを使用して`router({ ... })`呼び出しを見つけ、そのオブジェクトリテラルに新しいプロシージャを追加します。`where`句は冪等性を保証します — プロシージャがすでに存在する場合は変更を行いません。

GritQLパターンは[GritQLプレイグラウンド](https://docs.grit.io/playground)で試すことができます。また、GritQL CLIをインストールしてローカルでパターンをテストすることもできます:

```bash
npm install -g @getgrit/cli
grit apply '`router({ $props })` => `router({ $props, myProcedure })`' --dry-run
```
:::

ジェネレータをコンパイルしてテスト準備:

```bash

```

### 演習課題

さらにNxジェネレータの理解を深めるための拡張案:

#### 1. ネスト化操作

ドット表記の`procedure`入力（例: `games.query`）をサポートするようジェネレータを拡張:

- 逆ドット表記のプロシージャ名生成（例: `queryGames`）
- 適切なネスト化ルーターの生成/更新

#### 2. バリデーション

tRPC APIでないプロジェクトが選択された場合の防御処理を追加。`connection`ジェネレータを参考に実装。

#### 3. ユニットテスト

ジェネレータのユニットテスト実装。基本的な流れ:

1. `createTreeUsingTsSolutionSetup()`で空ワークスペース作成
2. 既存ファイル（`project.json`や`src/router.ts`）をツリーに追加
3. テスト対象ジェネレータ実行
4. 期待通りの変更が行われたか検証

#### 4. E2Eテスト

現在の「smoke test」に新規ジェネレータを含めるよう更新。

### コントリビュートを加速する一般的なガイダンス

このセクションでは、Nx Plugin for AWSの開発作業に役立つ一般的なガイダンスを紹介します。

#### 実際のプロジェクトから逆算して作業する

新しいジェネレータを構築したり、既存のジェネレータに機能/修正を追加する際の有用な方法は、_まず実際に構築してみる_ことです。この方法により、アイデアを検証し、必要な機能を達成するために素早く反復できます。望ましい結果が定まったら、その後でジェネレータを更新します。

実際には、このプロセスは次のようになります:

<Steps>

1. 新しいワークスペースを作成

    <CreateNxWorkspaceCommand workspace="my-project" />
  
1. 新しいジェネレータ/機能/修正の前提条件となる可能性のあるジェネレータを実行
1. 変更をコミット（`git commit`）
1. 必要な変更を行い、必要に応じてテスト
1. 変更の`git diff`を使用して、Nx Plugin for AWSに対して行うべき変更を把握
1. 最終的なエンドツーエンドテストを実行（`@aws/nx-plugin`をリンク）し、ジェネレータが必要な変更を提供することを確認

</Steps>
