## タスク1: すべてをローカルで実行

フルローカルスタック — game-ui開発サーバー、ローカルGame API、ローカルAG-UI Story Agent（これがInventory MCPサーバーを起動します）— を1つのコマンドで起動します:

<NxCommands commands={["serve-local game-ui"]} />

`game-ui:serve-local`は`game-api:serve-local`と`dungeon_adventure.story:agent-serve-local`に`dependsOn`しているため、Nxは各プロジェクトのローカルサーバーを並行して起動します。モジュール1で設定した接続を通じて、これらは順にInventory MCPサーバーとDynamoDB Localを起動します — **ゲーム全体がAWS呼び出しなしでマシン上で実行されます**。コンテナエンジンが実行されていることを確認してから、ブラウザで開発サーバーを開きます。

<Aside type="tip" title="ローカルではログイン不要">
`serve-local`モードでは、`CognitoAuth`コンポーネントがCognitoログインフローをスキップします（ローカルではユーザープールが実行されていないため）。そのため、サインインせずに直接ゲームに入ります。このモジュールの最後にAWSにデプロイする際に、実際のアカウントを作成します。
</Aside>

<Aside type="caution" title="開発サーバーを起動したまま">
このモジュールの残りの作業中は開発サーバーを起動したままにしてください。以下のすべての変更がホットリロードされます。
</Aside>

## タスク4: ゲームルートの作成

2つのルートが必要です。1つはヒーローを選ぶため、もう1つはプレイするためです。どちらもshadcnコンポーネントとCopilotKitチャットを使用します。手作りのチャットUIはありません。

<Tabs>
<TabItem label="routes/index.tsx">
<E2EDiff before="dungeon-adventure/4/routes/index.tsx.old.template" after="dungeon-adventure/4/routes/index.tsx.template" lang="tsx" />

これはゲームピッカーです: 新規ゲームフォーム（shadcnの`Input` + `Button` + `Card`）と、`useInfiniteQuery`を使用した`useGameApi().games.query`によって供給される「続きから」リスト — 下部のセンチネル`<div>`が`IntersectionObserver`によって監視され、ビューにスクロールされると自動的に`fetchNextPage()`を呼び出し、見出しの横とリストの下のスピナーがローディング状態を表示します。ゲームを開始すると`(playerName, genre)`ペアを`saveGame`し（次回表示されるように）、プレイルートに移動します。
</TabItem>
<TabItem label="routes/game/$playerName.tsx">
<E2ECode path="dungeon-adventure/4/routes/game/$playerName.tsx.template" lang="tsx" />

これはプレイルートです。決定論的な`threadId`（``{player}-{genre}``を33文字にパディング — AG-UIフックはこれをそのままAgentCoreセッションIDとして送信）を構築し、`<CopilotChat agentId="agent" threadId={threadId} />`をレンダリングし、`useGameApi().inventory.query`からのインベントリを上にオーバーレイします。マウント時に、`useGameApi().actions.query({ sessionId: threadId })`がエージェントがS3に保存した会話履歴を読み取り — 履歴がある場合は — `agent.setMessages(...)`を呼び出してチャットを再ハイドレートします。履歴がない場合は、ストーリーを開始するための1つのプライミングユーザーメッセージを送信します。`agent.messages`は`useAgent({ updates: [OnMessagesChanged] })`を介してサブスクライブされているため、新しいターンごとにインベントリクエリも再フェッチされます（MCPツール呼び出しはDynamoDBを直接変更します）。

<Aside type="tip" title="`isLoading`ではなく`isFetching`を待つ">
React Queryのデフォルトキャッシュは最初の結果を5分間保持するため、再訪問時には最初のロードからの古い空のリストが最初に表示されます。`staleTime: 0` + `refetchOnMount: 'always'`を設定し、ハイドレーション`useEffect`を`!isFetching && isSuccess`でガードすることで、再プライムパスが本当に空のスレッドでのみ実行されるようにします。
</Aside>

<Aside type="tip" title="決定論的なthreadIdにより/game/$playerName?genre=...が共有可能に">
同じURLを再訪問すると — リロード後、再デプロイ後、または別のデバイスでも — 同じ`threadId`に解決されるため、エージェントは同じストーリーを語り続けます。
</Aside>

<Aside type="tip" title="パスパラメータ">
`$playerName`構文は`@tanstack/react-router`に`playerName`を[パスパラメータ](https://tanstack.com/router/v1/docs/framework/react/guide/path-params)として扱うよう指示します。`validateSearch`は`genre`を列挙型に対して厳密に型付けします。
</Aside>
</TabItem>
</Tabs>

## タスク6: ローカルとデプロイ済みコンポーネントの混在

これまでに2つの極端なケースを見てきました: すべてローカル（`serve-local`）とすべてデプロイ済み。日常的な開発では、この2つを混在させることが便利な場合がよくあります — たとえば、*実際の*デプロイ済みAPIとエージェントに対してウェブサイトコードを反復処理したり、*実際の*DynamoDBテーブルに対してAPIをローカルで実行したりします。

鍵となるのは`RUNTIME_CONFIG_APP_ID`環境変数です。プロジェクトが**`SERVE_LOCAL=true`なしで**実行される場合、生成された`resolveTableName()`（およびエージェント/APIランタイム設定ルックアップ）は、このアプリケーションIDを使用してAWS AppConfigから設定を取得します — CDK出力の`RuntimeConfigApplicationId`値です。

各ウェブサイトには、デプロイされた`runtime-config.json`（Cognitoプール、APIエンドポイント、エージェントARN）をローカル開発サーバーにダウンロードする`load:runtime-config`ターゲットがあります。便利な組み合わせをいくつか紹介します:

- **ローカルウェブサイト → デプロイ済みバックエンド。** デプロイされた設定を一度取得してから、通常の`serve`ターゲット（`serve-local`ではない）を実行して、UIがデプロイされたAPIとエージェントと通信するようにします:

  <NxCommands commands={["run game-ui:load:runtime-config"]} />
  <NxCommands commands={["serve game-ui"]} />

- **ローカルAPI → 実際のDynamoDBテーブル。** `RUNTIME_CONFIG_APP_ID`をデプロイされたアプリケーションIDに設定して通常の`serve`ターゲットを実行すると、`resolveTableName()`がDynamoDB Localの代わりにデプロイされたテーブルを解決します:

  <NxCommands env={{RUNTIME_CONFIG_APP_ID:"<RuntimeConfigApplicationId from CDK outputs>"}} commands={["serve game-api"]} />

この柔軟性により、作業中の部分を正確にテストでき、どのコンポーネント — ローカルまたはデプロイ済み — が適切かに応じて選択できます。
