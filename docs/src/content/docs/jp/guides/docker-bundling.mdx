いくつかのジェネレーター(<Link path="/guides/ts-agent">`ts#agent`</Link>や<Link path="/guides/py-agent">`py#agent`</Link>など)は、Amazon ECRにプッシュされ、AWSインフラストラクチャによって使用されるDockerイメージを生成します。このガイドでは、これらのジェネレーターが従うパターンについて説明し、他のユースケースに適用できるようにします。例えば、Amazon ECS上で<Link path="/guides/fastapi">FastAPI</Link>プロジェクトを実行したり、コンテナ化されたExpressサーバーをデプロイしたりする場合などです。

:::tip[DockerまたはFinch]
イメージのビルドに使用されるコンテナエンジンは、ワークスペース作成時に`--containers`フラグ(`docker`、`finch`、または`infer` — デフォルト — インストールされているものを自動検出)を介して選択されます。[Finch](https://runfinch.com/)は、Dockerのオープンソースのドロップイン代替品です。選択は`aws-nx-plugin.config.mts`に記録され、コンテナビルドコマンドを出力するすべてのジェネレーターに適用されます。CDKイメージアセットビルドは、`CDK_DOCKER`環境変数を介してこの選択を尊重します。
:::

## パターン

```d2
direction: right

src: src/
dockerfile: Dockerfile {
  shape: document
}
bundle: "bundle target\nRolldown / uv"
ctx: "dist/.../bundle/\n(build context)" {
  shape: page
}

src -> bundle -> ctx
dockerfile -> ctx
```

推奨されるパターンには3つの要素があります:

1. **`bundle`ターゲット** - プロジェクト上に配置され、ランタイムアーティファクトの自己完結型ディレクトリを生成します。TypeScriptの場合、これは[Rolldown](https://rolldown.rs/)によって生成されるツリーシェイクされた単一ファイルのJavaScriptバンドルです。Pythonの場合、これは[uv](https://docs.astral.sh/uv/)によって生成される`requirements.txt`とインストールされた依存関係です。
2. **最小限の`Dockerfile`** - バンドル出力をベースイメージに単純に`COPY`するだけです。バンドリングによってすでにツリーシェイキングと依存関係のインストールが処理されているため、`Dockerfile`は`npm install`や`uv sync`を実行する必要がありません。
3. **`docker`ターゲット** - `Dockerfile`をバンドル出力と一緒にコピーし(Docker buildコンテキストがランタイムに必要なファイルのみを含むようにするため)、`docker build`を実行します。

Docker buildコンテキストはプロジェクトの`dist`フォルダーに書き込まれます。その後、Infrastructure as Code(CDKまたはTerraform)がそのディレクトリを指定してイメージをECRにプッシュします。

:::tip[なぜDockerfileの外でバンドルするのか?]
バンドルステップをNxに保持する(`Dockerfile`内ではなく)ことには以下のメリットがあります:

- Nxはバンドルターゲットを**キャッシュ**できるため、繰り返しのビルドが高速になります。
- `Dockerfile`はモノレポ、プライベートレジストリ、またはビルド時のシークレットへのアクセスを必要としません。
- イメージレイヤーが小さくなります — すでにビルドされたアーティファクトの単一の`COPY`で、推移的な`node_modules`やビルドツールチェーンはありません。
:::

## TypeScript

### Bundle Target

Rolldownを呼び出す`bundle`ターゲットを設定します。<Link path="/guides/typescript-project">`ts#project`</Link>から始める場合は、`project.json`に以下を追加します:

```json
{
  "targets": {
    "bundle": {
      "cache": true,
      "executor": "nx:run-commands",
      "outputs": ["{workspaceRoot}/dist/{projectRoot}/bundle"],
      "options": {
        "command": "rolldown -c rolldown.config.ts",
        "cwd": "{projectRoot}"
      },
      "dependsOn": ["compile"]
    }
  }
}
```

そして、プロジェクトのルートに`rolldown.config.ts`を配置します:

```ts
// rolldown.config.ts
import { defineConfig } from 'rolldown';

export default defineConfig([
  {
    tsconfig: 'tsconfig.lib.json',
    input: 'src/index.ts',
    output: {
      file: '../../dist/packages/my-project/bundle/index.js',
      format: 'cjs',
      inlineDynamicImports: true,
    },
    platform: 'node',
  },
]);
```

bundleターゲットを実行して`dist/packages/my-project/bundle/index.js`を生成します:

<NxCommands commands={['bundle my-project']} />

:::tip[バンドルできない依存関係]
一部のnpmパッケージは、動的な`require`、ネイティブバインディング、またはランタイムファイルパス解決に依存しているため、バンドルできません。これらを`rolldown.config.ts`の`external`配列に追加して、ランタイムの`require()`呼び出しとして残し、代わりに`Dockerfile`内でインストールします(以下を参照):

```ts
{
  // ...
  external: ['@aws/aws-distro-opentelemetry-node-autoinstrumentation'],
}
```
:::

### Dockerfile

プロジェクトのソースディレクトリに`Dockerfile`を作成します。このファイルは、バンドルをNodeベースイメージに`COPY`し、バンドルできなかった`external`パッケージを`npm install`するだけです。`RUN npm install`ステップを`COPY`の**前に**配置することで、Dockerがインストールされた`node_modules`レイヤーをキャッシュし、依存関係リストが実際に変更されたときにのみ再実行されるようにします:

```dockerfile
FROM public.ecr.aws/docker/library/node:lts

WORKDIR /app

# Install packages that cannot be bundled (declared as "external" in rolldown.config.ts).
# Kept above the COPY so this layer is cached and only invalidated when the install list changes.
RUN npm install @aws/aws-distro-opentelemetry-node-autoinstrumentation@0.10.0

# Copy bundled application
COPY index.js /app

EXPOSE 8080

CMD ["node", "index.js"]
```

:::tip[依存関係をピン留めする]
`Dockerfile`内でインストールするすべての依存関係を正確なバージョンにピン留めしてください(上記のように) — そうしないと、次回ビルド時に`npm install`が静かに新しいリリースを取得し、ビルドされた時期によって異なる動作をするイメージが生成されます。
:::

### Docker Target

以下を実行する`docker`ターゲットを追加します:

1. `Dockerfile`をバンドル出力ディレクトリにコピー(ビルドコンテキストがバンドル + `Dockerfile`のみを含むようにするため)
2. `docker build`を実行(CDKの場合はオプション — 以下を参照)

```json
{
  "targets": {
    "docker": {
      "cache": true,
      "executor": "nx:run-commands",
      "options": {
        "commands": [
          "ncp packages/my-project/src/Dockerfile dist/packages/my-project/bundle/Dockerfile",
          "docker build --platform linux/arm64 -t my-scope-my-project:latest dist/packages/my-project/bundle"
        ],
        "parallel": false
      },
      "dependsOn": ["bundle"]
    }
  }
}
```

:::tip[クロスプラットフォームファイルコピー]
`ncp`パッケージは、クロスプラットフォームのファイル/ディレクトリコピーコマンドを提供し、`cp`(Windowsでは利用不可)を回避します。ワークスペースのルートで`pnpm add -D -w ncp`を使用してインストールします。
:::

このターゲットを実行すると、`dist/packages/my-project/bundle/`の最小限のコンテキストからビルドされた、`my-scope-my-project:latest`というタグが付けられたローカルイメージが生成されます:

<NxCommands commands={['docker my-project']} />

:::note[CDKがイメージをビルドする]
CDKのみでデプロイする場合、上記の`docker build`コマンドはオプションです — CDKの`DockerImageAsset`([Infrastructure](#infrastructure)の下に表示)がsynth時にイメージをビルドします。CDKがそれを見つけられるように、`Dockerfile`をビルドコンテキストディレクトリにコピーする必要があります。ジェネレーターは、`docker run`でイメージをスモークテストする簡単な方法を提供するために、ローカルの`docker build`ステップを維持しています。

Terraformの場合、`docker build`ステップは必須です — 以下の`null_resource`パターンは、ローカルにタグ付けされたイメージをプッシュします。
:::

## Python

### Bundle Target

ターゲットプラットフォーム用の依存関係をエクスポートしてインストールするために`uv`を使用する`bundle`ターゲットを設定します。<Link path="/guides/python-project">`py#project`</Link>ジェネレーターと<Link path="/guides/python-lambda-function">`py#lambda-function`</Link>ジェネレーターの両方がこれを設定します。ターゲット設定は次のようになります:

```json
{
  "targets": {
    "bundle-arm": {
      "cache": true,
      "executor": "nx:run-commands",
      "outputs": ["{workspaceRoot}/dist/{projectRoot}/bundle-arm"],
      "options": {
        "commands": [
          "uv export --frozen --no-dev --no-editable --project {projectRoot} --package my_project -o dist/{projectRoot}/bundle-arm/requirements.txt",
          "uv pip install -n --no-deps --no-installer-metadata --no-compile-bytecode --python-platform aarch64-manylinux_2_28 --target dist/{projectRoot}/bundle-arm -r dist/{projectRoot}/bundle-arm/requirements.txt"
        ],
        "parallel": false
      },
      "dependsOn": ["compile"]
    }
  }
}
```

:::tip[ターゲットアーキテクチャ]
Dockerイメージがx86_64上で実行される場合は、`--python-platform`を`x86_64-manylinux_2_28`に変更してください。
:::

`nx bundle my-project`を実行すると、プロジェクトのソース、その依存関係、および`requirements.txt`を含む`dist/packages/my-project/bundle-arm/`が生成されます — イメージがランタイムに必要とするすべてのものです。

### Dockerfile

`Dockerfile`は単純にバンドルをPythonベースイメージにコピーします。`uv`がすでにすべての依存関係をバンドルディレクトリにインストールしているため、イメージ内で`pip install`を実行する必要はありません:

```dockerfile
FROM public.ecr.aws/docker/library/python:3.14-slim

WORKDIR /app

# Copy bundled package (source + installed dependencies)
COPY . /app

EXPOSE 8080

ENV PYTHONPATH=/app
ENV PATH="/app/bin:${PATH}"

CMD ["python", "-m", "my_project.main"]
```

### Docker Target

`Dockerfile`をバンドル出力ディレクトリにコピーし、`docker build`を実行する`docker`ターゲットを追加します:

```json
{
  "targets": {
    "docker": {
      "cache": true,
      "executor": "nx:run-commands",
      "options": {
        "commands": [
          "rimraf dist/packages/my-project/docker",
          "make-dir dist/packages/my-project/docker",
          "ncp dist/packages/my-project/bundle-arm dist/packages/my-project/docker",
          "ncp packages/my-project/src/Dockerfile dist/packages/my-project/docker/Dockerfile",
          "docker build --platform linux/arm64 -t my-scope-my-project:latest dist/packages/my-project/docker"
        ],
        "parallel": false
      },
      "dependsOn": ["bundle-arm"]
    }
  }
}
```

これにより、出力ディレクトリがクリアされ、バンドルコンテンツと`Dockerfile`の両方が`dist/.../docker`にコピーされ、これがDocker buildコンテキストになります。

<NxCommands commands={['docker my-project']} />

## Trivyによるイメージスキャン

イメージに既知の脆弱性がないかスキャンすることは良い習慣です。このパターンに従うジェネレーターは、[Trivy](https://trivy.dev/)でビルドされたイメージをスキャンする`trivy`ターゲットを追加します。これは[ECRホストのTrivyイメージ](https://gallery.ecr.aws/aquasecurity/trivy)から実行され、`HIGH`または`CRITICAL`の検出結果があるとゼロ以外で終了します。

`docker`ターゲットに`dependsOn`する`trivy`ターゲットを追加します。これはビルドされたイメージをtarballに保存し、ワークスペース相対のバインドマウントを介してスキャンするため、`docker`と`finch`の両方で同じコマンドが機能します:

<Code lang="json" code={trivyTarget} />

各プロジェクトのイメージごとのスキャンターゲットは、ワークスペース全体の`trivy`ターゲットの下に集約され、提供される`trivy`ルートスクリプトがそれを実行します(`nx run-many --target trivy`)。

<PackageManagerShortCommand commands={['trivy']} />

:::tip[CIでTrivyを実行してください]
スキャンは意図的に`build`に組み込まれていません。これは、Trivyの脆弱性データベースが新しいCVEで継続的に更新されるため、開発中の反復作業時に不要な摩擦を引き起こす可能性があるためです。

代わりに、本番ステージへのデプロイ前に、CIパイプラインの専用ステップとして上記のコマンドを実行することをお勧めします。
:::

:::tip[キャッシュは変更されていないイメージをスキップします]
イメージはプロジェクトソースによって完全に決定されるため、`inputs`を宣言する(ここでは`bundle`ターゲットに一致する`default`と`^production`)ことで、Nxはスキャンをキャッシュし、変更されていないイメージの再スキャンをスキップできます。Trivyイメージのバージョンをピン留めして(例: <code>trivy:<TrivyVersion /></code>)、スキャンを再現可能にします。
:::

:::note[検出結果の抑制]
Trivyはプロジェクトのルートから`.trivyignore`ファイル(脆弱性IDのリスト、1行に1つ)を読み取ります。`--ignore-unfixed`は修正が利用できない脆弱性をスキップするため、スキャンは`Dockerfile`内のツールをアップグレードすることで対処できる問題でのみ失敗します。詳細については、[Trivyフィルタリングドキュメント](https://trivy.dev/latest/docs/configuration/filtering/#by-finding-ids)を参照してください。
:::

## Infrastructure

結果のビルドコンテキストディレクトリをInfrastructure as Codeに配線することは、TypeScriptとPythonの両方で同じです — ビルドコンテキストディレクトリへのパスのみが異なります(TypeScriptの場合は`dist/packages/my-project/bundle`、Pythonの場合は`dist/packages/my-project/docker`)。

<Infrastructure>
<Fragment slot="cdk">
CDKの[`DockerImageAsset`](https://docs.aws.amazon.com/cdk/api/v2/docs/aws-cdk-lib.aws_ecr_assets.DockerImageAsset.html)をビルドコンテキストディレクトリに向けて使用します。CDKはデプロイ時にイメージをビルドし、CDKアセットECRリポジトリに公開します:

```d2
direction: right

app: Application {
  bundle: "bundle target\nRolldown / uv"
  dockerfile: Dockerfile {
    shape: document
  }
  ctx: "dist/.../bundle/" {
    shape: page
  }
  bundle -> ctx
  dockerfile -> ctx
}

infra: Infrastructure {
  asset: "DockerImageAsset\ndirectory: dist/.../bundle"
}

ecr: "CDK asset ECR" {
  shape: cylinder
}

infra.asset -> app.ctx: references
infra.asset -> ecr: cdk deploy\nbuilds + pushes
```


```ts
import { DockerImageAsset, Platform } from 'aws-cdk-lib/aws-ecr-assets';
import { findWorkspaceRoot } from ':my-scope/common-constructs';
import * as path from 'path';
import * as url from 'url';

const image = new DockerImageAsset(this, 'MyImage', {
  directory: path.join(
    // Resolve from the compiled construct location to the workspace root
    findWorkspaceRoot(url.fileURLToPath(new URL(import.meta.url))),
    'dist/packages/my-project/bundle',
  ),
  platform: Platform.LINUX_ARM64,
});
```

`findWorkspaceRoot`ヘルパーは、<Link path="/guides/typescript-infrastructure">`ts#infra`</Link>ジェネレーターによって生成され、`:my-scope/common-constructs`からエクスポートされます。共有コンストラクトを使用していない場合は、`cdk`が呼び出される場所(通常はワークスペースルート)から相対的な`dist`ディレクトリへのパスをハードコードし、`findWorkspaceRoot`呼び出しを完全に省略できます。

:::note[synth前のbundleの実行]
CDKは`bundle`/`docker`ターゲットを自動的に実行しません — `cdk deploy`の前に`nx build my-project`を実行する(またはdeployターゲットが`build`に依存するように配線する)必要があります。このパターンを使用するジェネレーターは、`docker`と`bundle`を`build`の依存関係として宣言しているため、これは透過的に行われます。
:::

`DockerImageAsset`は、コンテナイメージを受け入れる任意のAWSコンストラクトで使用できます。例えば、`aws_ecs.ContainerImage.fromDockerImageAsset(image)`などです。
</Fragment>
<Fragment slot="terraform">
TerraformのAWSプロバイダーには、ファーストクラスの「Dockerイメージをビルドしてプッシュする」リソースがありません。ジェネレーターが使用するパターンは次のとおりです:

1. プロジェクトの`build`ターゲットが`docker build`を実行し、`my-scope-my-project:latest`というタグが付けられたローカルイメージを生成します。
2. イメージを保持するための`aws_ecr_repository`。
3. ECRへの認証、ローカルでビルドされたイメージのコンテンツダイジェストによる再タグ付け、プッシュを行う`local-exec`プロビジョナーを持つ`null_resource`。
4. ダウンストリームリソース(例: `aws_ecs_task_definition`)が、イミュータブルなダイジェストベースのタグでイメージを参照。

```d2
direction: down

app: Application {
  bundle: "bundle target\nRolldown / uv"
  dockerfile: Dockerfile {
    shape: document
  }
  ctx: "dist/.../bundle/" {
    shape: page
  }
  build: "docker target\ndocker build"
  img: "local image\nmy-scope-my-project:latest" {
    shape: cylinder
  }

  bundle -> ctx
  dockerfile -> ctx
  ctx -> build -> img
}

infra: Infrastructure {
  direction: right
  publish: "null_resource\nlocal-exec: docker push"
  repo: aws_ecr_repository
  publish -> repo: tag + push
}

ecr: "Amazon ECR" {
  shape: cylinder
}

infra.publish -> app.img: reads local image
infra.repo -> ecr
```


```hcl
resource "aws_ecr_repository" "repo" {
  name                 = "my-project-repository"
  image_tag_mutability = "IMMUTABLE"
  force_delete         = true
}

# Invalidate the push whenever the locally-built image digest changes
data "external" "docker_digest" {
  program = ["sh", "-c", "echo '{\"digest\":\"'$(docker inspect my-scope-my-project:latest --format '{{.Id}}')'\"}'"]
}

locals {
  # Content-based, immutable image tag derived from the local image digest
  image_tag = replace(data.external.docker_digest.result.digest, "sha256:", "")
}

resource "null_resource" "docker_publish" {
  triggers = {
    docker_digest  = data.external.docker_digest.result.digest
    repository_url = aws_ecr_repository.repo.repository_url
    image_tag      = local.image_tag
  }

  provisioner "local-exec" {
    command = <<-EOT
      aws ecr get-login-password --region ${data.aws_region.current.id} \
        | docker login --username AWS --password-stdin ${self.triggers.repository_url}
      docker tag my-scope-my-project:latest ${self.triggers.repository_url}:${self.triggers.image_tag}
      docker push ${self.triggers.repository_url}:${self.triggers.image_tag}
    EOT
  }
}
```

`data.external.docker_digest`ブロックは、ローカルイメージハッシュが変更されるたびに`null_resource`が再実行されることを保証し、意味のあるコード変更ごとに新しいプッシュをトリガーします。イメージはそのダイジェストから派生したイミュータブルなコンテンツベースのタグでプッシュされるため、ECRリポジトリは`IMMUTABLE`タグミュータビリティを使用でき、既存のタグを上書きしようとする試みを拒否できます。

:::note[apply前のbundleの実行]
`nx apply <project>`には、イメージタグ`my-scope-my-project:latest`がローカルにすでに存在している必要があります。`nx apply <project>`の前に`nx build my-project`(または`nx docker my-project`)を実行してください。
:::
