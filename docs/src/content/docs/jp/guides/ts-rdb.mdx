このジェネレーターは、[Amazon Aurora](https://aws.amazon.com/rds/aurora/) PostgreSQLを基盤とする新しいリレーショナルデータベースプロジェクトを作成します。AWS CDKまたはTerraformを使用してデータベースをプロビジョニングおよび管理するために必要なアプリケーションコードとインフラストラクチャを生成します。

生成されるインフラストラクチャには、以下が含まれます:

- 再利用可能なAuroraコンストラクトまたはモジュール
- アプリケーション固有のデータベースコンストラクトまたはモジュール
- IAM認証されたアプリケーションデータベースユーザーハンドラー
- AWS AppConfigへのランタイム設定の登録
- Prismaマイグレーションの自動デプロイ用マイグレーションハンドラー

## データベースの実装

### データモデリング

生成されたプロジェクトは、データベーススキーマの定義と型安全なクライアントの生成に[Prisma ORM](https://www.prisma.io/docs/orm)を使用します。`prisma/schema.prisma`でテーブル、リレーション、フィールドタイプをモデル化します。

`User`モデルの例:

```ts title="packages/postgres/prisma/schema.prisma"
model User {
  id          Int      @id @default(autoincrement()) 
  firstName   String
  lastName    String
}
```

詳細については、公式の[Prismaデータモデリングガイド](https://www.prisma.io/docs/orm/core-concepts/data-modeling#data-modeling-with-prisma-orm)を参照してください。

### データベースクライアントの生成

ジェネレーターは、プロジェクトをビルドするたびに型安全なTypeScript Prismaクライアントを作成するように`generate`ターゲットを自動的に構成します。クライアントは`generated/prisma`に書き込まれます(`.gitignore`に追加されます)。

また、いつでも手動でクライアントを生成できます:

<Tabs>
  <TabItem label="npm" icon="seti:npm">

```bash
npm run nx generate <your-db-package-name>
```

  </TabItem>
  <TabItem label="pnpm" icon="seti:pnpm">

```bash
pnpm nx generate <your-db-package-name>
```

  </TabItem>
  <TabItem label="yarn" icon="seti:yarn">

```bash
yarn nx generate <your-db-package-name>
```

  </TabItem>
  <TabItem label="bun" icon="seti:bun">

```bash
bun nx generate <your-db-package-name>
```

  </TabItem>
</Tabs>

`src/prisma.ts`のランタイムラッパーは以下をエクスポートします:

- `DB_PACKAGE_NAME` - AWS AppConfigの`database`ランタイム設定名前空間で使用されるキー
- `getPrisma()` - AWS AppConfigからデータベース接続設定を読み込み、IAM認証を使用してPrismaクライアントを作成します

クライアントは自動的に以下を行います:
- `RUNTIME_CONFIG_APP_ID`環境変数を使用してAWS AppConfigからデータベース設定を取得
- IAM認証用にAWS RDS Signerを介して一時的な認証トークンを生成
- 証明書検証を使用してSSL/TLS接続を管理
- 永続的なデータベース接続プールを通じて接続プーリングを処理

tRPC APIでの使用例:

```ts title="packages/api/src/procedures/list-user.ts"
import { getPrisma } from ':aws-nx-poc/postgres';
import { publicProcedure } from '../init.js';
import { ListUserOutputSchema } from '../schema/index.js';

export const listUser = publicProcedure
  .output(ListUserOutputSchema)
  .query(async () => {
    const prisma = await getPrisma();
    const users = await prisma.user.findMany({
      orderBy: { id: 'asc' },
    });

    return { users };
  });
```

### マイグレーションの作成

:::note[パッケージディレクトリからPrismaを実行]
Prisma CLIを使用する場合は、生成されたデータベースパッケージディレクトリ(例: `packages/postgres`)からコマンドを実行してください。そうすることで、Prismaが`prisma/schema.prisma`と`prisma.config.ts`を見つけることができます。
:::

Prisma Migrateは、スキーマを現在のデータベース状態と比較してSQL変更を生成するために[シャドウデータベース](https://www.prisma.io/docs/orm/prisma-migrate/understanding-prisma-migrate/shadow-database)を必要とします。

## AWSへのデプロイ

リレーショナルデータベースジェネレーターは、選択した`iacProvider`に基づいてCDKまたはTerraformのインフラストラクチャを作成します。

<Infrastructure>
<Fragment slot="cdk">
CDKコンストラクトは`common/constructs`に作成されます。使用例:

```ts title="packages/infra/src/stacks/application-stack.ts"
import { Postgres } from ':my-scope/common-constructs';

export class ApplicationStack extends Stack {
  constructor(scope: Construct, id: string, props?: StackProps) {
    super(scope, id, props);

    const vpc = new Vpc(this, 'Vpc', {
      subnetConfiguration: [
        {
          name: 'public',
          subnetType: SubnetType.PUBLIC,
        },
        {
          name: 'private_with_egress',
          subnetType: SubnetType.PRIVATE_WITH_EGRESS,
        },
        {
          name: 'private_isolated',
          subnetType: SubnetType.PRIVATE_ISOLATED,
        },
      ],
    });

    const db = new Postgres(this, 'Db', {
      vpc,
      vpcSubnets: {
        subnetType: SubnetType.PRIVATE_ISOLATED,
      }
    });
  }
}
```

これにより、RDS Proxy、管理者認証情報、アプリケーションデータベースユーザー、ランタイム設定の登録、マイグレーションハンドラーを備えたAuroraクラスターがプロビジョニングされます。直接的なインターネットアクセスを防ぐために、プライベート分離サブネットにデプロイしてください。

生成されたインフラストラクチャは、2つのデータベースユーザーを作成します:
- **管理者ユーザー** - クラスターのプロビジョニング時に作成され、認証情報はAWS Secrets Managerに保存されます
- **アプリケーションユーザー** - Lambda カスタムリソースを介して作成され、IAM認証が有効化され、アプリケーションデータベースに対する完全な権限を持ちます
</Fragment>
<Fragment slot="terraform">
Terraformモジュールは`common/terraform`に作成されます。使用例:

```hcl title="packages/infra/src/main.tf"
module "postgres" {
  source = "../../common/terraform/src/app/dbs/postgres"

  vpc_id     = module.vpc.vpc_id
  subnet_ids = module.vpc.private_isolated_subnet_ids

  tags = local.common_tags
}
```

これにより、RDS Proxy、管理者認証情報、create-db-user Lambda、ランタイム設定の登録、マイグレーションLambda、コンテナレジストリリソースを備えたAuroraクラスターがプロビジョニングされます。

生成されたインフラストラクチャは、2つのデータベースユーザーを作成します:
- **管理者ユーザー** - クラスターのプロビジョニング時に作成され、認証情報はAWS Secrets Managerに保存されます
- **アプリケーションユーザー** - Lambda関数を介して作成され、IAM認証が有効化され、アプリケーションデータベースに対する完全な権限を持ちます
</Fragment>
</Infrastructure>

### Lambda関数の接続

Lambda関数からPrismaクライアントを使用するには、以下の要件が満たされていることを確認してください:

1. データベースと同じVPCに関数をデプロイする
2. データベースポートへのネットワークアクセスを許可する
3. 生成されたランタイムユーザーにIAM `rds-db:connect`権限を付与する
4. 必要な環境変数を設定する:
   - `NODE_EXTRA_CA_CERTS=/var/runtime/ca-cert.pem` (TLS検証に必要)
   - `RUNTIME_CONFIG_APP_ID` (生成されたインフラストラクチャによって自動的に設定)
   - `AWS_REGION` (Lambda実行環境で自動的に設定)

:::note[Prisma CLI vs ランタイムクライアント]
**Prisma CLI**コマンド(`prisma migrate dev`、`prisma migrate deploy`)は、環境変数から`DATABASE_URL`を読み取ります。

**ランタイムクライアント**(`src/prisma.ts`)は、AWS AppConfigから接続設定を読み込み、IAM認証を使用します。
:::

<Infrastructure>
<Fragment slot="cdk">

```ts title="packages/infra/src/stacks/application-stack.ts"
import { RuntimeConfig } from ':my-scope/common-constructs';

const runtimeConfig = RuntimeConfig.ensure(this);

const api = new Api(this, 'Api', {
  integrations: Api.defaultIntegrations(this)
    .withDefaultOptions({
      vpc,
      vpcSubnets: {
        subnetType: SubnetType.PRIVATE_WITH_EGRESS,
      },
      environment: {
        NODE_EXTRA_CA_CERTS: '/var/runtime/ca-cert.pem',
      },
    })
    .build(),
});

Object.entries(api.integrations).forEach(([operation, integration]) => {
  db.allowDefaultPortFrom(
    integration.handler,
    `Allow ${operation} to connect to the database`,
  );
  db.grantConnect(integration.handler);
});
```

これにより、各API Lambda関数が同じVPCで実行され、AppConfigからランタイム設定を読み取り、IAM認証を使用してデータベースに接続するように構成されます。
</Fragment>
<Fragment slot="terraform">
Lambda関数をデータベースに接続する場合は、以下のモジュール出力を使用してください:

- `appconfig_application_id` - コンピュート環境に公開
- `cluster_port` - ネットワークアクセスを許可
- `database_runtime_user` - `rds-db:connect`権限を付与

コンピュートが同じVPCとプライベートサブネットで実行されることを確認し、コンピュートロールにAppConfig読み取り権限を付与してください。
</Fragment>
</Infrastructure>

### RDS Proxy

生成されたインフラストラクチャには、デフォルトでRDS Proxyが含まれており、アプリケーションとAuroraクラスターの間に配置されます。RDS Proxyは、いくつかの利点を提供します:

- **接続プーリング** - アプリケーションインスタンス間で共有できるデータベース接続のプールを維持し、新しい接続を確立するオーバーヘッドを削減します
- **接続の回復力** - Auroraインスタンスの交換やメンテナンス中のフェイルオーバーと再接続を自動的に処理します
- **IAM認証** - IAMベースのデータベース認証をサポートし、アプリケーションコードでデータベース認証情報を管理する必要がなくなります
- **セキュリティの向上** - すべての接続にTLS暗号化を強制します

<Infrastructure>
<Fragment slot="cdk">

デフォルトでは、RDS Proxyが有効になっています。生成されたランタイムクライアント(`getPrisma()`)は、プロキシエンドポイントを介して自動的に接続します。必要に応じて無効にできます:

```ts title="packages/infra/src/stacks/application-stack.ts"
const db = new Postgres(this, 'Db', {
  ...
  enableRdsProxy: false,
});
```

RDS Proxyが無効になっている場合、アプリケーションはAuroraクラスターエンドポイントに直接接続します。
</Fragment>
<Fragment slot="terraform">

デフォルトでは、RDS Proxyが有効になっています。生成されたランタイムクライアント(`getPrisma()`)は、プロキシエンドポイントを介して自動的に接続します。必要に応じて無効にできます:

```hcl title="packages/infra/src/main.tf"
module "postgres" {
  source = "../../common/terraform/src/app/dbs/postgres"
  ...
  enable_rds_proxy = false
}
```

RDS Proxyが無効になっている場合、アプリケーションはAuroraクラスターエンドポイントに直接接続します。プロキシは、TLS暗号化を要求し、IAM認証を使用するように構成されています。
</Fragment>
</Infrastructure>

## データベースアーキテクチャのカスタマイズ

生成されたインフラストラクチャは、ワークロードの要件に合わせてカスタマイズできます。以下の例は、利用可能な一般的なカスタマイズオプションのいくつかを示しています。

### クラスターインスタンス

Auroraクラスターのwriterおよびreaderインスタンスを構成します。

<Infrastructure>
<Fragment slot="cdk">

```ts title="packages/infra/src/stacks/application-stack.ts"
const db = new Postgres(this, 'Db', {
  ...
  writer: ClusterInstance.serverlessV2('writer'),
  readers: [ClusterInstance.serverlessV2('reader')],
});
```
</Fragment>
<Fragment slot="terraform">

```hcl title="packages/infra/src/main.tf"
module "postgres" {
  source = "../../common/terraform/src/app/dbs/postgres"

```
</Fragment>
</Infrastructure>
