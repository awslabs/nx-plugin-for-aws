`connection`ジェネレーターは、<Link path="guides/py-agent">Python Agent</Link>を<Link path="guides/agentcore-gateway">AgentCore Gateway</Link>に接続できます。

このジェネレーターは、デプロイ時にGatewayに対してIAM SigV4（`httpx`リクエスト署名経由）で認証し、ローカル実行時には接続されたMCPサーバーに直接多重化するようにエージェントを配線します。

## 前提条件

このジェネレーターを使用する前に、以下を確認してください：

1. <Link path="guides/py-agent">Agent</Link>コンポーネント（`infra: agentcore`）を持つPythonプロジェクト
2. <Link path="guides/agentcore-gateway">`agentcore-gateway`</Link>プロジェクト

## ジェネレーターの出力

ジェネレーターは、共有のcore-gatewayモジュールを`agent_connection` Pythonプロジェクトに出力し、さらにGatewayごとのラッパーを追加し、エージェントを変更します：

<FileTree>

- packages/common/agent\_connection
  - \<scope>\_agent\_connection
    - core/
      - agentcore\_gateway\_mcp\_client.py `httpx`を使用したSigV4 MCPクライアント
      - agentcore\_gateway\_mcp\_local\_client.py 接続されたMCPサーバー用のローカル多重化クライアント
    - app/
      - \<gateway\_snake>\_client.py `ATTACHED_MCP_SERVERS`リストを持つGatewayごとのラッパー
    - \_\_init\_\_.py Gatewayクライアントを再エクスポート

</FileTree>

さらに、ジェネレーターは以下を行います：

- エージェントの`agent.py`を変更して、Gatewayクライアントをインポートし、`with`ブロックで入り、そのツールを`tools`に登録します
- エージェントのワークスペース依存関係として`agent_connection`を追加します
- エージェントの`<agent>-serve-local`ターゲットをGatewayの`<gateway>-serve-local`アグリゲーターに依存するように配線します

## 接続されたGatewayの使用

ジェネレーターは、エージェントの`agent.py`をGatewayクライアントを使用するように変換します：

```python title="packages/example/example/my_agent/agent.py" {4,8,9-13}
from contextlib import contextmanager
from strands import Agent

from my_scope_agent_connection import MyGatewayClient

@contextmanager
def get_agent():
    my_gateway = MyGatewayClient.create()
    with (
        my_gateway,
    ):
        yield Agent(
            system_prompt="...",
            tools=[*my_gateway.list_tools_sync()],
        )
```

`MyGatewayClient.create()`は、`list_tools_sync()`がGatewayを通じて利用可能なすべてのツールを生成する、単一のコンテキスト管理可能なクライアントを返します：

- **デプロイモード**（`SERVE_LOCAL`未設定）：GatewayのMCPエンドポイントを指す単一の`MCPClient`、SigV4署名付き。
- **ローカルモード**（`SERVE_LOCAL=true`）：接続された各MCPサーバーにファンアウトし、それらのツールの和集合を提示する`AgentCoreGatewayLocalMultiplexClient`。

セッションIDは下流のMCPサーバーに自動的に伝播されます：基盤となる`httpx`トランスポートは`get_current_session_id()`（Strandsランタイムミドルウェアによって設定）を読み取り、それを`X-Amzn-Bedrock-AgentCore-Runtime-Session-Id`ヘッダーとしてGatewayに転送します。

## ローカル開発

ジェネレーターは、エージェントの`serve-local`ターゲットを次のように設定します：

1. 接続されたGatewayに接続されているすべてのMCPサーバーを起動
2. `SERVE_LOCAL=true`を設定して、生成されたクライアントがデプロイされたGatewayを呼び出す代わりに、それらのローカルMCPサーバーにファンアウトするようにします

エージェントをローカルで実行するには：

<NxCommands commands={["<agent-name>-serve-local <project-name>"]} />

エージェントをローカルで**デプロイされたGatewayに対して**実行するには（例えば、Cedarポリシーを実行するため）、エージェントの`serve`ターゲットを使用します。`SERVE_LOCAL`が設定されていない場合、クライアントはランタイム設定からデプロイされたGateway URLを解決し、ローカルのAWS認証情報でリクエストにSigV4署名します：

<NxCommands commands={["<agent-name>-serve <project-name>"]} />

### ローカルの忠実性

ローカル多重化モードはGateway自体をバイパスするため：

- **Cedarポリシー評価なし。** すべてのツールは、ポリシーに関係なくエージェントに表示されます。デプロイされたGatewayに対してポリシーを実行するには、`serve`ターゲットを使用してください。
- **ツール名のプレフィックスは保持されます。** 各ローカルMCPサーバーのツールは`<target-name>___<tool-name>`として公開され、デプロイされたGatewayが出力するものと一致します。これにより、エージェントのシステムプロンプトと参照するCedarアクション名が、ローカル実行とデプロイ実行の間で一貫性を保ちます。
