## Tarea 1: Implementar la Game API

Implementaremos las siguientes APIs en esta sección:

1. `saveGame` - crear o actualizar un juego.
2. `queryGames` - devolver una lista paginada de juegos guardados anteriormente.
3. `queryInventory` - devolver una lista paginada de ítems en el inventario de un jugador.
4. `queryActions` - devolver el historial de conversación para un juego dado.

:::note[Dónde vive cada pieza de estado]
- `games` e `inventory` están en **DynamoDB** (vía ElectroDB) — las lambdas de la Game API y el servidor MCP de Inventario los leen/escriben.
- `actions` (transcripción del chat) está en **S3** — el `S3SessionManager` del Story Agent escribe un blob JSON por turno, y `queryActions` en la Game API los lee de vuelta para que la UI pueda reproducir el historial al recargar la página. No hay un procedimiento `saveAction`; el agente es el único escritor.
:::

### Modelado de entidades

Este es el diagrama ER para nuestra aplicación.


Modelaremos nuestros datos con [ElectroDB](https://electrodb.dev/en/core-concepts/introduction/) sobre la tabla `DungeonDb` que creamos con el generador `ts#dynamodb` en el <Link path="get_started/tutorials/dungeon-game/1">Módulo 1</Link>. Observa que las entidades usan los helpers `getDynamoDBClient()` y `resolveTableName()` del `client.ts` generado del proyecto — esto es lo que permite que el mismo código de entidad se comunique con DynamoDB Local durante el desarrollo y con la tabla desplegada en AWS.

Una ventaja de este enfoque es que a medida que crece el número de tablas u otros identificadores de infraestructura, solo necesitas una única variable de entorno `RUNTIME_CONFIG_APP_ID` cuando ejecutas contra infraestructura desplegada.
:::

## Tarea 4: Probar la Game API localmente

No es necesario desplegar en AWS para probar nuestra API — el target `serve-local` ejecuta la Game API contra [DynamoDB Local](https://docs.aws.amazon.com/amazondynamodb/latest/developerguide/DynamoDBLocal.html). Como conectamos la Game API al proyecto `DungeonDb` en el Módulo 1, este target también inicia DynamoDB Local automáticamente.

Primero, corrige cualquier problema de linting:

<PackageManagerShortCommand commands={["lint"]} />

Luego construye la base de código:

<PackageManagerShortCommand commands={["build"]} />

<Aside type="note" title="Workspace fuera de sincronización">
Agregar procedimientos y las nuevas referencias de proyecto significa que Nx puede volver a solicitar que "El workspace está fuera de sincronización" (como en el <Link path="get_started/tutorials/dungeon-game/1">Módulo 1</Link>). Selecciona **Yes, sync the changes and run the tasks** para continuar. Verás este mensaje cada vez que cambien las referencias de proyecto.
</Aside>

### Iniciar el servidor local

Inicia la Game API localmente con el target `serve-local`. Asegúrate de que tu motor de contenedores (Docker o Finch) esté ejecutándose primero, ya que esto también arranca DynamoDB Local:

<NxCommands commands={["serve-local game-api"]} />

El target `serve-local` establece `SERVE_LOCAL=true`, por lo que `resolveTableName()` devuelve el nombre de la tabla local y `getDynamoDBClient()` se conecta a DynamoDB Local en lugar de AWS — sin despliegue y sin `RUNTIME_CONFIG_APP_ID` requerido.

<Aside type="tip" title="Primera ejecución">
La primera ejecución descarga la imagen del contenedor de DynamoDB Local, lo que puede tomar un momento. Deja el servidor ejecutándose en esta terminal y abre una segunda terminal para los comandos `curl` a continuación.
</Aside>

### Probar la API

Una vez que tu servidor esté en funcionamiento, consulta la lista (vacía) de juegos:

```bash
curl -X GET 'http://localhost:2022/games.query?input=%7B%7D'
```

:::note[Codificación de URL]
El `%7B%7D` que pasamos para probar la API es un objeto JSON vacío codificado en URI (`{}`).
:::

Verás una lista vacía:

```json
{"result":{"data":{"items":[],"cursor":null}}}
```

Ahora guarda un juego:

```bash
curl -X POST 'http://localhost:2022/games.save' \
  -H 'Content-Type: application/json' \
  -d '{"playerName":"Alice","genre":"zombie"}'
```

El guardado devuelve el juego persistido (con el timestamp `lastUpdated` que la entidad establece por ti):

```json
{"result":{"data":{"playerName":"Alice","genre":"zombie","lastUpdated":"..."}}}
```

Consulta nuevamente para confirmar que está persistido en DynamoDB Local:

```bash
curl -X GET 'http://localhost:2022/games.query?input=%7B%7D'
```

Esta respuesta ahora incluye el juego guardado:

```json
{"result":{"data":{"items":[{"playerName":"Alice","genre":"zombie","lastUpdated":"..."}],"cursor":null}}}
```

Puedes detener el servidor local (`Ctrl+C`) una vez que hayas terminado.

## Tarea 5: Probar el servidor MCP de Inventario localmente

Podemos probar las herramientas del servidor MCP con el [MCP Inspector](https://github.com/modelcontextprotocol/inspector) usando el target `mcp-server-inspect` generado. El inspector se conecta al servidor a través de STDIO, por lo que ejecutamos DynamoDB Local por separado y le indicamos al servidor que lo use.

En una terminal, inicia DynamoDB Local:

<NxCommands commands={["serve-local dungeon-db"]} />

En una segunda terminal, inicia el inspector:

<NxCommands commands={["mcp-server-inspect inventory"]} />

Esto inicia el MCP Inspector en `http://localhost:6274`. En la UI del inspector, antes de conectar, agrega una variable de entorno `SERVE_LOCAL` con el valor `true` para que las herramientas del servidor se comuniquen con DynamoDB Local, luego haz clic en **Connect**.

<Aside type="tip" title="¿Por qué establecer SERVE_LOCAL?">
A diferencia de `serve-local`, el target `mcp-server-inspect` ejecuta el servidor STDIO simple, por lo que no establece `SERVE_LOCAL` por ti. Establecerlo en el entorno del inspector hace que `getDynamoDBClient()` y `resolveTableName()` se conecten a la tabla local en lugar de AWS.
</Aside>

Cambia a la pestaña **Tools**, haz clic en **List Tools**, y prueba `add-to-inventory` (por ejemplo, `playerName: Alice`, `itemName: Rusty Sword`, `emoji: ⚔️`) seguido de `list-inventory-items` para verlo persistido en DynamoDB Local. Detén ambas terminales (`Ctrl+C`) cuando hayas terminado.

<Aside type="tip" title="¿Prefieres el transporte HTTP?">
Si prefieres controlar el servidor a través de HTTP (por ejemplo, para scripting), el target `mcp-server-serve-local` ejecuta el transporte HTTP Streamable con `SERVE_LOCAL=true` establecido por ti, e inicia DynamoDB Local automáticamente:

<NxCommands commands={["run @dungeon-adventure/inventory:mcp-server-serve-local"]} />

Luego inicializa una sesión contra `http://localhost:8000/mcp`:
```bash
curl -X POST http://localhost:8000/mcp \
  -H 'Content-Type: application/json' \
  -H 'Accept: application/json, text/event-stream' \
  -d '{"jsonrpc":"2.0","id":1,"method":"initialize","params":{"protocolVersion":"2025-03-26","capabilities":{},"clientInfo":{"name":"curl","version":"1.0.0"}}}'
```
</Aside>

¡Felicidades, has construido y probado tu primera API tRPC y servidor MCP contra una tabla DynamoDB local!  🎉🎉🎉
