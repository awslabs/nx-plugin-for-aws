No todos los proyectos comienzan con `create-nx-workspace --preset=@aws/nx-plugin`. Si ya tienes un [espacio de trabajo Nx](https://nx.dev) — o un monorepo al que puedes agregar Nx — puedes adoptar `@aws/nx-plugin` de forma incremental sin recrear tu proyecto.

## Requisitos previos

- **Node.js** — una [versión LTS activa](https://nodejs.org/en/about/previous-releases).
- **Un gestor de paquetes** — se recomienda `pnpm` y es el mejor soportado; `npm`, `yarn` y `bun` también funcionan.
- **TypeScript 5 o 6** — si tu espacio de trabajo fija `typescript` 7 (compilador nativo), degrada por ahora: los ejecutores de Nx y los generadores del plugin dependen de la API JS de TypeScript que TS 7 elimina.
- **Un espacio de trabajo Nx.** Si tu proyecto aún no es un espacio de trabajo Nx, agrega Nx primero — consulta [Agregar Nx a un proyecto no-Nx](#agregar-nx-a-un-proyecto-no-nx). Este es un paso que Nx mismo posee; el plugin se construye sobre un espacio de trabajo Nx funcional.

## Agregar el plugin

Usa [`nx add`](https://nx.dev/nx-api/nx/documents/add), que instala el plugin en una versión compatible con tu instalación de Nx y luego ejecuta su generador `init` para configurar tu espacio de trabajo:

<NxCommands commands={['add @aws/nx-plugin']} />

`nx add` ejecuta el generador `init` del plugin automáticamente. Para ejecutarlo nuevamente más tarde (por ejemplo, para cambiar el proveedor de IaC), invócalo directamente:

<RunGenerator generator="init" />

Luego verifica la configuración generando un proyecto y construyéndolo:

<Steps>

1. <RunGenerator generator="ts#project" requiredParameters={{ name: 'my-lib' }} />

2. <NxCommands commands={['sync']} />

3. <NxCommands commands={['run-many --target build --all']} />

</Steps>

### Qué configura el generador `init`

El generador `init` realiza los cambios determinísticos que un espacio de trabajo necesita para ejecutar los generadores del plugin. Es **idempotente** — ejecutarlo nuevamente no hace nada — y preserva el formato de módulo de tu espacio de trabajo: un espacio de trabajo cuyo `package.json` raíz tiene `type: "module"` permanece ESM, cualquier otra cosa permanece CommonJS, y el código generado sigue el mismo patrón. **No** reescribe un `tsconfig.base.json` existente, migra tu gestor de paquetes, ni reestructura tus proyectos existentes.

Realiza lo siguiente:

- Crea `aws-nx-plugin.config.mts` registrando tu <Link path="guides/typescript-infrastructure">proveedor de infraestructura como código</Link> elegido (CDK o Terraform) y motor de contenedores. Los generadores leen `iac.provider` de este archivo.
- Registra los generadores de sincronización (`@nx/js:typescript-sync` y `@aws/nx-plugin:ts#sync`) en el objetivo `compile` en `nx.json`, para que las referencias de proyecto TypeScript se mantengan sincronizadas a medida que agregas proyectos.
- Asegura que existan un `tsconfig.json` raíz y un `tsconfig.base.json` (la sincronización de TypeScript de Nx y los proyectos del plugin los necesitan). Los archivos faltantes se crean; **los existentes se dejan intactos** para que no perturben tus otros proyectos.
- Para espacios de trabajo `pnpm`, permite los scripts de construcción que las dependencias del plugin necesitan (`@swc/core`, `esbuild`, `nx`, `sharp`) en `pnpm-workspace.yaml`.
- Agrega scripts de conveniencia (sin sobrescribir ninguno de los tuyos) al `package.json` raíz, además de las dependencias de desarrollo `nx`, `@nx/js`, `@nx/workspace`, `typescript` y Biome — `nx` fijado a una sola versión para evitar el [bloqueo de nx duplicado](#nx-sync-se-cuelga-o-plugin-worker--exited-before-the-connection-was-established) — y escribe un `biome.json` predeterminado si no existe ninguno. Un espacio de trabajo ESM mantiene su `type: "module"`; el campo `type` de un espacio de trabajo CommonJS (o su ausencia) se deja tal cual.
- Configura el <Link path="get_started/building-with-ai">servidor MCP</Link> para agentes de codificación compatibles (a menos que esté deshabilitado).

## Agregar Nx a un proyecto no-Nx

Si tu proyecto aún no usa Nx, agrégalo primero con [`nx init`](https://nx.dev/recipes/adopting-nx/adding-to-existing-project). Funciona en un paquete simple, un espacio de trabajo npm/pnpm/yarn, un Turborepo o un monorepo Lerna:

```bash
npx nx@latest init
```

`nx init` es la propia herramienta de Nx para adoptar Nx — detecta tus herramientas, escribe `nx.json`, y conecta tus scripts existentes en Nx. Sigue las indicaciones. Cuando se ejecuta de forma no interactiva (CI o un agente de IA), pasa `--interactive=false`; si Nx detecta plugins de herramientas para tu stack (por ejemplo, `@nx/vite`) también pedirá una elección explícita — pasa `--plugins=skip` para diferir (los plugins se pueden agregar más tarde con `nx add`) o `--plugins=all` para aceptarlos. Una vez que se complete y tu espacio de trabajo se construya bajo Nx, [agrega el plugin](#agregar-el-plugin) como se indicó anteriormente.

### Proyectos no-Node (Python, Go, Java, Rust, …)

Nx y el plugin se distribuyen como paquetes npm, por lo que un proyecto sin herramientas de Node.js necesita un `package.json` raíz mínimo antes de que `nx init` pueda hacer algo útil:

```json
{
  "name": "my-project",
  "private": true,
  "type": "module"
}
```

Crea ese archivo, luego ejecuta `nx init` y [agrega el plugin](#agregar-el-plugin) como de costumbre. Tus herramientas de lenguaje existentes no se tocan — los proyectos Nx generados por el plugin (incluidos los <Link path="guides/py-project">proyectos Python</Link>) viven junto a tu código existente, y puedes conectar tu construcción existente en Nx de forma incremental.

### Proyectos de un solo paquete

El plugin está diseñado para monorepos — genera cada proyecto en su propio directorio bajo `packages/`. Si estás comenzando desde un **proyecto de un solo paquete** (un `package.json` en la raíz con tu código fuente directamente debajo, sin espacios de trabajo), mueve tu paquete existente a `packages/` antes de adoptar el plugin para que tu proyecto se sitúe junto a los que genera el plugin:

<Steps>

1. Crea un directorio `packages/<tu-paquete>/` y mueve tu código fuente, `package.json` y `tsconfig.json` en él.
2. Declara el espacio de trabajo: para `pnpm`, agrega un `pnpm-workspace.yaml` con `packages: ['packages/*']`; para `npm`/`yarn`/`bun`, agrega `"workspaces": ["packages/*"]` al `package.json` raíz. El `package.json` raíz se convierte en un manifiesto de espacio de trabajo en lugar de un proyecto en sí mismo.
3. Ejecuta `nx init` (si no lo has hecho), luego [agrega el plugin](#agregar-el-plugin).

</Steps>

Mantener la raíz como un manifiesto de espacio de trabajo (en lugar de un proyecto) es lo que permite que `nx run-many --target build --all` construya cada proyecto sin que una tarea se refiera a sí misma — consulta [la nota de recursión](#recursive-task-invocation-detected) a continuación.

<Aside type="note" title="Actualizar más tarde">
Para actualizar el plugin (y obtener cualquier migración de código automatizada que incluya), usa [`nx migrate`](https://nx.dev/features/automate-updating-dependencies) — `nx migrate @aws/nx-plugin` — en lugar de aumentar la versión manualmente.
</Aside>

<Aside type="note" title="Gestor de paquetes y diseño">
Los proyectos del plugin se generan bajo un directorio `packages/`. `pnpm` lee los globs del espacio de trabajo desde `pnpm-workspace.yaml`; otros gestores de paquetes leen el campo `workspaces` del `package.json` raíz — `init` actualiza el que use tu espacio de trabajo. Si tu espacio de trabajo usa un diseño diferente (como `apps/` y `libs/`), los proyectos del plugin viven bajo `packages/` junto a tu estructura existente.
</Aside>

## Adopción incremental

No tienes que convertir todo tu espacio de trabajo de una vez. Un buen camino es:

<Steps>

1. Agrega el plugin, luego genera un solo proyecto (por ejemplo, un <Link path="guides/typescript-project">proyecto TypeScript</Link> o una <Link path="guides/ts-api">API</Link>) y constrúyelo.
2. Agrega más proyectos según los necesites — sitios web, APIs, infraestructura — usando las <Link path="guides/index">guías de generadores</Link>.
3. Migra tu código existente al espacio de trabajo con el tiempo, si lo deseas. Los proyectos del plugin se sitúan junto a los existentes; no requieren que muevas nada.

</Steps>

## Solución de problemas

La mayoría de la fricción de adopción proviene de la configuración de Nx en sí misma en lugar del plugin. Una vez que `nx add @aws/nx-plugin` se haya ejecutado, estos son los desajustes que aún podrías ver.

#### `The workspace is out of sync`

Las referencias de proyecto TypeScript necesitan sincronización. `init` registra los generadores de sincronización; ejecuta:

<NxCommands commands={['sync']} />

#### `ERR_PNPM_IGNORED_BUILDS` durante la instalación

pnpm omite los scripts de instalación para paquetes que no están en la lista de permitidos. `init` permite los que las herramientas del plugin necesitan (`@swc/core`, `esbuild`, `nx`, `sharp`) en `pnpm-workspace.yaml`, pero el bloqueo puede activarse durante `nx add @aws/nx-plugin` en sí — antes de que `init` se haya ejecutado. Ejecuta `pnpm approve-builds` y aprueba los paquetes listados (o agrégalos bajo `allowBuilds:`), luego vuelve a ejecutar la instalación.

#### `nx sync` se cuelga, o `plugin worker ... exited before the connection was established`

Dos versiones diferentes de `nx` están presentes en el mismo árbol node_modules (ejecuta `npm ls nx` para confirmar) — su IPC de plugin-worker se bloquea. El generador `init` fija la devDependency `nx` raíz a la versión que los propios paquetes `@nx/*` del plugin resuelven, pero una instalación posterior de otro paquete `@nx/*` en una versión de parche diferente puede reintroducir el desajuste. Alinea cada entrada `nx` y `@nx/*` en tu `package.json` raíz a una sola versión y reinstala.

#### `Recursive task invocation detected`

Tu `package.json` **raíz** está registrado como un proyecto Nx (tiene una clave `"nx"`, que `nx init` agrega para un repositorio de un solo paquete) *y* lleva un script `build` de `nx run-many --target build`. Nx entonces infiere un objetivo `build` en la raíz que se llama a sí mismo. Mueve tu código fuente a `packages/<tu-paquete>/` para que la raíz se convierta en un manifiesto de espacio de trabajo en lugar de un proyecto — consulta [Proyectos de un solo paquete](#proyectos-de-un-solo-paquete). (Los propios presets independientes de Nx evitan esto estableciendo `"nx": { "includedScripts": [] }` en el paquete raíz; puedes hacer lo mismo como una alternativa más rápida.)

#### Errores de TypeScript en tus proyectos **existentes** después de agregar el plugin (`TS6059`, `TS7016`, `TS5011`, `NG4006`)

Los propios proyectos generados del plugin llevan la configuración de TypeScript que necesitan en su `tsconfig.lib.json` por proyecto, por lo que se construyen independientemente de tu `tsconfig.base.json`. Estos errores aparecen en cambio en un **proyecto preexistente tuyo** que hereda un `tsconfig.base.json` que `init` creó (con `composite`/`emitDeclarationOnly`/`nodenext`) pero no es compatible con esas configuraciones — por ejemplo, el compilador Angular rechaza `emitDeclarationOnly` (`NG4006`), o un proyecto que establece `outDir` sin un `rootDir` ahora necesita uno (`TS5011`). Agrega sobrescrituras de `tsconfig` por proyecto redeclarando las opciones conflictivas para esos proyectos (por ejemplo, `"rootDir": "src"`), o apunta los proyectos del plugin y tus proyectos a configuraciones base separadas.

`init` nunca reescribe un `tsconfig.base.json` existente, precisamente para que no pueda romper silenciosamente los proyectos que heredan de él — resolver estos desajustes es una decisión que solo tú puedes tomar para tu código base.

<Aside type="note" title="ESM y CommonJS">
El formato de módulo es a nivel de espacio de trabajo, leído del campo `type` del `package.json` raíz: `type: "module"` significa ESM, cualquier otra cosa (incluida la ausencia del campo `type`) significa CommonJS. `init` preserva el que use tu espacio de trabajo, y los generadores TypeScript del plugin emiten código coincidente. Si más tarde quieres cambiar, cambia el `type` raíz — pero esa es una migración de tu código existente, no algo que `init` haga por ti.
</Aside>

</Aside>
