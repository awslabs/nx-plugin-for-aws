El generador `connection` puede registrar un servidor MCP (ya sea <Link path="guides/ts-mcp-server">TypeScript</Link> o <Link path="guides/py-mcp-server">Python</Link>) como destino de un <Link path="guides/agentcore-gateway">AgentCore Gateway</Link>.

Una vez conectado, el Gateway agrega las herramientas del servidor MCP en su único endpoint MCP, evalúa las llamadas contra su motor de políticas Cedar y firma el tráfico saliente al servidor MCP con IAM SigV4.

## Requisitos previos

Antes de usar este generador, asegúrate de tener:

1. Un proyecto <Link path="guides/agentcore-gateway">`agentcore-gateway`</Link>
2. Un componente de servidor MCP (<Link path="guides/ts-mcp-server">`ts#mcp-server`</Link> o <Link path="guides/py-mcp-server">`py#mcp-server`</Link>) creado con `infra: agentcore` y `auth: iam`

El Gateway debe tener `protocol: mcp` y el servidor MCP debe tener `auth: iam` — el generador valida ambos. Los servidores MCP sin IAM no pueden adjuntarse porque el Gateway firma el tráfico saliente con SigV4.

## Salida del generador

El generador conecta proyectos existentes en lugar de emitir nuevos archivos fuente. Los siguientes archivos se modifican:

<FileTree>

- packages/\<gateway>
  - project.json `<gateway>-serve-local` gana una dependencia en el `<mcp>-serve-local` del servidor MCP
- packages/common/agent-connection (TypeScript, si existe un cliente de gateway TS)
  - src/app
    - \<gateway-kebab>-client.ts `ATTACHED_MCP_SERVERS` actualizado para incluir el servidor MCP
- packages/common/agent\_connection (Python, si existe un cliente de gateway Py)
  - \<scope>\_agent\_connection/app
    - \<gateway\_snake>\_client.py `ATTACHED_MCP_SERVERS` actualizado para incluir el servidor MCP

</FileTree>

El objetivo agregador `<gateway>-serve-local` del proyecto Gateway gana una dependencia en el objetivo `<mcp>-serve-local` del servidor MCP, por lo que ejecutar el Gateway localmente también inicia el servidor MCP. Cualquier archivo `<Gateway>Client.ts` / `_client.py` existente (emitido por los generadores de conexión de gateway de <Link path="guides/connection/ts-agent-gateway">TypeScript</Link> o <Link path="guides/connection/py-agent-gateway">Python</Link>) tiene su lista `ATTACHED_MCP_SERVERS` actualizada para incluir este servidor MCP, de modo que la multiplexación local lo detecte.

## Agregar el destino del servidor MCP a tu stack

El generador **no puede** conectar automáticamente el destino del servidor MCP en tu infraestructura porque no sabe qué stack o módulo instancia el Gateway. Agrega una única llamada a `gateway.addMcpServer(server)` tú mismo.

<Infrastructure>
<Fragment slot="cdk">
En el stack donde instancias el Gateway, registra el servidor MCP como destino:

```ts title="packages/infra/src/stacks/application-stack.ts" {5-6}
const myMcpServer = new MyMcpServer(this, 'MyMcpServer');
const myGateway = new MyGateway(this, 'MyGateway');

// Register the MCP server as a target of the Gateway. The target name
// defaults to the MCP server construct's id in kebab-case (`my-mcp-server`).
myGateway.addMcpServer(myMcpServer);
```

El nombre del destino del Gateway (el id del constructo en kebab-case por defecto) se usa como prefijo para los nombres de acción de Cedar — el formato de acción es ``AgentCore::Action::"<targetName>___<toolName>"``. Consulta la <Link path="guides/agentcore-gateway">sección Escribir políticas</Link>. Mantén el nombre del destino corto y estable; cambiarlo más tarde invalida cualquier política de Cedar que haga referencia al nombre antiguo.

Para anular el nombre de destino predeterminado, pasa `gatewayTargetName`:

```ts
myGateway.addMcpServer(myMcpServer, { gatewayTargetName: 'my-mcp' });
```

El constructo configura el destino con `iamCredentialProvider.service = 'bedrock-agentcore'` para que el Gateway firme las llamadas salientes usando su propio rol de ejecución.
</Fragment>
<Fragment slot="terraform">
En el archivo Terraform donde instancias el Gateway, conecta el destino del servidor MCP:

```hcl title="packages/infra/src/main.tf" {10-16}
module "my_mcp_server" {
  source = "../../common/terraform/src/app/mcp-servers/my-mcp-server"
}

module "my_gateway" {
  source = "../../common/terraform/src/app/gateways/my-gateway"
}

# Register the MCP server as a target of the Gateway
resource "aws_bedrockagentcore_gateway_target" "my_mcp_server" {
  gateway_identifier    = module.my_gateway.gateway_id
  name                  = "my-mcp-server"
  mcp_server_runtime_arn = module.my_mcp_server.agent_core_runtime_arn

  credential_provider_configuration {
    credential_provider_type = "GATEWAY_IAM_ROLE"
  }
}
```

El `name` del destino (`my-mcp-server` arriba) se usa como prefijo para los nombres de acción de Cedar — consulta la <Link path="guides/agentcore-gateway">sección Escribir políticas</Link>.
</Fragment>
</Infrastructure>

## Desarrollo local

Ejecutar el Gateway localmente con:

<NxCommands commands={["<gateway-name>-serve-local"]} />

inicia cada servidor MCP adjunto en su puerto local asignado. Los agentes conectados al Gateway a través de los generadores de conexión de gateway de <Link path="guides/connection/ts-agent-gateway">TypeScript</Link> o <Link path="guides/connection/py-agent-gateway">Python</Link> multiplexan directamente a los servidores MCP usando la ruta de respaldo local de su cliente.

:::caution[Fidelidad local]
El desarrollo local **omite el Gateway por completo** — el cliente de gateway generado detecta `SERVE_LOCAL=true` y se conecta directamente a cada servidor MCP adjunto a través de HTTP simple. Como consecuencia, **las políticas de Cedar no se evalúan localmente**. Cada agente ve cada herramienta en cada servidor MCP adjunto. Para ejercitar las políticas de Cedar, ejecuta el objetivo `serve` del agente en su lugar (consulta las guías de conexión de agente de <Link path="guides/connection/ts-agent-gateway">TypeScript</Link> / <Link path="guides/connection/py-agent-gateway">Python</Link>) para que el agente que se ejecuta localmente llame al Gateway desplegado.

Los nombres de las herramientas todavía tienen el prefijo local como `<target-name>___<tool-name>` para coincidir con el espacio de nombres del Gateway desplegado, por lo que el prompt del sistema de un agente y los nombres de acción de Cedar a los que haces referencia permanecen consistentes entre ejecuciones locales y desplegadas.
:::
