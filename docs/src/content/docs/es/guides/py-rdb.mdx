Este generador crea un nuevo proyecto de base de datos relacional Python respaldado por [Amazon Aurora](https://aws.amazon.com/rds/aurora/) (PostgreSQL o MySQL), [SQLModel](https://sqlmodel.tiangolo.com/) para el modelado de datos, y [Alembic](https://alembic.sqlalchemy.org/) para las migraciones de esquema. Genera el código de aplicación y la infraestructura necesarios para aprovisionar y administrar una base de datos usando AWS CDK o Terraform, con definición de esquema declarativa, despliegue automático de migraciones y un cliente de base de datos.

## Desarrollo Local

### Modelado de Datos

El proyecto generado usa [SQLModel](https://sqlmodel.tiangolo.com/) para definir el esquema de tu base de datos. El flujo de trabajo es model-first: agrega o actualiza clases de tabla SQLModel en el directorio `<name>/models/` de tu proyecto de base de datos, luego genera una migración a partir de esos cambios de modelo.

Modelo de ejemplo:

```python title="packages/my_db/my_db/models/example.py"
from sqlalchemy import Column, String
from sqlmodel import Field, SQLModel


class ExampleModel(SQLModel, table=True):
    id: int | None = Field(default=None, primary_key=True)
    name: str = Field(sa_column=Column(String(255), nullable=False))
    description: str | None = Field(default=None, sa_column=Column(String(255), nullable=True))
```

Importa tus modelos en `<name>/models/__init__.py` para que Alembic pueda descubrirlos durante la autogeneración.

### Crear Migraciones

Después de agregar o actualizar modelos, usa Alembic para generar y aplicar scripts de migración. El objetivo `alembic` generado inicia automáticamente un contenedor de base de datos local antes de ejecutarse:

<NxCommands commands={['run <project>:alembic revision --autogenerate -m "describe your change"']} />

Esto genera un nuevo script de migración en `migrations/versions/`. Revisa el script generado antes de aplicarlo.

Aplica la migración a tu base de datos local:

<NxCommands commands={['run <project>:migrate']} />

Cuando despliegues el stack de AWS, la infraestructura generada aplica automáticamente las migraciones generadas a la base de datos desplegada.

### Aplicar Migraciones Existentes

Cuando descargues archivos de migración creados por otros desarrolladores, aplícalos a tu base de datos local:

<NxCommands commands={['run <project>:migrate']} />

### Ejecutar Comandos Alembic

El objetivo `alembic` generado expone la CLI de Alembic, por lo que puedes ejecutar cualquier comando Alembic contra la base de datos local. Consulta la [referencia de comandos de Alembic](https://alembic.sqlalchemy.org/en/latest/ops.html) para ver los comandos disponibles.

<NxCommands commands={['run <project>:alembic <alembic-command>']} />

### Detener la Base de Datos Local

Detener `dev` (por ejemplo, con `Ctrl+C`) elimina automáticamente el contenedor de base de datos local, pero conserva el volumen nombrado para que tus datos persistan entre reinicios.

:::caution[Windows]
Debido a limitaciones con el manejo de señales en Windows, el contenedor no se elimina automáticamente cuando se detiene `dev`. Necesitarás eliminarlo manualmente:

```bash
<engine> rm -f <scope>-<db-name>
```

Reemplaza `<engine>` con tu motor de contenedor (`docker` o `finch`), `<scope>` con el scope de tu espacio de trabajo Nx (por ejemplo, `proj`), y `<db-name>` con el nombre de tu proyecto de base de datos (por ejemplo, `my-db`).
:::

## Conectarse a la Base de Datos

Importa `session_context` desde tu paquete de base de datos y úsalo como un administrador de contexto para obtener una `Session` de SQLModel:

```python
from sqlmodel import select

from my_scope.my_db import session_context

with session_context() as session:
    results = session.exec(select(ExampleModel)).all()
```

El cliente de base de datos automáticamente:
- Recupera la configuración de la base de datos desde AWS AppConfig usando la variable de entorno `RUNTIME_CONFIG_APP_ID`
- Genera tokens de autenticación temporales a través del `boto3` RDS Signer para autenticación IAM
- Administra conexiones SSL/TLS con validación de certificados

### Requisitos SSL al Conectarse sin RDS Proxy

Al conectarse directamente al clúster Aurora (sin RDS Proxy), pasa la ruta del paquete CA de Amazon RDS usando el parámetro opcional `rds_ca`:

```python
from my_scope.my_db import session_context

with session_context(rds_ca='/opt/global-bundle.pem') as session:
    ...
```

El valor `rds_ca` debe apuntar a un paquete de certificados CA disponible en el sistema de archivos de tu tiempo de ejecución. Para Amazon RDS, usa el paquete CA global de:

```text
https://truststore.pki.rds.amazonaws.com/global/global-bundle.pem
```

Cómo hacer que ese archivo esté disponible depende del tiempo de ejecución que se conecta a la base de datos.

#### Imágenes de Contenedor de Tiempo de Ejecución

Si preparas tu propia imagen de contenedor para el tiempo de ejecución, agrega el paquete CA de RDS en tu Dockerfile y pasa esa ruta a `session_context`:

```dockerfile
ADD --chmod=0644 https://truststore.pki.rds.amazonaws.com/global/global-bundle.pem /opt/global-bundle.pem
```

```python
with session_context(rds_ca='/opt/global-bundle.pem') as session:
    ...
```

#### Funciones Lambda Comprimidas

Si despliegas una función Lambda comprimida, adjunta la capa Lambda `RdsCaLayer` (CDK) o el módulo Terraform `ca-layer`. La capa CDK usa empaquetado Docker para agregar el paquete CA, y la capa desplegada lo proporciona en `/opt/global-bundle.pem`, que puedes pasar a `session_context`:

```python
with session_context(rds_ca='/opt/global-bundle.pem') as session:
    ...
```
