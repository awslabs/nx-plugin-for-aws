Los nuevos espacios de trabajo creados con `@aws/nx-plugin` incluyen hooks de pre-commit de [git-secrets](https://github.com/awslabs/git-secrets) configurados por defecto. Esto escanea los archivos preparados en busca de patrones que coincidan con credenciales de AWS antes de cada commit, previniendo fugas accidentales de credenciales.

## Qué Está Incluido

Cuando creas un nuevo espacio de trabajo, lo siguiente se configura automáticamente:

<FileTree>
  - .git-secrets/
    - git-secrets El script bash de git-secrets (vendorizado desde awslabs/git-secrets)
    - setup.sh Registra patrones de credenciales de AWS en la configuración local de git
  - .husky/
    - pre-commit Ejecuta el escaneo de git-secrets antes de cada commit
  - package.json Incluye husky y el script prepare para configuración automática
</FileTree>

## Cómo Funciona

1. El script `prepare` se ejecuta después de `npm install` / `pnpm install`, que inicializa los hooks de git de husky y registra los patrones de credenciales de AWS en tu configuración local de git.
2. En cada `git commit`, el hook de pre-commit de husky invoca `git-secrets` para escanear los archivos preparados.
3. Si se detecta un patrón que coincide con una credencial de AWS, el commit se bloquea con un mensaje de error.

## Patrones Detectados

Se escanean los siguientes patrones:

- **AWS Access Key IDs** — `AKIA...`, `ASIA...`, y otros prefijos de claves IAM seguidos de 16 caracteres alfanuméricos en mayúsculas
- **AWS Secret Access Keys** — cadenas base64 de 40 caracteres precedidas por una etiqueta `SECRET_ACCESS_KEY`
- **AWS Account IDs** — números de 12 dígitos precedidos por una etiqueta `ACCOUNT_ID`
- **Bedrock API Keys** — claves de Bedrock de larga duración (`ABSK...`) y de corta duración
- **Valores de archivo de credenciales** — valores de claves literales encontrados en `~/.aws/credentials`

Las claves de ejemplo conocidas (por ejemplo, `AKIAIOSFODNN7EXAMPLE`) se agregan automáticamente a la lista de permitidos.

## Permitir Falsos Positivos

Si git-secrets bloquea un commit que contiene un falso positivo:

```sh
# Permitir un patrón específico
git config --add secrets.allowed 'my-pattern-here'
```

O agrega patrones a un archivo `.gitallowed` en la raíz del repositorio (una expresión regular por línea).

Para una anulación única:

```sh
git commit --no-verify
```

## Optar por No Participar

Para crear un espacio de trabajo sin git-secrets:

```sh
pnpm create @aws/nx-workspace my-project --no-gitSecrets
```

O pasa `--gitSecrets=false` al generador de preset.

## Requisitos

- **Git** debe estar inicializado en el espacio de trabajo (`git init`)
- **Bash** debe estar disponible en PATH (se incluye con Git para Windows en Windows)
- Los patrones se registran cuando se ejecuta `prepare` — si clonas una copia nueva, ejecuta el comando de instalación de tu gestor de paquetes para volver a registrar
