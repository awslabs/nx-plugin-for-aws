Este generador crea un nuevo proyecto de base de datos relacional respaldado por [Amazon Aurora](https://aws.amazon.com/rds/aurora/) PostgreSQL. Genera el código de aplicación y la infraestructura necesaria para aprovisionar y gestionar una base de datos usando AWS CDK o Terraform.

La infraestructura generada incluye:

- Constructo o módulo Aurora reutilizable
- Constructo o módulo de base de datos específico de la aplicación
- Manejador de usuario de base de datos de aplicación autenticado por IAM
- Registro de configuración en tiempo de ejecución en AWS AppConfig
- Manejador de migración para despliegue automático de migraciones de Prisma

## Implementando tu Base de Datos

### Modelado de Datos

El proyecto generado usa [Prisma ORM](https://www.prisma.io/docs/orm) para definir tu esquema de base de datos y generar un cliente con tipos seguros. Modela tus tablas, relaciones y tipos de campos en `prisma/schema.prisma`.

Ejemplo de modelo `User`:

```ts title="packages/postgres/prisma/schema.prisma"
model User {
  id          Int      @id @default(autoincrement()) 
  firstName   String
  lastName    String
}
```

Para más detalles, consulta la guía oficial sobre [modelado de datos con Prisma](https://www.prisma.io/docs/orm/core-concepts/data-modeling#data-modeling-with-prisma-orm).

### Generando el Cliente de Base de Datos

El generador configura automáticamente el objetivo `generate` para crear un cliente TypeScript de Prisma con tipos seguros cada vez que construyes el proyecto. El cliente se escribe en `generated/prisma` (agregado a `.gitignore`).

También puedes generar manualmente el cliente en cualquier momento:

<Tabs>

```bash

```

  </TabItem>
  <TabItem label="pnpm" icon="seti:pnpm">

```bash

```

  </TabItem>
  <TabItem label="yarn" icon="seti:yarn">

```bash

```

  </TabItem>
  <TabItem label="bun" icon="seti:bun">

```bash

```

  </TabItem>

El envoltorio en tiempo de ejecución en `src/prisma.ts` exporta:

- `DB_PACKAGE_NAME` - la clave usada bajo el espacio de nombres de configuración en tiempo de ejecución `database` en AWS AppConfig
- `getPrisma()` - carga la configuración de conexión de base de datos desde AWS AppConfig y crea un cliente Prisma usando autenticación IAM

El cliente automáticamente:
- Recupera la configuración de base de datos desde AWS AppConfig usando la variable de entorno `RUNTIME_CONFIG_APP_ID`
- Genera tokens de autenticación temporales a través de AWS RDS Signer para autenticación IAM
- Gestiona conexiones SSL/TLS con validación de certificados
- Maneja el pooling de conexiones a través de pools de conexión de base de datos persistentes

Ejemplo de uso en una API tRPC:

```ts title="packages/api/src/procedures/list-user.ts"
import { getPrisma } from ':aws-nx-poc/postgres';
import { publicProcedure } from '../init.js';
import { ListUserOutputSchema } from '../schema/index.js';

export const listUser = publicProcedure
  .output(ListUserOutputSchema)
  .query(async () => {
    const prisma = await getPrisma();
    const users = await prisma.user.findMany({
      orderBy: { id: 'asc' },
    });

    return { users };
  });
```

### Creando Migraciones

:::note[Ejecutar Prisma desde el Directorio del Paquete]
Al usar Prisma CLI, ejecuta los comandos desde el directorio del paquete de base de datos generado (por ejemplo, `packages/postgres`) para que Prisma pueda encontrar `prisma/schema.prisma` y `prisma.config.ts`.
:::

Prisma Migrate requiere una [base de datos sombra](https://www.prisma.io/docs/orm/prisma-migrate/understanding-prisma-migrate/shadow-database) para comparar tu esquema con el estado actual de la base de datos y generar cambios SQL.

Para generar migraciones localmente, apunta Prisma a una base de datos de desarrollo exportando `DATABASE_URL`:

```bash
export DATABASE_URL="postgresql://username:password@hostname:5432/dbname"
```

Luego crea los archivos de migración sin aplicarlos:

## Desplegando en AWS

El generador de base de datos relacional crea infraestructura CDK o Terraform basándose en tu `iacProvider` seleccionado.

<Infrastructure>
<Fragment slot="cdk">
El constructo CDK se crea en `common/constructs`. Ejemplo de uso:

```ts title="packages/infra/src/stacks/application-stack.ts"
import { Postgres } from ':my-scope/common-constructs';

export class ApplicationStack extends Stack {
  constructor(scope: Construct, id: string, props?: StackProps) {
    super(scope, id, props);

    const vpc = new Vpc(this, 'Vpc', {
      subnetConfiguration: [
        {
          name: 'public',
          subnetType: SubnetType.PUBLIC,
        },
        {
          name: 'private_with_egress',
          subnetType: SubnetType.PRIVATE_WITH_EGRESS,
        },
        {
          name: 'private_isolated',
          subnetType: SubnetType.PRIVATE_ISOLATED,
        },
      ],
    });

    const db = new Postgres(this, 'Db', {
      vpc,
      vpcSubnets: {
        subnetType: SubnetType.PRIVATE_ISOLATED,
      }
    });
  }
}
```

Esto aprovisiona un clúster Aurora con RDS Proxy, credenciales de administrador, usuario de base de datos de aplicación, registro de configuración en tiempo de ejecución y manejador de migración. Despliega en subredes privadas aisladas para prevenir acceso directo desde internet.

La infraestructura generada crea dos usuarios de base de datos:
- **Usuario administrador** - Creado durante el aprovisionamiento del clúster con credenciales almacenadas en AWS Secrets Manager
- **Usuario de aplicación** - Creado a través de un recurso personalizado Lambda con autenticación IAM habilitada y privilegios completos en la base de datos de aplicación
</Fragment>

### Conectando Funciones Lambda

Para usar el cliente Prisma desde funciones Lambda, asegúrate de que se cumplan los siguientes requisitos:

1. Despliega las funciones en la misma VPC que la base de datos
2. Permite acceso de red al puerto de la base de datos
3. Otorga permisos IAM `rds-db:connect` para el usuario en tiempo de ejecución generado
4. Establece las variables de entorno requeridas:
   - `NODE_EXTRA_CA_CERTS=/var/runtime/ca-cert.pem` (requerido para validación TLS)
   - `RUNTIME_CONFIG_APP_ID` (establecido automáticamente por la infraestructura generada)
   - `AWS_REGION` (establecido automáticamente en el entorno de ejecución Lambda)

:::note[Prisma CLI vs Cliente en Tiempo de Ejecución]
Los comandos de **Prisma CLI** (`prisma migrate dev`, `prisma migrate deploy`) leen `DATABASE_URL` de las variables de entorno.

El **cliente en tiempo de ejecución** (`src/prisma.ts`) carga la configuración de conexión desde AWS AppConfig y usa autenticación IAM.
:::

<Infrastructure>
<Fragment slot="cdk">

```ts title="packages/infra/src/stacks/application-stack.ts"
import { RuntimeConfig } from ':my-scope/common-constructs';

const runtimeConfig = RuntimeConfig.ensure(this);

const api = new Api(this, 'Api', {
  integrations: Api.defaultIntegrations(this)
    .withDefaultOptions({
      vpc,
      vpcSubnets: {
        subnetType: SubnetType.PRIVATE_WITH_EGRESS,
      },
      environment: {
        NODE_EXTRA_CA_CERTS: '/var/runtime/ca-cert.pem',
      },
    })
    .build(),
});

Object.entries(api.integrations).forEach(([operation, integration]) => {
  db.allowDefaultPortFrom(
    integration.handler,
    `Allow ${operation} to connect to the database`,
  );
  db.grantConnect(integration.handler);
});
```

Esto configura cada función Lambda de la API para ejecutarse en la misma VPC, leer la configuración en tiempo de ejecución desde AppConfig, y conectarse a la base de datos usando autenticación IAM.
</Fragment>
<Fragment slot="terraform">
Al conectar funciones Lambda a la base de datos, usa estas salidas del módulo:

- `appconfig_application_id` - exponer al entorno de cómputo
- `cluster_port` - permitir acceso de red
- `database_runtime_user` - otorgar permiso `rds-db:connect`

Asegúrate de que el cómputo se ejecute en la misma VPC y subredes privadas, y otorga permisos de lectura de AppConfig al rol de cómputo.
</Fragment>
</Infrastructure>

### RDS Proxy

La infraestructura generada incluye un RDS Proxy por defecto, que se sitúa entre tu aplicación y el clúster Aurora. RDS Proxy proporciona varios beneficios:

- **Connection pooling** - Mantiene un pool de conexiones de base de datos que pueden compartirse entre instancias de aplicación, reduciendo la sobrecarga de establecer nuevas conexiones
- **Resiliencia de conexión** - Maneja automáticamente failovers y reconexiones durante reemplazos o mantenimiento de instancias Aurora
- **Autenticación IAM** - Soporta autenticación de base de datos basada en IAM, eliminando la necesidad de gestionar credenciales de base de datos en tu código de aplicación
- **Seguridad mejorada** - Impone cifrado TLS para todas las conexiones

<Infrastructure>
<Fragment slot="cdk">

Por defecto, RDS Proxy está habilitado. El cliente en tiempo de ejecución generado (`getPrisma()`) se conecta automáticamente a través del endpoint del proxy. Puedes deshabilitarlo si es necesario:

```ts title="packages/infra/src/stacks/application-stack.ts"
const db = new Postgres(this, 'Db', {
  ...
  enableRdsProxy: false,
});
```

Cuando RDS Proxy está deshabilitado, tu aplicación se conecta directamente al endpoint del clúster Aurora.
</Fragment>
<Fragment slot="terraform">

Por defecto, RDS Proxy está habilitado. El cliente en tiempo de ejecución generado (`getPrisma()`) se conecta automáticamente a través del endpoint del proxy. Puedes deshabilitarlo si es necesario:

```hcl title="packages/infra/src/main.tf"
module "postgres" {
  source = "../../common/terraform/src/app/dbs/postgres"
  ...
  enable_rds_proxy = false
}
```

Cuando RDS Proxy está deshabilitado, tu aplicación se conecta directamente al endpoint del clúster Aurora. El proxy está configurado para requerir cifrado TLS y usar autenticación IAM.
</Fragment>
</Infrastructure>

## Personalizando la Arquitectura de la Base de Datos

La infraestructura generada puede personalizarse para coincidir con los requisitos de tu carga de trabajo. Los siguientes ejemplos demuestran algunas opciones de personalización comunes disponibles.

### Instancias del Clúster

Configura las instancias writer y reader para tu clúster Aurora.

<Infrastructure>
<Fragment slot="cdk">

```ts title="packages/infra/src/stacks/application-stack.ts"
const db = new Postgres(this, 'Db', {
  ...
  writer: ClusterInstance.serverlessV2('writer'),
  readers: [ClusterInstance.serverlessV2('reader')],
});
```
</Fragment>

