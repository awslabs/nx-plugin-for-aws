## 生成器输出

生成器将在`<directory>/<name>`目录下创建以下项目结构：

<FileTree>

  - src TypeScript 源代码
    - index.ts
  - package.json 定义项目包名的最小项目清单
  - project.json 项目配置和构建目标
  - tsconfig.json 此项目的基础 TypeScript 配置（继承工作区根目录的 tsconfig.base.json）
  - tsconfig.lib.json 库的 TypeScript 配置（运行时或打包源码）
  - tsconfig.spec.json 测试的 TypeScript 配置

:::tip[项目 package.json]
项目的 `package.json` 起初是最小化的。当您添加运行时依赖时，它们会在此处声明（共享的构建/测试工具保留在工作区根目录）——您可以在[下方](#dependencies)了解更多信息。
:::

您还会发现工作区根目录的以下文件发生了变化：

<FileTree>

  - nx.json Nx 配置已更新以为您的项目配置 @nx/js/typescript 插件
  - tsconfig.base.json 为您的项目设置了 TypeScript 别名，以便工作区中的其他项目可以导入它
  - tsconfig.json 为您的项目添加了 TypeScript 项目引用

</FileTree>

## 编写TypeScript源码

在`src`目录中添加TypeScript代码。

### ESM导入语法

由于TypeScript项目是ES模块，请确保使用正确的ESM语法编写导入语句，显式指定文件扩展名：

```ts title="index.ts" ".js"
import { sayHello } from './hello.js';
```

:::note[TypeScript引用]
虽然我们使用TypeScript且`sayHello`定义在`hello.ts`中，但导入时使用`.js`扩展名。更多信息请阅读[此处](https://www.typescriptlang.org/docs/handbook/modules/reference.html)。
:::

### 为其他TypeScript项目导出

项目的入口点是`src/index.ts`。您可以在此处添加需要被其他项目导入的导出：

```ts title="src/index.ts"
export { sayHello } from './hello.js';
export * from './algorithms/index.js';
```

### 在其他项目中导入库代码

工作区`tsconfig.base.json`中配置了[TypeScript别名](https://www.typescriptlang.org/docs/handbook/modules/reference.html#paths)，允许从其他TypeScript项目引用您的项目：

```ts title="packages/my-other-project/src/index.ts"
import { sayHello } from '@my-scope/my-library';
```

:::note[项目别名]
别名与您的项目包名匹配，因此工作区中的任何项目都可以通过其 `package.json` 中声明的名称导入——无需将本地项目声明为依赖。
:::

首次在工作区中导入新项目时，IDE可能会显示类似以下错误：

<details>
<summary>导入错误</summary>

```bash wrap

  File is ECMAScript module because '/path/to/my/workspace/package.json' has field "type" with value "module" ts(6307)
```

这是因为尚未设置[项目引用](https://www.typescriptlang.org/docs/handbook/project-references.html)。

TypeScript项目默认配置了Nx TypeScript同步生成器，您无需手动配置项目引用。只需运行以下命令，Nx将自动添加所需配置：

<NxCommands commands={['sync']} />

此后IDE错误应消失，即可正常使用库。

### 保持路径别名同步

如果您在项目的 `tsconfig.json` 中添加自定义的 `compilerOptions.paths` 条目，TypeScript 将停止继承 `tsconfig.base.json` 中定义的工作区别名。隐藏的 `ts#sync` 生成器会在 `compile` 目标之前运行（在 `nx.json` 中配置），将任何缺失的基础别名复制到已声明 `paths` 的 `tsconfig.json`、`tsconfig.lib.json` 和 `tsconfig.app.json` 文件中。要禁用此功能，请从 `nx.json` 的 `targetDefaults.compile.syncGenerators` 中移除 `@aws/nx-plugin:ts#sync`。

### 依赖管理

每个 TypeScript 项目在其自己的 `package.json` 中声明**其源代码导入的第三方运行时依赖**。这使每个项目都能自我描述——其清单准确列出了它需要的内容——并让 Biome 的 [`noUndeclaredDependencies`](https://biomejs.dev/linter/rules/no-undeclared-dependencies/) lint 规则强制项目永远不会导入未声明的包。

**构建和测试工具**（例如 `vitest`、`tsx`、`rolldown`、Nx 工具链（`nx`、`@nx/js`、...）、`typescript`、`@biomejs/biome`）在工作区中共享，因此它们位于**根目录** `package.json` 的 `devDependencies` 中，而不是在每个项目中。这些工具仅从配置文件、构建脚本和测试中引用，lint 规则会豁免这些文件。

要向项目添加运行时依赖，请将其添加到该项目的 `package.json` 中。使用包管理器的过滤器/工作区标志来定位正确的项目，例如使用 pnpm：

```bash
pnpm add some-npm-package --filter @my-scope/my-library
```

要添加共享的开发工具，请将其添加到工作区根目录：

<InstallCommand pkg="-D some-dev-tool" />

这遵循[单一版本策略](https://nx.dev/concepts/decisions/dependency-management#single-version-policy)，意味着所有项目使用每个依赖的相同版本——避免类型冲突和部署包中的多个包副本。对于支持目录（[pnpm](https://pnpm.io/catalogs)、[yarn](https://yarnpkg.com/features/catalogs) 和 [bun](https://bun.com/docs/install/catalogs)）的包管理器，生成器会在目录中记录每个版本，并使用 `catalog:` 协议引用它（例如 `"zod": "catalog:"`），因此无论哪个项目声明依赖，目录都是版本的唯一真实来源。提供的 `biome.json` 启用了 Biome 的 [`experimentalPnpmCatalogs`](https://biomejs.dev/reference/configuration/#javascriptresolverexperimentalpnpmcatalogs) 解析器，以便 lint 规则理解 `catalog:` 引用。

:::tip[在 npm 上对齐版本]
npm（和 yarn classic）没有目录功能，因此版本直接在每个 `package.json` 中声明。要在项目之间保持一致，我们推荐使用 [syncpack](https://github.com/JamieMason/syncpack)（`npx syncpack lint`）。
:::

:::note[bun]
在 bun 上，一些 Nx 生成器从根清单读取版本的包（`vite`、`react`、`react-dom`）保留直接版本范围而不是 `catalog:` 引用——Nx 尚未解析 bun 目录，`catalog:` 引用会破坏这些生成器。
:::

#### 选择退出目录

目录默认启用。要关闭它们，请使用 `--catalog false` 创建工作区：

```bash
pnpm create @aws/nx-workspace my-project --catalog false
```

或随时在 `aws-nx-plugin.config.mts` 中设置：

```ts title="aws-nx-plugin.config.mts"
export default {
  packageManager: {
    catalogs: false,
  },
} satisfies AwsNxPluginConfig;
```

禁用目录后，生成器会将依赖版本直接写入每个项目的 `package.json`，保持项目间版本一致是您的责任（例如使用 [syncpack](https://github.com/JamieMason/syncpack)）。在 pnpm 上启用目录时，工作区还会设置 [`catalogMode: strict`](https://pnpm.io/settings#catalogmode)，以便 `pnpm add` 在目录中记录新依赖，在无需手动编辑的情况下保持单一版本策略。

当工作区中的一个项目导入另一个项目时，您无需手动声明：TypeScript 路径别名会解析构建时的导入，而 `ts#sync` 生成器（在 `compile` 之前运行，或通过 `nx sync` 按需运行）会在导入项目的 `package.json` 中将被导入的项目声明为工作区依赖（在包管理器支持工作区协议的情况下为 `workspace:*`，在 npm 和 yarn classic 上为 `*`）。这使每个项目的清单都能准确记录其使用的内容——满足 `noUndeclaredDependencies` lint 规则——而无需手动记账。

#### 在根目录声明所有依赖

如果您更喜欢仅在根目录 `package.json` 中声明所有依赖（不进行每个项目的依赖声明），请在工作区 `biome.json` 中关闭 lint 规则：

```json title="biome.json"
{
  "linter": {
    "rules": {
      "correctness": {
        "noUndeclaredDependencies": "off"
      }
    }
  }
}
```

项目在构建时解析根目录安装的包，因此一切仍然可以构建——您放弃了每个项目清单作为每个项目依赖的准确记录（如果您以后发布项目或拆分仓库，这很重要）。

#### 运行时代码

当您将 TypeScript 项目用作运行时代码（例如作为 AWS Lambda 函数的处理程序）时，建议使用 [Rolldown](https://rolldown.rs/) 等工具来打包项目，因为这可以进行 [tree-shaking](https://rolldown.rs/in-depth/why-bundlers) 以确保仅包含项目实际引用的依赖。

您可以通过在 `project.json` 文件中添加如下目标来实现：

```json
{
  ...
  "targets": {
    ...
    "bundle": {
      "cache": true,
      "executor": "nx:run-commands",
      "outputs": ["{workspaceRoot}/dist/packages/my-library/bundle"],
      "options": {
        "command": "rolldown -c rolldown.config.ts"
      }
    },
  },
}
```

并添加 `rolldown.config.ts` 文件如下：

```ts
// rolldown.config.ts
import { defineConfig } from 'rolldown';

export default defineConfig([
  {
    input: 'src/index.ts',
    output: {
      file: '../../dist/packages/my-library/bundle/index.js',
      format: 'cjs',
      inlineDynamicImports: true,
    },
  },
]);
```

:::note[打包配置]
请注意，在上述目标中，我们选择 `src/index.ts` 作为打包的入口点，这意味着从此文件导出的代码将包含在包中，以及其所有依赖。
:::

:::tip[Lambda 函数]
如果您正在构建 AWS Lambda 函数，请查看 <Link path="/zh/guides/ts-lambda-function">`ts#lambda-function`</Link> 生成器，因为它会为您配置打包，并生成基础设施和添加可观测性和类型安全。
:::

#### 发布到 NPM

如果您要将 TypeScript 项目发布到 NPM，其 `package.json` 必须声明入口点（`main`/`exports`）以及其依赖，生成器已经保持依赖的准确性（参见[依赖管理](#dependencies)）。

### 构建

您的 TypeScript 项目配置了 `build` 目标（定义在 `project.json` 中），您可以通过以下方式运行：

<NxCommands commands={['build <project-name>']} />

其中 `<project-name>` 是项目的完全限定名称。

`build` 目标将编译、lint 和测试您的项目。

构建输出可以在工作区根目录的 `dist` 文件夹中找到，位于您的包和目标的目录中，例如 `dist/packages/<my-library>/tsc`

要构建工作区中所有项目，请运行：

<NxCommands commands={['run-many --target build']} />

或使用简写命令：

<PackageManagerShortCommand commands={["build"]} />

## 测试

[Vitest](https://vitest.dev/) 已配置用于测试您的项目。

### 编写测试

测试应写在 `.spec.ts` 或 `.test.ts` 文件中，与项目的 `src` 文件夹中的代码共置。

例如：

<FileTree>
  - src
    - hello.ts 库源代码
    - hello.spec.ts hello.ts 的测试
</FileTree>

Vitest 提供类似 Jest 的语法来定义测试，包括 `describe`、`it`、`test` 和 `expect` 等工具。

```ts title="hello.spec.ts"
import { sayHello } from './hello.js';

describe('sayHello', () => {

  it('should greet the caller', () => {
    expect(sayHello('Darth Vader')).toBe('Hello, Darth Vader!');
  });

});

```

有关如何编写测试以及模拟依赖等功能的更多详细信息，请参阅 [Vitest 文档](https://vitest.dev/guide/#writing-tests)

### 运行测试

测试将作为项目 `build` 目标的一部分运行，但您也可以通过运行 `test` 目标单独运行它们：

<NxCommands commands={['test <project-name>']} />

您可以使用 Vitest 的 `-t` 标志运行单个测试或测试套件。在 `--` 分隔符之后传递它，以便 Nx 将其转发给 Vitest，而不是将其作为自己的 `--target` 选项使用：

<NxCommands commands={["test <project-name> -- -t 'sayHello'"]} />

:::tip[VSCode 集成]
如果您是 VSCode 用户，我们建议安装 [Vitest Runner for VSCode that actually works](https://marketplace.visualstudio.com/items?itemName=rluvaton.vscode-vitest) 扩展，它允许您从 IDE 运行、监视或调试测试。
:::

## 代码检查

TypeScript 项目使用 [Biome](https://biomejs.dev/) 进行代码检查和格式化。Biome 在工作区根目录的 `biome.json` 文件中配置——对此文件的更改将应用于工作区中的所有 TypeScript 项目并确保一致性。

### 运行检查器

要调用检查器来检查您的项目，您可以运行 `lint` 目标。

<NxCommands commands={["lint <project-name>"]} />

### 修复 Lint 问题

大多数代码检查或格式化问题可以通过使用 `--configuration=fix` 参数运行来自动修复。

<NxCommands commands={["lint <project-name> --configuration=fix"]} />

同样，如果您想修复工作区中所有包的所有 lint 问题，可以运行：

<NxCommands commands={["run-many --target lint --all --configuration=fix"]} />

:::tip[简写命令]
这有一个从工作区根目录运行的简写命令：

<PackageManagerShortCommand commands={["lint"]} />
:::

### 跳过 Lint 问题

为了避免代码检查问题在开发过程中拖慢您的速度（特别是如果您的项目中有无法自动修复的问题），您可以使用 `skip-lint` 配置运行构建：

<NxCommands commands={["run-many --target build --configuration=skip-lint"]} />

这会在构建期间完全跳过lint目标。

:::tip[简写命令]
这有一个从工作区根目录运行的简写命令：

<PackageManagerShortCommand commands={["build:skip-lint"]} />
:::
