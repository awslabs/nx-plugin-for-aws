## 任务一：配置本地开发服务器

将已部署的 `runtime-config.json` 拉取到本地，以便开发服务器可以与真实后端通信：

<NxCommands commands={["run game-ui:load:runtime-config"]} />

然后启动开发服务器：

<NxCommands commands={["serve game-ui"]} />

在浏览器中打开本地网站，登录并按照提示创建新用户。您应该会看到基础的 Shadcn 网站：

## 任务二：CopilotKit 已经配置好的地方

当您在 <Link path="get_started/tutorials/dungeon-game/1">模块一</Link> 中为 `game-ui → story` 运行 `connection` 生成器时，Shadcn 网站的 AG-UI 集成已经为您生成。值得快速了解一下：

<FileTree>
- packages/game-ui/src/
  - components/
    - AguiProvider.tsx 单个 `CopilotKitProvider` 注册了所有 AG-UI 代理。
    - copilot/
      - index.tsx 重新导出主题化的 `CopilotChat` / `CopilotSidebar` / `CopilotPopup`。
      - ShadcnAssistantMessage.tsx, ShadcnUserMessage.tsx, ShadcnChatInput.tsx, ShadcnCursor.tsx, copilot.css
  - hooks/
    - useAguiStoryAgent.tsx 实例化一个指向已部署故事代理的 `@ag-ui/client` `HttpAgent`，并将 `threadId` 填充到 AgentCore 的 33 字符最小会话 ID。
  - main.tsx 用 `<AguiProvider>` 包装 `<App />`
</FileTree>

我们需要做的就是将 `<CopilotChat agentId="agent" threadId={...} />` 放入路由中。有关集成如何组合的更多详细信息，请参阅 <Link path="guides/connection/react-agui">React → AG-UI 连接指南</Link>。

## 任务四：创建游戏路由

我们需要两个路由——一个用于选择英雄，一个用于游玩。两者都使用 shadcn 组件和 CopilotKit 聊天；没有手工编写的聊天 UI。

<Tabs>
<TabItem label="routes/index.tsx">
<E2EDiff before="dungeon-adventure/4/routes/index.tsx.old.template" after="dungeon-adventure/4/routes/index.tsx.template" lang="tsx" />

这是游戏选择器：新游戏表单（shadcn `Input` + `Button` + `Card`）加上由 `useGameApi().games.query` 通过 `useInfiniteQuery` 提供的"继续"列表——底部的哨兵 `<div>` 由 `IntersectionObserver` 监视，当滚动到视图中时会自动调用 `fetchNextPage()`，标题旁边和列表下方的加载指示器显示加载状态。开始游戏会 `saveGame` `(playerName, genre)` 对（以便下次显示）并导航到游玩路由。
</TabItem>
<TabItem label="routes/game/$playerName.tsx">
<E2ECode path="dungeon-adventure/4/routes/game/$playerName.tsx.template" lang="tsx" />

这是游玩路由。它构建确定性的 `threadId`（``{player}-{genre}`` 填充到 33 个字符——AG-UI 钩子将其原样作为 AgentCore 会话 ID 发送），渲染 `<CopilotChat agentId="agent" threadId={threadId} />`，并在顶部叠加来自 `useGameApi().inventory.query` 的库存。在挂载时，`useGameApi().actions.query({ sessionId: threadId })` 读取代理存储在 S3 中的对话历史记录，如果有任何历史记录，则调用 `agent.setMessages(...)` 来重新激活聊天；否则发送一条启动用户消息来开始故事。`agent.messages` 通过 `useAgent({ updates: [OnMessagesChanged] })` 订阅，因此每个新回合也会重新获取库存查询（MCP 工具调用直接修改 DynamoDB）。

<Aside type="tip" title="等待 `isFetching`，而不是 `isLoading`">
React Query 的默认缓存会保留第一个结果 5 分钟，因此返回访问最初会看到第一次加载时的陈旧空列表。我们设置 `staleTime: 0` + `refetchOnMount: 'always'` 并在 `!isFetching && isSuccess` 上保护水合 `useEffect`，以便重新启动路径仅在真正空的线程上触发。
</Aside>

<Aside type="tip" title="确定性 threadId 使 /game/$playerName?genre=... 可共享">
重新访问相同的 URL——无论是在重新加载、重新部署还是在另一台设备上——都会解析为相同的 `threadId`，因此代理会继续讲述相同的故事。
</Aside>

<Aside type="tip" title="路径参数">
`$playerName` 语法告诉 `@tanstack/react-router` 将 `playerName` 作为[路径参数](https://tanstack.com/router/v1/docs/framework/react/guide/path-params)处理。`validateSearch` 使 `genre` 针对我们的枚举进行强类型化。
</Aside>
</TabItem>
</Tabs>
