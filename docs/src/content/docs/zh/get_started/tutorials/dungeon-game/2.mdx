## 任务 1：实现游戏 API

我们将在本节中实现以下 API：

1. `saveGame` - 创建或更新游戏。
2. `queryGames` - 返回分页的历史游戏列表。
3. `queryInventory` - 返回分页的玩家库存物品列表。
4. `queryActions` - 返回给定游戏的对话历史。

:::note[各部分状态的存储位置]
- `games` 和 `inventory` 存储在 **DynamoDB**（通过 ElectroDB）—— 游戏 API 的 Lambda 函数和库存 MCP 服务器读写它们。
- `actions`（聊天记录）存储在 **S3** —— 故事智能体的 `S3SessionManager` 在每轮对话时写入一个 JSON 文件，游戏 API 中的 `queryActions` 读取它们，以便 UI 在页面重新加载时能够重放历史记录。没有 `saveAction` 流程；智能体是唯一的写入者。
:::

### 实体建模

这是应用的实体关系图。

```d2

  shape: sql_table

  shape: sql_table
```

无需部署到 AWS 即可试用我们的 API——`serve-local` 目标针对 [DynamoDB Local](https://docs.aws.amazon.com/amazondynamodb/latest/developerguide/DynamoDBLocal.html) 运行游戏 API。因为我们在模块 1 中将游戏 API 连接到了 `DungeonDb` 项目，此目标也会自动启动 DynamoDB Local。

<Aside type="note" title="工作区不同步">

### 启动本地服务器

使用 `serve-local` 目标在本地启动游戏 API。首先确保您的容器引擎（Docker 或 Finch）正在运行，因为这也会启动 DynamoDB Local：

<NxCommands commands={["serve-local game-api"]} />

`serve-local` 目标设置 `SERVE_LOCAL=true`，因此 `resolveTableName()` 返回本地表名，`getDynamoDBClient()` 连接到 DynamoDB Local 而不是 AWS——无需部署，也不需要 `RUNTIME_CONFIG_APP_ID`。

<Aside type="tip" title="首次运行">
首次运行会拉取 DynamoDB Local 容器镜像，可能需要一些时间。让服务器在此终端中保持运行，并为下面的 `curl` 命令打开第二个终端。
</Aside>

### 测试 API

服务器启动并运行后，查询（空的）游戏列表：

```bash
curl -X GET 'http://localhost:2022/games.query?input=%7B%7D'
```

:::note[URL编码]
测试 API 时传递的 `%7B%7D` 是 URI 编码的空 JSON 对象（`{}`）。
:::

您将看到一个空列表：

```json
{"result":{"data":{"items":[],"cursor":null}}}
```

现在保存一个游戏：

```bash
curl -X POST 'http://localhost:2022/games.save' \
  -H 'Content-Type: application/json' \
  -d '{"playerName":"Alice","genre":"zombie"}'
```

保存操作返回持久化的游戏（带有实体为您设置的 `lastUpdated` 时间戳）：

```json
{"result":{"data":{"playerName":"Alice","genre":"zombie","lastUpdated":"..."}}}
```

再次查询以确认它已持久化到 DynamoDB Local：

```bash
curl -X GET 'http://localhost:2022/games.query?input=%7B%7D'
```

此响应现在包含已保存的游戏：

```json
{"result":{"data":{"items":[{"playerName":"Alice","genre":"zombie","lastUpdated":"..."}],"cursor":null}}}
```

完成后可以停止本地服务器（`Ctrl+C`）。

## 任务 5：本地测试库存 MCP 服务器

我们可以使用生成的 `mcp-server-inspect` 目标通过 [MCP Inspector](https://github.com/modelcontextprotocol/inspector) 试用 MCP 服务器的工具。检查器通过 STDIO 连接到服务器，因此我们单独运行 DynamoDB Local 并告诉服务器使用它。

在一个终端中，启动 DynamoDB Local：

<NxCommands commands={["serve-local dungeon-db"]} />

在第二个终端中，启动检查器：

<NxCommands commands={["mcp-server-inspect inventory"]} />

这会在 `http://localhost:6274` 启动 MCP Inspector。在检查器 UI 中，在连接之前，添加一个环境变量 `SERVE_LOCAL`，值为 `true`，以便服务器的工具与 DynamoDB Local 通信，然后点击 **Connect**。

<Aside type="tip" title="为什么设置 SERVE_LOCAL？">
与 `serve-local` 不同，`mcp-server-inspect` 目标运行纯 STDIO 服务器，因此它不会为您设置 `SERVE_LOCAL`。在检查器的环境中设置它可以使 `getDynamoDBClient()` 和 `resolveTableName()` 连接到本地表而不是 AWS。
</Aside>

切换到 **Tools** 选项卡，点击 **List Tools**，然后尝试 `add-to-inventory`（例如 `playerName: Alice`、`itemName: Rusty Sword`、`emoji: ⚔️`），然后使用 `list-inventory-items` 查看它已持久化到 DynamoDB Local。完成后停止两个终端（`Ctrl+C`）。

<Aside type="tip" title="更喜欢 HTTP 传输？">
如果您更喜欢通过 HTTP 驱动服务器（例如用于脚本），`mcp-server-serve-local` 目标会运行 Streamable HTTP 传输，并为您设置 `SERVE_LOCAL=true`，并自动启动 DynamoDB Local：

<NxCommands commands={["run @dungeon-adventure/inventory:mcp-server-serve-local"]} />

然后针对 `http://localhost:8000/mcp` 初始化会话：
```bash
curl -X POST http://localhost:8000/mcp \
  -H 'Content-Type: application/json' \
  -H 'Accept: application/json, text/event-stream' \
  -d '{"jsonrpc":"2.0","id":1,"method":"initialize","params":{"protocolVersion":"2025-03-26","capabilities":{},"clientInfo":{"name":"curl","version":"1.0.0"}}}'
```
</Aside>
