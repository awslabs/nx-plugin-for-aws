Generator dự án TypeScript có thể được sử dụng để tạo một thư viện hoặc ứng dụng [TypeScript](https://www.typescriptlang.org/) hiện đại được cấu hình với các phương pháp hay nhất như [ECMAScript Modules (ESM)](https://www.typescriptlang.org/docs/handbook/modules/reference.html), [project references](https://www.typescriptlang.org/docs/handbook/project-references.html) của TypeScript, [Vitest](https://vitest.dev/) để chạy tests và [Biome](https://biomejs.dev/) để linting và định dạng.

## Kết Quả từ Generator

Generator sẽ tạo cấu trúc dự án sau trong thư mục `<directory>/<name>`:

<FileTree>

  - src Mã nguồn TypeScript
    - index.ts
  - package.json Manifest dự án định nghĩa tên package và dependencies của dự án
  - project.json Cấu hình dự án và các build targets
  - tsconfig.json Cấu hình TypeScript cơ bản cho dự án này (kế thừa từ tsconfig.base.json ở workspace root)
  - tsconfig.lib.json Cấu hình TypeScript cho thư viện của bạn (mã nguồn runtime hoặc đóng gói)
  - tsconfig.spec.json Cấu hình TypeScript cho các tests
  - vitest.config.mts Cấu hình cho Vitest

</FileTree>

:::tip[package.json của dự án]
`package.json` của dự án bắt đầu tối thiểu. Khi bạn thêm các runtime dependencies, chúng được khai báo ở đây (các công cụ build/test dùng chung vẫn ở workspace root) — bạn có thể tìm hiểu thêm [bên dưới](#dependencies).
:::

Bạn cũng sẽ nhận thấy một số thay đổi đối với các file sau trong workspace root của bạn:

<FileTree>

  - nx.json Cấu hình Nx được cập nhật để cấu hình @nx/js/typescript plugin cho dự án của bạn
  - tsconfig.base.json một alias TypeScript được thiết lập cho dự án của bạn để nó có thể được import bởi các dự án khác trong workspace
  - tsconfig.json một project reference TypeScript được thêm vào cho dự án của bạn

</FileTree>

## Viết Mã Nguồn TypeScript

Thêm mã TypeScript của bạn vào thư mục `src`.

### Cú Pháp Import ESM

Vì dự án TypeScript của bạn là một ES Module, hãy chắc chắn viết các câu lệnh import với cú pháp ESM chính xác, tham chiếu rõ ràng đến phần mở rộng file:

```ts title="index.ts" ".js"
import { sayHello } from './hello.js';
```

:::note[Tham chiếu TypeScript]
Mặc dù chúng ta đang sử dụng TypeScript, và `sayHello` được định nghĩa trong `hello.ts`, chúng ta sử dụng phần mở rộng file `.js` trong import. Bạn có thể đọc thêm về điều này [tại đây](https://www.typescriptlang.org/docs/handbook/modules/reference.html).
:::

### Export cho Các Dự Án TypeScript Khác

Entry point cho dự án TypeScript của bạn là `src/index.ts`. Bạn có thể thêm exports ở đây cho bất cứ thứ gì bạn muốn các dự án khác có thể import:

```ts title="src/index.ts"
export { sayHello } from './hello.js';
export * from './algorithms/index.js';
```

### Import Mã Thư Viện của Bạn trong Các Dự Án Khác

[TypeScript aliases](https://www.typescriptlang.org/docs/handbook/modules/reference.html#paths) cho dự án của bạn được cấu hình trong `tsconfig.base.json` của workspace, cho phép bạn tham chiếu dự án TypeScript của mình từ các dự án TypeScript khác:

```ts title="packages/my-other-project/src/index.ts"
import { sayHello } from '@my-scope/my-library';
```

Khi bạn thêm một câu lệnh import cho một dự án mới trong workspace lần đầu tiên, bạn có thể sẽ thấy lỗi trong IDE tương tự như bên dưới:

<details>
<summary>Lỗi import</summary>

```bash wrap
File '/path/to/my/workspace/packages/my-library/src/index.ts' is not under 'rootDir' '/path/to/my/workspace/packages/my-consumer'. 'rootDir' is expected to contain all source files.
  File is ECMAScript module because '/path/to/my/workspace/package.json' has field "type" with value "module" ts(6059)
File '/path/to/my/workspace/packages/my-library/src/index.ts' is not listed within the file list of project '/path/to/my/workspace/packages/my-consumer/tsconfig.lib.json'. Projects must list all files or use an 'include' pattern.
  File is ECMAScript module because '/path/to/my/workspace/package.json' has field "type" with value "module" ts(6307)
```

</details>

Điều này là do một [project reference](https://www.typescriptlang.org/docs/handbook/project-references.html) chưa được thiết lập, và dự án được import chưa được khai báo làm workspace dependency trong `package.json` của dự án của bạn.

Workspace của bạn được cấu hình với các sync generators quản lý cả hai cho bạn, vì vậy bạn không cần cấu hình chúng thủ công. Chỉ cần chạy lệnh sau và Nx sẽ thêm cấu hình cần thiết:

<NxCommands commands={['sync']} />

```bash wrap
 NX   The workspace is out of sync

[@nx/js:typescript-sync]: Some TypeScript configuration files are missing project references to the projects they depend on, contain stale project references, or have duplicate project references.
[@aws/nx-plugin:ts#sync]: Local project dependencies are out of sync. The following workspace dependencies will be declared:
packages/my-other-project/package.json:
- @my-scope/my-library: workspace:*

Syncing the workspace...
The workspace was synced successfully!
```

TypeScript sync generator của Nx thêm project reference, và generator `ts#sync` của plugin khai báo dự án được import trong `package.json` của dự án của bạn (`workspace:*` khi package manager hỗ trợ workspace protocol, `*` trên npm và yarn classic), vì vậy package manager liên kết dự án cục bộ khi cài đặt. Sau đó, lỗi trong IDE của bạn sẽ biến mất và bạn đã sẵn sàng sử dụng thư viện của mình.

:::tip[Phím tắt build]
Bạn cũng có thể chỉ cần build dự án — các sync generators chạy trước và bạn sẽ được nhắc áp dụng bất kỳ thay đổi nào còn tồn đọng:

```bash wrap
 NX   The workspace is out of sync

...

? Would you like to sync the identified changes to get your workspace up to date? …
Yes, sync the changes and run the tasks
No, run the tasks without syncing the changes
```

Chọn `Yes` để cho phép Nx đồng bộ workspace của bạn.
:::

### Giữ Đồng Bộ Path Aliases

Nếu bạn thêm các entries `compilerOptions.paths` tùy chỉnh trong `tsconfig.json` của một dự án, TypeScript sẽ ngừng kế thừa các aliases workspace được định nghĩa trong `tsconfig.base.json`. Generator ẩn `ts#sync` chạy trước `compile` target (được cấu hình trong `nx.json`) để sao chép bất kỳ base aliases còn thiếu nào vào các files `tsconfig.json`, `tsconfig.lib.json` và `tsconfig.app.json` đã khai báo `paths`. Để tắt tính năng này, hãy xóa `@aws/nx-plugin:ts#sync` khỏi `targetDefaults.compile.syncGenerators` trong `nx.json`.

### Dependencies

Mỗi dự án TypeScript khai báo các runtime dependencies của bên thứ ba trong `package.json` riêng của nó. Các công cụ build và test dùng chung giữa các dự án được định nghĩa trong `package.json` root.

Để thêm một runtime dependency vào một dự án, hãy cài đặt nó vào dự án đó:

<InstallCommand pkg="some-npm-package" project="@my-scope/my-library" />

Bạn cũng có thể chạy lệnh `add`/`install` thông thường của package manager từ trong thư mục của dự án.

Để thêm một công cụ dev dùng chung, hãy cài đặt nó ở workspace root:

<InstallCommand pkg="some-dev-tool" dev />

:::note
Quy tắc lint [`noUndeclaredDependencies`](https://biomejs.dev/linter/rules/no-undeclared-dependencies/) của Biome đảm bảo rằng mã nguồn của một dự án chỉ import các packages được khai báo trong `package.json` riêng của nó. Các file cấu hình, build scripts và tests được miễn trừ, vì vậy các công cụ dùng chung có thể ở lại root.
:::

#### Catalogs

Đối với các package managers hỗ trợ catalogs ([pnpm](https://pnpm.io/catalogs), [yarn](https://yarnpkg.com/features/catalogs) và [bun](https://bun.com/docs/install/catalogs)), các generators ghi lại mỗi phiên bản trong catalog và tham chiếu nó bằng protocol `catalog:` (ví dụ `"zod": "catalog:"`), vì vậy catalog là nguồn duy nhất về phiên bản bất kể dự án nào khai báo dependency. Điều này giúp bạn tuân theo [single version policy](https://nx.dev/concepts/decisions/dependency-management#single-version-policy), tránh xung đột kiểu và nhiều bản sao của các packages trong deployment bundles.

Khi bạn tự thêm một dependency mới, hãy giữ nó trong catalog:

<Tabs syncKey="package-manager">
  <TabItem label="pnpm">
    Workspace đặt [`catalogMode: strict`](https://pnpm.io/settings#catalogmode), vì vậy `pnpm add` ghi lại phiên bản trong catalog và tham chiếu nó tự động — không cần bước bổ sung.

    ```bash
    pnpm add some-npm-package --filter my-library
    ```
  </TabItem>
  <TabItem label="yarn">
    Protocol `catalog:` của yarn chỉ *tham chiếu* một entry catalog hiện có — nó không thể tạo một entry mới — vì vậy hãy ghi lại phiên bản trước.

    <Steps>
    1. Thêm phiên bản dưới `catalog` trong `.yarnrc.yml`:

       ```yaml title=".yarnrc.yml"
       catalog:
         some-npm-package: ^1.0.0
       ```

    2. Tham chiếu nó từ dự án:

       ```bash
       yarn workspace @my-scope/my-library add some-npm-package@catalog:
       ```
    </Steps>
  </TabItem>
  <TabItem label="bun">
    Protocol `catalog:` của bun chỉ *tham chiếu* một entry catalog hiện có — nó không thể tạo một entry mới — vì vậy hãy ghi lại phiên bản trước.

    <Steps>
    1. Thêm phiên bản dưới `catalog` trong `package.json` root:

       ```json title="package.json"
       {
         "catalog": {
           "some-npm-package": "^1.0.0"
         }
       }
       ```

    2. Tham chiếu nó từ dự án:

       ```bash
       bun add some-npm-package@catalog: --cwd packages/my-library
       ```
    </Steps>
  </TabItem>
</Tabs>

:::tip[Căn chỉnh phiên bản trên npm]
npm không có tính năng catalog, vì vậy các phiên bản được khai báo trực tiếp trong mỗi `package.json`. Để giữ chúng nhất quán giữa các dự án, chúng tôi khuyên dùng [syncpack](https://github.com/JamieMason/syncpack) (`npx syncpack lint`).
:::

##### Tắt catalogs

Catalogs được bật theo mặc định. Để tắt chúng, hãy tạo workspace của bạn với `--catalog false`:

<CreateNxWorkspaceCommand workspace="my-project" extraArgs="--catalog false" />

Hoặc đặt nó trong `aws-nx-plugin.config.mts` bất cứ lúc nào:

```ts title="aws-nx-plugin.config.mts"
export default {
  packageManager: {
    catalogs: false,
  },
} satisfies AwsNxPluginConfig;
```

Khi catalogs bị tắt, các generators ghi phiên bản dependency trực tiếp vào `package.json` của mỗi dự án, và việc giữ các phiên bản căn chỉnh giữa các dự án là trách nhiệm của bạn.

#### Khai báo tất cả dependencies ở root

Nếu bạn muốn khai báo mọi dependency chỉ trong `package.json` root (không có khai báo dependency cho từng dự án), hãy tắt quy tắc lint trong `biome.json` của workspace:

```json title="biome.json"
{
  "linter": {
    "rules": {
      "correctness": {
        "noUndeclaredDependencies": "off"
      }
    }
  }
}
```

Các dự án resolve các packages được cài đặt ở root tại thời điểm build, vì vậy mọi thứ vẫn build được — bạn từ bỏ việc các manifest của từng dự án là một bản ghi chính xác về các dependencies của mỗi dự án (điều này quan trọng nếu sau này bạn publish một dự án hoặc tách repo).

#### Mã Runtime

Khi bạn sử dụng dự án TypeScript của mình làm mã runtime (ví dụ như handler cho một AWS Lambda function), được khuyến nghị là bạn sử dụng một công cụ như [Rolldown](https://rolldown.rs/) để bundle dự án của bạn, vì điều này có thể [tree-shake](https://rolldown.rs/in-depth/why-bundlers) để đảm bảo rằng chỉ các dependencies mà dự án của bạn thực sự tham chiếu được bao gồm.

Bạn có thể đạt được điều này bằng cách thêm một target như sau vào file `project.json` của bạn:

```json
{
  ...
  "targets": {
    ...
    "bundle": {
      "cache": true,
      "executor": "nx:run-commands",
      "outputs": ["{workspaceRoot}/dist/packages/my-library/bundle"],
      "options": {
        "command": "rolldown -c rolldown.config.ts"
      }
    },
  },
}
```

Và thêm file `rolldown.config.ts` như sau:

```ts
// rolldown.config.ts
import { defineConfig } from 'rolldown';

export default defineConfig([
  {
    input: 'src/index.ts',
    output: {
      file: '../../dist/packages/my-library/bundle/index.js',
      format: 'cjs',
      inlineDynamicImports: true,
    },
  },
]);
```

:::note[Cấu hình đóng gói]
Lưu ý rằng trong target trên, chúng ta đã chọn `src/index.ts` làm entrypoint cho bundle, có nghĩa là mã được export từ file này sẽ được bao gồm trong bundle, cùng với tất cả các dependencies của nó.
:::

:::tip[Hàm Lambda]
Nếu bạn đang xây dựng một AWS Lambda function, hãy xem generator <Link path="/guides/ts-lambda-function">`ts#lambda-function`</Link> vì nó cấu hình bundling cho bạn, cũng như tạo infrastructure và thêm observability và type-safety.
:::

#### Publish lên NPM

Nếu bạn đang publish dự án TypeScript của mình lên NPM, `package.json` của nó phải khai báo các entry points (`main`/`exports`) ngoài các dependencies của nó, mà các generators đã giữ chính xác (xem [Dependencies](#dependencies)).

### Building

Dự án TypeScript của bạn được cấu hình với một `build` target (được định nghĩa trong `project.json`), bạn có thể chạy qua:

<NxCommands commands={['build <project-name>']} />

Trong đó `<project-name>` là tên đầy đủ của dự án của bạn.

`build` target sẽ compile, lint và test dự án của bạn.

Kết quả build có thể được tìm thấy trong thư mục `dist` root trong workspace của bạn, bên trong một thư mục cho package và target của bạn, ví dụ `dist/packages/<my-library>/tsc`

Để build tất cả các dự án trong workspace của bạn, chạy:

<NxCommands commands={['run-many --target build']} />

Hoặc sử dụng lệnh rút gọn:

<PackageManagerShortCommand commands={["build"]} />

## Testing

[Vitest](https://vitest.dev/) được cấu hình để test dự án của bạn.

### Viết Tests

Tests nên được viết trong các file `.spec.ts` hoặc `.test.ts`, đặt cùng vị trí trong thư mục `src` của dự án.

Ví dụ:

<FileTree>
  - src
    - hello.ts Mã nguồn thư viện
    - hello.spec.ts Tests cho hello.ts
</FileTree>

Vitest cung cấp cú pháp giống Jest để định nghĩa tests, với các tiện ích như `describe`, `it`, `test` và `expect`.

```ts title="hello.spec.ts"
import { sayHello } from './hello.js';

describe('sayHello', () => {

  it('should greet the caller', () => {
    expect(sayHello('Darth Vader')).toBe('Hello, Darth Vader!');
  });

});

```

Để biết thêm chi tiết về cách viết tests và các tính năng như mocking dependencies, tham khảo [tài liệu Vitest](https://vitest.dev/guide/#writing-tests)

### Chạy Tests

Tests sẽ chạy như một phần của `build` target cho dự án của bạn, nhưng bạn cũng có thể chạy chúng riêng biệt bằng cách chạy `test` target:

<NxCommands commands={['test <project-name>']} />

Bạn có thể chạy một test riêng lẻ hoặc một suite tests bằng cách sử dụng flag `-t` của Vitest. Truyền nó sau dấu phân cách `--` để Nx chuyển tiếp nó cho Vitest thay vì sử dụng nó như tùy chọn `--target` của chính nó:

<NxCommands commands={["test <project-name> -- -t 'sayHello'"]} />

:::tip[Tích hợp VSCode]
Nếu bạn là người dùng VSCode, chúng tôi khuyên bạn nên cài đặt extension [Vitest Runner for VSCode that actually works](https://marketplace.visualstudio.com/items?itemName=rluvaton.vscode-vitest), cho phép bạn chạy, watch hoặc debug tests từ IDE của mình.
:::

## Linting

Các dự án TypeScript sử dụng [Biome](https://biomejs.dev/) để linting và định dạng. Biome được cấu hình trong file `biome.json` ở workspace root — các thay đổi tại đây áp dụng cho tất cả các dự án TypeScript trong workspace của bạn và đảm bảo tính nhất quán.

### Chạy Linter

Để gọi linter kiểm tra dự án của bạn, bạn có thể chạy `lint` target.

<NxCommands commands={["lint <project-name>"]} />

### Sửa Các Vấn Đề Lint

Phần lớn các vấn đề linting hoặc formatting có thể được sửa tự động bằng cách chạy với argument `--configuration=fix`.

<NxCommands commands={["lint <project-name> --configuration=fix"]} />

Tương tự, nếu bạn muốn sửa tất cả các vấn đề lint trong tất cả các packages trong workspace của bạn, bạn có thể chạy:

<NxCommands commands={["run-many --target lint --all --configuration=fix"]} />

:::tip[Lệnh viết tắt]
Lệnh này có một lệnh rút gọn từ root của workspace:

<PackageManagerShortCommand commands={["lint"]} />
:::

### Bỏ Qua Các Vấn Đề Lint

Để tránh các vấn đề linting làm chậm bạn trong quá trình phát triển (đặc biệt nếu bạn có các vấn đề không thể tự động sửa trong dự án của mình), bạn có thể chạy một build với cấu hình `skip-lint`:

<NxCommands commands={["run-many --target build --configuration=skip-lint"]} />

Điều này bỏ qua hoàn toàn lint target trong quá trình build.

:::tip[Lệnh viết tắt]
Lệnh này có một lệnh rút gọn từ root của workspace:

<PackageManagerShortCommand commands={["build:skip-lint"]} />
:::
