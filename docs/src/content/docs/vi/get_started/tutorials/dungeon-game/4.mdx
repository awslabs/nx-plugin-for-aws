## Nhiệm vụ 1: Chạy mọi thứ cục bộ

Khởi động toàn bộ stack cục bộ — máy chủ phát triển game-ui cùng với Game API cục bộ và AG-UI Story Agent cục bộ (lần lượt khởi động Inventory MCP server) — chỉ với một lệnh:

<NxCommands commands={["dev game-ui"]} />

Target `dev` trên `game-ui` có `dependsOn` trên `game-api:dev` và `dungeon_adventure.story:agent-dev`, vì vậy Nx khởi động máy chủ cục bộ của mọi dự án song song. Thông qua các kết nối chúng ta đã thiết lập trong Module 1, chúng lần lượt khởi động Inventory MCP server và DynamoDB Local. Đảm bảo container engine của bạn đang chạy, sau đó mở máy chủ phát triển trong trình duyệt.

<Aside type="tip" title="Không cần đăng nhập cục bộ">
Trong chế độ `local-dev`, component `CognitoAuth` bỏ qua luồng đăng nhập Cognito (không có user pool chạy cục bộ), vì vậy bạn sẽ vào thẳng trò chơi mà không cần đăng nhập. Bạn sẽ tạo tài khoản thực khi chúng ta triển khai lên AWS ở cuối module này.
</Aside>

<Aside type="caution" title="Giữ máy chủ phát triển chạy">
Để máy chủ phát triển chạy trong suốt phần còn lại của module này — nó sẽ tự động tải lại mọi thay đổi bên dưới.
</Aside>

## Nhiệm vụ 2: Nơi CopilotKit đã được kết nối

Khi bạn chạy generator `connection` cho `game-ui → story` trong <Link path="get_started/tutorials/dungeon-game/1">Module 1</Link>, tích hợp AG-UI của trang web Shadcn đã được tạo cho bạn. Đáng để xem qua nhanh:

<FileTree>
- packages/game-ui/src/
  - components/
    - AguiProvider.tsx Single `CopilotKitProvider` registered with every AG-UI agent.
    - copilot/
      - index.tsx Re-exports themed `CopilotChat` / `CopilotSidebar` / `CopilotPopup`.
      - ShadcnAssistantMessage.tsx, ShadcnUserMessage.tsx, ShadcnChatInput.tsx, ShadcnCursor.tsx, copilot.css
  - hooks/
    - useAguiStoryAgent.tsx Instantiates an `@ag-ui/client` `HttpAgent` pointing at the deployed Story Agent and pads `threadId` to AgentCore's 33-character minimum session id.
  - main.tsx Wraps `<App />` in `<AguiProvider>`
</FileTree>

Tất cả những gì chúng ta cần làm là thêm `<CopilotChat agentId="agent" threadId={...} />` vào một route. Để biết thêm chi tiết về cách tích hợp được kết hợp, hãy xem <Link path="guides/connection/react-agui">hướng dẫn kết nối React → AG-UI</Link>.

## Nhiệm vụ 4: Tạo các route của trò chơi

Chúng ta cần hai route — một để chọn nhân vật, một để chơi. Cả hai đều sử dụng các component shadcn và chat CopilotKit; không có giao diện chat tự viết.

<Tabs>
<TabItem label="routes/index.tsx">
<E2EDiff before="dungeon-adventure/4/routes/index.tsx.old.template" after="dungeon-adventure/4/routes/index.tsx.template" lang="tsx" />

Đây là bộ chọn trò chơi: form trò chơi mới (shadcn `Input` + `Button` + `Card`) cộng với danh sách "Tiếp tục" được cung cấp bởi `useGameApi().games.query` với `useInfiniteQuery` — một `<div>` sentinel ở dưới cùng được theo dõi bởi `IntersectionObserver` tự động gọi `fetchNextPage()` khi được cuộn vào tầm nhìn, và các spinner bên cạnh tiêu đề và bên dưới danh sách hiển thị trạng thái đang tải. Bắt đầu một trò chơi sẽ `saveGame` cặp `(playerName, genre)` (để nó hiển thị lần sau) và điều hướng đến route chơi.
</TabItem>
<TabItem label="routes/game/$playerName.tsx">
<E2ECode path="dungeon-adventure/4/routes/game/$playerName.tsx.template" lang="tsx" />

Đây là route chơi. Nó xây dựng `threadId` xác định (``{player}-{genre}`` được đệm đến 33 ký tự — hook AG-UI sẽ gửi nó nguyên văn làm session id của AgentCore), render `<CopilotChat agentId="agent" threadId={threadId} />`, và phủ lên trên inventory từ `useGameApi().inventory.query`. Khi mount, `useGameApi().actions.query({ sessionId: threadId })` đọc lịch sử hội thoại mà agent đã lưu trữ trong S3 và — nếu có — gọi `agent.setMessages(...)` để rehydrate chat; nếu không nó gửi một tin nhắn người dùng khởi tạo để bắt đầu câu chuyện. `agent.messages` được đăng ký qua `useAgent({ updates: [OnMessagesChanged] })` nên mỗi lượt mới cũng refetch lại inventory query (các lời gọi MCP tool thay đổi DynamoDB trực tiếp).

<Aside type="tip" title="Đợi `isFetching`, không phải `isLoading`">
Cache mặc định của React Query giữ lại kết quả đầu tiên trong 5 phút, vì vậy một lần truy cập lại ban đầu sẽ thấy danh sách trống cũ từ lần tải đầu tiên. Chúng ta đặt `staleTime: 0` + `refetchOnMount: 'always'` và bảo vệ `useEffect` hydration bằng `!isFetching && isSuccess` để đường dẫn re-prime chỉ kích hoạt trên một thread thực sự trống.
</Aside>

<Aside type="tip" title="threadId xác định làm cho /game/$playerName?genre=... có thể chia sẻ">
Truy cập lại cùng một URL — dù sau khi tải lại, triển khai lại, hoặc trên thiết bị khác — sẽ phân giải thành cùng một `threadId`, vì vậy agent tiếp tục kể cùng một câu chuyện.
</Aside>

<Aside type="tip" title="Tham số đường dẫn">
Cú pháp `$playerName` yêu cầu `@tanstack/react-router` xử lý `playerName` như một [path param](https://tanstack.com/router/v1/docs/framework/react/guide/path-params). `validateSearch` làm cho `genre` được định kiểu mạnh theo enum của chúng ta.
</Aside>

## Nhiệm vụ 6: Kết hợp các thành phần cục bộ và đã triển khai

Bạn đã thấy hai đầu của phổ: mọi thứ cục bộ (`serve-local`) và mọi thứ đã triển khai. Trong quá trình phát triển hàng ngày, thường hữu ích khi kết hợp cả hai — ví dụ, lặp lại trên code website với API và agent *thực* đã triển khai, hoặc chạy API cục bộ với bảng DynamoDB *thực*.

Chìa khóa là biến môi trường `RUNTIME_CONFIG_APP_ID`. Khi một dự án chạy **không có** `SERVE_LOCAL=true`, các tra cứu runtime config lấy cấu hình của chúng từ AWS AppConfig sử dụng application id này — giá trị `RuntimeConfigApplicationId` từ các đầu ra CDK của bạn.

Mỗi website có một target `load:runtime-config` tải xuống `runtime-config.json` đã triển khai (Cognito pool, API endpoints, agent ARN) vào máy chủ phát triển cục bộ. Một số kết hợp hữu ích:

- **Website cục bộ → backend đã triển khai.** Tải config đã triển khai một lần, sau đó chạy target `serve` thông thường để UI giao tiếp với API và agent đã triển khai:

  <NxCommands commands={["run game-ui:load:runtime-config"]} />
  <NxCommands commands={["serve game-ui"]} />

- **API cục bộ → bảng DynamoDB thực.** Chạy target `serve` thông thường với `RUNTIME_CONFIG_APP_ID` được đặt thành application id đã triển khai:

  <NxCommands env={{RUNTIME_CONFIG_APP_ID:"<RuntimeConfigApplicationId from CDK outputs>"}} commands={["serve game-api"]} />

Tính linh hoạt này cho phép bạn kiểm tra chính xác phần bạn đang làm việc, với bất kỳ thành phần nào — cục bộ hoặc đã triển khai — có ý nghĩa.
