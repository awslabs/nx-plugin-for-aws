## Nhiệm vụ 1: Triển khai Game API

Chúng ta sẽ triển khai các API sau trong phần này:

1. `saveGame` - tạo hoặc cập nhật một trò chơi.
2. `queryGames` - trả về danh sách phân trang các trò chơi đã lưu trước đó.
3. `queryInventory` - trả về danh sách phân trang các vật phẩm trong kho của người chơi.
4. `queryActions` - trả về lịch sử hội thoại cho một trò chơi nhất định.

:::note[Nơi lưu trữ từng phần trạng thái]
- `games` và `inventory` được lưu trong **DynamoDB** (thông qua ElectroDB) — các lambda của Game API và Inventory MCP server đọc/ghi chúng.
- `actions` (bản ghi trò chuyện) được lưu trong **S3** — `S3SessionManager` của Story Agent ghi một blob JSON cho mỗi lượt chơi, và `queryActions` trong Game API đọc chúng lại để UI có thể phát lại lịch sử khi tải lại trang. Không có thủ tục `saveAction`; agent là người ghi duy nhất.
:::

### Mô hình hóa thực thể

Đây là sơ đồ ER cho ứng dụng của chúng ta.

```d2
direction: right

game: Game {
  shape: sql_table
  playerName: "string  PK"
  genre: string
  lastUpdated: string
}

item: Item {
  shape: sql_table
  playerName: "string  PK"
  itemName: "string  SK"
  emoji: "string  (optional)"
  lastUpdated: string
  quantity: number
}

game -> item: 1..*
```

Chúng ta sẽ mô hình hóa dữ liệu với [ElectroDB](https://electrodb.dev/en/core-concepts/introduction/) trên bảng `DungeonDb` mà chúng ta đã tạo bằng generator `ts#dynamodb` trong <Link path="get_started/tutorials/dungeon-game/1">Module 1</Link>. Generator đó đã thêm `electrodb` và `@aws-sdk/client-dynamodb` vào workspace, vì vậy dependency bổ sung duy nhất chúng ta cần là S3 client được sử dụng bởi procedure `queryActions` của Game API để đọc các lượt hội thoại mà agent đã ghi:

<InstallCommand pkg={getDungeonAdventureElectroDbDependencies()} />

<Aside title="Phụ thuộc gốc">
Tất cả các dependencies được thêm vào `package.json` gốc vì `@aws/nx-plugin` tuân theo nguyên tắc [single version policy](https://nx.dev/concepts/decisions/dependency-management#single-version-policy). Để biết thêm thông tin, tham khảo <Link path="guides/typescript-project#dependencies">hướng dẫn ts#project</Link>.

Chúng ta mô hình hóa các entities trong dự án `dungeon-db` để cả Game API và Inventory MCP server có thể import chúng mà không phụ thuộc lẫn nhau. Thay thế entity mẫu được tạo trong `packages/dungeon-db/src/entities/index.ts` bằng các entities `Game` và `Inventory` của chúng ta, và xóa `packages/dungeon-db/src/entities/example.ts`:

Lưu ý rằng các entities sử dụng các helper `getDynamoDBClient()` và `resolveTableName()` từ file `client.ts` được tạo của dự án — đây là điều cho phép cùng một mã entity giao tiếp với DynamoDB Local trong quá trình phát triển và bảng đã triển khai trong AWS.

ElectroDB cho phép chúng ta không chỉ định nghĩa các kiểu của mình mà còn có thể cung cấp giá trị mặc định cho các giá trị nhất định như timestamps. ## Nhiệm vụ 4: Kiểm thử Game API locally

Không cần triển khai lên AWS để thử API của chúng ta — target `serve-local` chạy Game API với [DynamoDB Local](https://docs.aws.amazon.com/amazondynamodb/latest/developerguide/DynamoDBLocal.html). Vì chúng ta đã kết nối Game API với dự án `DungeonDb` trong Module 1, target này cũng tự động khởi động DynamoDB Local.

Đầu tiên, sửa mọi vấn đề lint:

Sau đó build codebase:

<Aside type="note" title="Workspace không đồng bộ">
Việc thêm procedures và các tham chiếu dự án mới có nghĩa là Nx có thể lại nhắc rằng "The workspace is out of sync" (như trong <Link path="get_started/tutorials/dungeon-game/1">Module 1</Link>). Chọn **Yes, sync the changes and run the tasks** để tiếp tục. Bạn sẽ thấy lời nhắc này bất cứ khi nào các tham chiếu dự án thay đổi.
</Aside>

### Khởi động local server

Khởi động Game API locally với target `serve-local`. Đảm bảo container engine của bạn (Docker hoặc Finch) đang chạy trước, vì điều này cũng khởi động DynamoDB Local:

<NxCommands commands={["serve-local game-api"]} />

Target `serve-local` đặt `SERVE_LOCAL=true`, vì vậy `resolveTableName()` trả về tên bảng local và `getDynamoDBClient()` kết nối với DynamoDB Local thay vì AWS — không cần triển khai và không cần `RUNTIME_CONFIG_APP_ID`.

<Aside type="tip" title="Lần chạy đầu tiên">
Lần chạy đầu tiên sẽ pull container image DynamoDB Local, có thể mất một chút thời gian. Để server chạy trong terminal này và mở terminal thứ hai cho các lệnh `curl` bên dưới.
</Aside>

### Kiểm thử API

Khi server của bạn đã chạy, truy vấn danh sách trò chơi (rỗng):

```bash
curl -X GET 'http://localhost:2022/games.query?input=%7B%7D'
```

:::note[Mã hóa URL]
`%7B%7D` mà chúng ta truyền để kiểm thử API là một đối tượng JSON rỗng được mã hóa URI (`{}`).
:::

Bạn sẽ thấy một danh sách rỗng:

```json
{"result":{"data":{"items":[],"cursor":null}}}
```

Bây giờ lưu một trò chơi:

```bash
curl -X POST 'http://localhost:2022/games.save' \
  -H 'Content-Type: application/json' \
  -d '{"playerName":"Alice","genre":"zombie"}'
```

Lệnh save trả về trò chơi đã được lưu (với timestamp `lastUpdated` mà entity đặt cho bạn):

```json
{"result":{"data":{"playerName":"Alice","genre":"zombie","lastUpdated":"..."}}}
```

Truy vấn lại để xác nhận nó đã được lưu trong DynamoDB Local:

```bash
curl -X GET 'http://localhost:2022/games.query?input=%7B%7D'
```

Phản hồi này bây giờ bao gồm trò chơi đã lưu:

```json
{"result":{"data":{"items":[{"playerName":"Alice","genre":"zombie","lastUpdated":"..."}],"cursor":null}}}
```

Bạn có thể dừng local server (`Ctrl+C`) khi đã xong.

## Nhiệm vụ 5: Kiểm thử Inventory MCP server locally

Chúng ta có thể thử các công cụ của MCP server với [MCP Inspector](https://github.com/modelcontextprotocol/inspector) bằng cách sử dụng target `mcp-server-inspect` được tạo. Inspector kết nối với server qua STDIO, vì vậy chúng ta chạy DynamoDB Local riêng biệt và yêu cầu server sử dụng nó.

Trong một terminal, khởi động DynamoDB Local:

<NxCommands commands={["serve-local dungeon-db"]} />

Trong terminal thứ hai, khởi động inspector:

<NxCommands commands={["mcp-server-inspect inventory"]} />

Điều này khởi động MCP Inspector tại `http://localhost:6274`. Trong giao diện inspector, trước khi kết nối, thêm một biến môi trường `SERVE_LOCAL` với giá trị `true` để các công cụ của server giao tiếp với DynamoDB Local, sau đó nhấp **Connect**.

<Aside type="tip" title="Tại sao đặt SERVE_LOCAL?">
Không giống như `serve-local`, target `mcp-server-inspect` chạy plain STDIO server, vì vậy nó không đặt `SERVE_LOCAL` cho bạn. Đặt nó trong môi trường của inspector làm cho `getDynamoDBClient()` và `resolveTableName()` kết nối với bảng local thay vì AWS.
</Aside>

Chuyển sang tab **Tools**, nhấp **List Tools**, và thử `add-to-inventory` (ví dụ: `playerName: Alice`, `itemName: Rusty Sword`, `emoji: ⚔️`) theo sau là `list-inventory-items` để xem nó được lưu vào DynamoDB Local. Dừng cả hai terminal (`Ctrl+C`) khi đã xong.

<Aside type="tip" title="Thích HTTP transport hơn?">
Nếu bạn muốn điều khiển server qua HTTP (ví dụ: cho scripting), target `mcp-server-serve-local` chạy Streamable HTTP transport với `SERVE_LOCAL=true` được đặt cho bạn, và tự động khởi động DynamoDB Local:

<NxCommands commands={["run @dungeon-adventure/inventory:mcp-server-serve-local"]} />

Sau đó khởi tạo một session với `http://localhost:8000/mcp`:
```bash
curl -X POST http://localhost:8000/mcp \
  -H 'Content-Type: application/json' \
  -H 'Accept: application/json, text/event-stream' \
  -d '{"jsonrpc":"2.0","id":1,"method":"initialize","params":{"protocolVersion":"2025-03-26","capabilities":{},"clientInfo":{"name":"curl","version":"1.0.0"}}}'
```
</Aside>

Chúc mừng, bạn đã xây dựng và kiểm thử API tRPC đầu tiên và MCP server với bảng DynamoDB local!  🎉🎉🎉
