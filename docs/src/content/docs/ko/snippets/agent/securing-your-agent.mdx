title: 에이전트 보안
---

에이전트는 신뢰할 수 없는 입력에 대해 작동하며 도구를 통해 실제 작업을 수행할 수 있으므로, 처음부터 보안을 고려할 가치가 있습니다. 다음 관행은 생성된 에이전트에 적용됩니다.

### 모델 입력 및 출력을 신뢰할 수 없는 것으로 취급

프롬프트에는 적대적인 지시사항(프롬프트 인젝션)이 포함될 수 있으며, 모델 출력은 비결정적입니다. 따라서 보안에 민감한 로직에서는 둘 다 신뢰해서는 안 됩니다:

- 생성된 예제 도구와 같이 도구에 대한 엄격한 입력 스키마를 정의하세요. 자유 형식 문자열을 허용하는 대신 도구가 실제로 필요로 하는 값(열거형, 길이 제한, 숫자 범위)으로 제한하세요.
- 검증이나 인코딩 없이 모델 출력을 셸 명령, SQL 쿼리, 코드 평가 또는 렌더링된 HTML에 직접 전달하지 마세요.
- 도구 및 다운스트림 서비스에 권한 부여 검사를 적용하세요. 모델이 액세스 권한이 있는 도구를 오용하는 것을 방지하기 위해 시스템 프롬프트에만 의존하지 마세요.

### 도구 권한을 엄격하게 범위 지정

에이전트의 IAM 역할에 도구가 필요로 하는 권한만 부여하세요. 제공되는 CDK 구성 및 Terraform 모듈은 이러한 목적을 위해 `grant*` 메서드와 범위가 지정된 정책을 노출합니다. 예를 들어 광범위한 관리형 정책을 연결하는 대신 특정 API를 호출할 수 있는 액세스 권한을 에이전트에 부여합니다. 도구가 사용자를 대신하여 작동하는 경우, 에이전트 자체의 주변 권한보다 호출하는 사용자의 ID(요청 컨텍스트를 통해 전달됨)를 사용하여 작업을 승인하는 것을 선호하세요.

### 킬 스위치 제공

모델 동작이 예상치 못한 방식으로 변경될 수 있으므로, 코드 변경 없이 모델을 신속하게 비활성화하거나 교체할 수 있도록 계획하세요:

- 운영자가 구성을 업데이트하여 다른 모델로 전환하거나 롤백할 수 있도록 구성에서 모델 ID를 읽으세요(예: `MODEL_ID` 환경 변수).
- 에이전트를 기능 플래그 뒤에 배치하여 AI 기능을 완전히 비활성화할 수 있도록 하세요. 비활성화된 경우 오류 대신 일반 메시지를 반환하고 애플리케이션의 나머지 부분이 정상적으로 저하되도록 하세요.

운영 런북에 이러한 제어를 전환하는 방법을 문서화하세요.

### 민감한 데이터 보호

- 사용자 데이터가 포함될 수 있는 프롬프트 및 완료를 로깅하지 마세요. 생성된 에이전트의 모델 오류 로깅 훅은 대화 내용이 아닌 오류 메타데이터만 로깅합니다. 자체 로깅을 추가할 때 이 속성을 유지하세요.
- 사용자에게 일반적인 오류 메시지를 반환하고, 자세한 오류는 서버 측에서 로깅하세요.
- 사용자와 세션 간의 대화 상태를 격리하고, 지속된 세션 데이터에 대한 액세스를 승인하세요.
