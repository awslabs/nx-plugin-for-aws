TypeScript 프로젝트 생성기는 [TypeScript](https://www.typescriptlang.org/)로 최신 라이브러리 또는 애플리케이션을 생성하는 데 사용할 수 있으며, [ECMAScript 모듈 (ESM)](https://www.typescriptlang.org/docs/handbook/modules/reference.html), TypeScript [프로젝트 참조](https://www.typescriptlang.org/docs/handbook/project-references.html), 테스트 실행을 위한 [Vitest](https://vitest.dev/), 정적 분석을 위한 [ESLint](https://eslint.org/)와 같은 모범 사례로 구성됩니다.

## 사용법

### TypeScript 프로젝트 생성

다음 두 가지 방법으로 새로운 TypeScript 프로젝트를 생성할 수 있습니다:

<RunGenerator generator="ts#project" />

## 제너레이터 출력

제너레이터는 `<directory>/<name>` 디렉터리에 다음과 같은 프로젝트 구조를 생성합니다:

<FileTree>

  - src TypeScript 소스 코드
    - index.ts
  - project.json 프로젝트 구성 및 빌드 대상
  - tsconfig.json 이 프로젝트의 기본 TypeScript 구성 (워크스페이스 루트 tsconfig.base.json 확장)
  - tsconfig.lib.json 라이브러리를 위한 TypeScript 구성 (런타임 또는 패키지된 소스)
  - tsconfig.spec.json 테스트를 위한 TypeScript 구성
  - vite.config.ts Vitest 구성
  - eslint.config.mjs ESLint 구성

</FileTree>

:::tip
이 프로젝트에 대해 `package.json` 파일이 생성되지 않습니다! 그 이유는 [아래](#dependencies)에서 확인할 수 있습니다.
:::

워크스페이스 루트의 다음 파일들에 변경 사항이 있음을 알 수 있습니다:

<FileTree>

  - nx.json Nx 구성이 프로젝트의 @nx/js/typescript 플러그인을 구성하도록 업데이트됨
  - tsconfig.base.json 다른 프로젝트에서 가져올 수 있도록 프로젝트에 대한 TypeScript 별칭 설정
  - tsconfig.json 프로젝트에 대한 TypeScript 프로젝트 참조 추가

</FileTree>

## TypeScript 소스 코드 작성하기

`src` 디렉토리에 TypeScript 코드를 추가하세요.

### ESM 가져오기 구문

TypeScript 프로젝트가 ES 모듈이므로, 파일 확장자를 명시적으로 참조하여 가져오기 문을 올바르게 작성해야 합니다:

```ts title="index.ts" ".js"
import { sayHello } from './hello.js';
```

:::note
TypeScript를 사용하고 있으며 `sayHello`가 `hello.ts`에 정의되어 있더라도, 가져오기에는 `.js` 파일 확장자를 사용합니다. 자세한 내용은 [여기](https://www.typescriptlang.org/docs/handbook/modules/reference.html)에서 확인할 수 있습니다.
:::

### 다른 TypeScript 프로젝트로 내보내기

TypeScript 프로젝트의 진입점은 `src/index.ts`입니다. 다른 프로젝트에서 가져올 수 있는 항목을 여기에 내보낼 수 있습니다:

```ts title="src/index.ts"
export { sayHello } from './hello.js';
export * from './algorithms/index.js';
```

### 다른 프로젝트에서 라이브러리 코드 가져오기

프로젝트의 [TypeScript 별칭](https://www.typescriptlang.org/docs/handbook/modules/reference.html#paths)은 작업 공간 `tsconfig.base.json`에 구성되어 있어 다른 TypeScript 프로젝트에서 TypeScript 프로젝트를 참조할 수 있습니다:

```ts title="packages/my-other-project/src/index.ts"
import { sayHello } from ':my-scope/my-library';
```

:::note
TypeScript 프로젝트의 별칭은 전통적인 `@` 대신 `:`로 시작하여 작업 공간의 로컬 패키지와 [NPM](https://www.npmjs.com/)의 원격 패키지 간 이름 충돌 가능성을 방지합니다.
:::

작업 공간의 새 프로젝트에 대한 가져오기 문을 처음 추가할 때 IDE에서 다음과 유사한 오류가 표시될 수 있습니다:

<details>
<summary>가져오기 오류</summary>

```bash wrap
File '/path/to/my/workspace/packages/my-library/src/index.ts' is not under 'rootDir' '/path/to/my/workspace/packages/my-consumer'. 'rootDir' is expected to contain all source files.
  File is ECMAScript module because '/path/to/my/workspace/package.json' has field "type" with value "module" ts(6059)
File '/path/to/my/workspace/packages/my-library/src/index.ts' is not listed within the file list of project '/path/to/my/workspace/packages/my-consumer/tsconfig.lib.json'. Projects must list all files or use an 'include' pattern.
  File is ECMAScript module because '/path/to/my/workspace/package.json' has field "type" with value "module" ts(6307)
```

</details>

이는 [프로젝트 참조](https://www.typescriptlang.org/docs/handbook/project-references.html)가 아직 설정되지 않았기 때문입니다.

TypeScript 프로젝트는 Nx TypeScript 동기화 생성기로 기본적으로 구성되어 있어 프로젝트 참조를 수동으로 구성할 필요가 없습니다. 다음 명령을 실행하면 Nx가 필요한 구성을 추가합니다:

<NxCommands commands={['sync']} />

이후 IDE의 오류가 사라지고 라이브러리를 사용할 준비가 됩니다.

:::tip
프로젝트를 빌드하면 다음과 같은 메시지가 표시됩니다:

```bash wrap
[@nx/js:typescript-sync]: Some TypeScript configuration files are missing project references to the projects they depend on or contain outdated project references.

This will result in an error in CI.

? Would you like to sync the identified changes to get your workspace up to date?
Yes, sync the changes and run the tasks
No, run the tasks without syncing the changes
```

작업 공간을 최신 상태로 만들려면 `Yes`를 선택하여 Nx가 프로젝트 참조를 업데이트하도록 하세요.
:::

### 종속성

TypeScript 프로젝트에 `package.json` 파일이 없다는 것을 알 수 있을 것입니다. 이는 전통적인 TypeScript 모노레포에 익숙하다면 예상치 못한 상황일 수 있습니다.

모노레포의 모든 TypeScript 패키지에 종속성을 추가하려면 작업 공간 루트의 `package.json`에 종속성을 추가하면 됩니다. 패키지 관리자의 명령줄을 통해 이를 수행할 수 있습니다:

<InstallCommand pkg="some-npm-package" />

그러면 작업 공간의 모든 TypeScript 프로젝트에서 해당 종속성을 사용할 수 있습니다.

#### 런타임 코드

TypeScript 프로젝트를 런타임 코드(예: AWS Lambda 함수 핸들러)로 사용할 때는 [`esbuild`](https://esbuild.github.io/)와 같은 도구를 사용하여 프로젝트를 번들링하는 것이 좋습니다. 이를 통해 [트리 쉐이킹](https://esbuild.github.io/api/#tree-shaking)을 수행하여 프로젝트에서 실제로 참조하는 종속성만 포함할 수 있습니다.

`project.json` 파일에 다음과 같은 대상을 추가하여 이를 달성할 수 있습니다:

```json
{
  ...
  "targets": {
    ...
    "bundle": {
      "cache": true,
      "executor": "nx:run-commands",
      "outputs": ["{workspaceRoot}/dist/packages/my-library/bundle"],
      "options": {
        "command": "esbuild packages/my-library/src/index.ts --bundle --outfile=dist/packages/my-library/bundle/index.js --platform=node --format=cjs"
      }
    },
  },
}
```

:::note
위의 대상에서 번들의 진입점으로 `src/index.ts`를 선택했으므로, 이 파일에서 내보낸 코드와 모든 종속성이 번들에 포함됩니다.
:::

#### NPM에 게시하기

TypeScript 프로젝트를 NPM에 게시하려면 `package.json` 파일을 생성해야 합니다.

이 파일은 프로젝트가 참조하는 종속성을 선언해야 합니다. 빌드 시 작업 공간 루트 `package.json`을 통해 설치된 종속성을 확인하므로, 게시된 프로젝트의 `package.json`에 프로젝트에서 사용하는 모든 종속성이 포함되도록 [Nx 종속성 검사 ESLint 플러그인](https://nx.dev/nx-api/eslint-plugin/documents/dependency-checks)을 구성하는 것이 좋습니다.

### 빌드

TypeScript 프로젝트는 `project.json`에 정의된 `build` 대상으로 구성되어 있으며, 다음과 같이 실행할 수 있습니다:

<NxCommands commands={['run <project-name>:build']} />

여기서 `<project-name>`은 프로젝트의 정규화된 이름입니다.

`build` 대상은 프로젝트를 컴파일, 린트 및 테스트합니다.

빌드 출력은 작업 공간의 루트 `dist` 폴더에서 찾을 수 있으며, 패키지 및 대상의 디렉토리 내부에 있습니다. 예: `dist/packages/<my-library>/tsc`

## 테스팅

[Vitest](https://vitest.dev/)가 프로젝트 테스팅을 위해 구성되었습니다.

### 테스트 작성

테스트는 프로젝트의 `src` 폴더에 위치한 `.spec.ts` 또는 `.test.ts` 파일에 작성해야 합니다.

예를 들어:

<FileTree>
  - src
    - hello.ts 라이브러리 소스 코드
    - hello.spec.ts hello.ts에 대한 테스트
</FileTree>

Vitest는 `describe`, `it`, `test`, `expect`와 같은 유틸리티를 포함하여 Jest와 유사한 구문으로 테스트를 정의할 수 있습니다.

```ts title="hello.spec.ts"
import { sayHello } from './hello.js';

describe('sayHello', () => {

  it('should greet the caller', () => {
    expect(sayHello('Darth Vader')).toBe('Hello, Darth Vader!');
  });

});

```

테스트 작성 방법 및 종속성 모킹과 같은 기능에 대한 자세한 내용은 [Vitest 문서](https://vitest.dev/guide/#writing-tests)를 참조하세요.

### 테스트 실행

테스트는 프로젝트의 `build` 대상의 일부로 실행되지만, `test` 대상을 실행하여 별도로 실행할 수도 있습니다:

<NxCommands commands={['run <project-name>:test']} />

`-t` 플래그를 사용하여 개별 테스트 또는 테스트 모음을 실행할 수 있습니다:

<NxCommands commands={["run <project-name>:test -t 'sayHello'"]} />

:::tip
VSCode 사용자의 경우, IDE에서 테스트를 실행, 감시 또는 디버깅할 수 있는 [실제로 작동하는 VSCode용 Vitest 러너](https://marketplace.visualstudio.com/items?itemName=rluvaton.vscode-vitest) 확장 프로그램을 설치하는 것이 좋습니다.
:::

## 린팅

TypeScript 프로젝트는 린팅을 위해 [ESLint](https://eslint.org/)를, 포맷팅을 위해 [Prettier](https://prettier.io/)를 사용합니다.

작업 공간 루트의 `eslint.config.mjs` 파일에서 ESLint를 구성하는 것을 권장합니다. 이 파일에 대한 변경 사항은 작업 공간의 모든 TypeScript 프로젝트에 적용되어 일관성을 보장합니다.

마찬가지로, 루트의 `.prettierrc` 파일에서 Prettier를 구성할 수 있습니다.

### 린터 실행

프로젝트를 검사하기 위해 린터를 호출하려면 `lint` 대상을 실행할 수 있습니다.

<NxCommands commands={["run <project-name>:lint"]} />

### 린트 문제 수정

대부분의 린팅 또는 포맷팅 문제는 자동으로 수정할 수 있습니다. `--configuration=fix` 인수와 함께 실행하여 ESLint에 린트 문제를 수정하도록 지시할 수 있습니다.

<NxCommands commands={["run <project-name>:lint --configuration=fix"]} />

마찬가지로 작업 공간의 모든 패키지에서 모든 린트 문제를 수정하려면 다음을 실행할 수 있습니다:

<NxCommands commands={["run-many --target lint --all --configuration=fix"]} />
