<FileTree>

또한 tRPC API의 `serve-local` 타겟을 업데이트하여 로컬에서 실행할 때 데이터베이스가 자동으로 시작되도록 합니다.

## 미들웨어 사용하기

### 플러그인 등록

생성된 플러그인을 tRPC 라우터에 추가하면 이를 사용하는 모든 프로시저가 데이터베이스에 접근할 수 있습니다:

```ts title="packages/api/src/router.ts" {2,5}
import { t } from './init.js';
import { createMyDbPlugin } from './middleware/my-db.js';

export const authenticatedProcedure = t.procedure
  .concat(createMyDbPlugin());
```

### 프로시저에서 데이터베이스 접근

플러그인은 `IMyDbContext`를 프로시저 컨텍스트에 병합하여 `myDb`를 선택적 속성으로 사용할 수 있게 합니다:

```ts title="packages/api/src/procedures/users.ts" {8,12}
import { z } from 'zod';
import { authenticatedProcedure } from '../router.js';

export const listUsers = authenticatedProcedure
  .output(z.array(z.object({ id: z.string(), name: z.string() })))
  .query(async ({ ctx }) => {
    // ctx.myDb는 Prisma 클라이언트입니다 — Awaited<ReturnType<typeof getPrisma>>로 타입 지정됨
    return await ctx.myDb!.user.findMany();
  });
```

### MySQL: 각 요청 후 연결 해제

타겟 데이터베이스가 MySQL 엔진을 사용하는 경우, 생성된 미들웨어는 `opts.next()`를 `try/finally` 블록으로 감싸서 `$disconnect()`를 호출합니다:

```ts title="packages/api/src/middleware/my-db.ts" {6}
return t.procedure.use(async (opts) => {
  const myDb = await getPrisma();
  try {
    return await opts.next({ ctx: { ...opts.ctx, myDb } });
  } finally {
    await myDb.$disconnect();
  }
});
```

이는 MySQL 어댑터가 쿼리 후 Node.js 이벤트 루프를 열린 상태로 유지하는 문제를 해결합니다. 이를 해결하지 않으면 Lambda가 스트리밍 응답을 플러시하지 못할 수 있습니다. `finally`에서 연결을 해제하면 이벤트 루프가 해제되어 응답이 완료될 수 있습니다. 자세한 내용은 <Link path="guides/ts-rdb#mysql-api-gateway-streaming-mode">MySQL: API Gateway 스트리밍 모드</Link>를 참조하세요.

PostgreSQL은 이 작업이 필요하지 않습니다 — 해당 어댑터는 `allowExitOnIdle: true`로 구성된 연결 풀을 사용합니다.

:::caution[MySQL: IAM 토큰 만료]
RDS IAM 토큰은 **15분** 후에 만료됩니다. MySQL 어댑터는 연결 시점에 토큰을 캡처합니다. Lambda가 토큰이 만료될 만큼 충분히 오래 웜 상태를 유지하면 새로운 연결 시도가 실패할 수 있습니다. 연결 오류를 확인하고 `getPrisma()`를 다시 호출하여 새 토큰이 있는 새 클라이언트를 얻으세요. 자세한 내용과 해결 방법은 <Link path="guides/ts-rdb#mysql-iam-token-expiry">MySQL: IAM 토큰 만료</Link>를 참조하세요.
:::

### 여러 데이터베이스

다른 타겟으로 생성기를 다시 실행하여 추가 데이터베이스를 연결할 수 있습니다. 각 데이터베이스는 자체 플러그인과 컨텍스트 인터페이스를 갖습니다:

```ts title="packages/api/src/router.ts"
export const dbProcedure = t.procedure
  .concat(createPostgresDbPlugin())
  .concat(createMySqlDbPlugin());
```

## 인프라

런타임에 tRPC API가 데이터베이스에 연결할 수 있도록 하려면, API Lambda 함수가 데이터베이스와 동일한 VPC에 배포되어야 하며 네트워크 및 IAM 액세스 권한이 부여되어야 합니다.

<Infrastructure>
<Fragment slot="cdk">

애플리케이션 스택에서 API를 데이터베이스와 동일한 VPC에 배포한 다음, `allowDefaultPortFrom` 및 `grantConnect`를 호출하여 네트워크 경로를 열고 각 Lambda 핸들러에 IAM `rds-db:connect` 권한을 부여하세요:

```ts title="packages/infra/src/stacks/application-stack.ts"
import { MyDatabase } from ':my-scope/common-constructs';

const db = new MyDatabase(this, 'Db', { vpc, ... });

const api = new Api(this, 'Api', {
  integrations: Api.defaultIntegrations(this)
    .withDefaultOptions({
      vpc,
      vpcSubnets: { subnetType: SubnetType.PRIVATE_WITH_EGRESS },
    })
    .build(),
});

Object.entries(api.integrations).forEach(([operation, integration]) => {
  db.allowDefaultPortFrom(integration.handler, `Allow ${operation} to connect to the database`);
  db.grantConnect(integration.handler);
});
```

API Lambda 함수를 프라이빗 격리 서브넷이 아닌 **이그레스가 있는 프라이빗 서브넷**에 배포하세요. 런타임에 `getPrisma()`는 AWS AppConfig에서 데이터베이스 연결 세부 정보를 가져오는데, 이는 아웃바운드 인터넷 액세스가 필요한 퍼블릭 AWS 서비스 엔드포인트입니다.

:::note
이 예제는 API의 모든 핸들러에 액세스 권한을 부여하지만, 일부 핸들러만 액세스가 필요한 경우 개별적으로 구성하는 것이 좋습니다.
:::

module "my_database" {

  lambda_subnet_ids   = module.vpc.private_subnet_ids

  source             = "..."
  vpc_id             = module.vpc.vpc_id
  private_subnet_ids = module.vpc.private_subnet_ids

    RUNTIME_CONFIG_APP_ID = module.my_database.appconfig_application_id
  }

API Lambda 함수를 프라이빗 격리 서브넷이 아닌 **이그레스가 있는 프라이빗 서브넷**에 배포하세요. API Lambda 역할에 `rds-db:connect` 권한이 있고 해당 보안 그룹이 데이터베이스 포트에서 데이터베이스 보안 그룹에 도달할 수 있는지 확인하세요.

## 로컬 개발

생성기는 tRPC API의 `serve-local` 타겟이 데이터베이스의 `serve-local` 타겟에 의존하도록 구성하므로 다음을 실행하면:

<NxCommands commands={["serve-local <api-project-name>"]} />

API와 함께 로컬 데이터베이스가 자동으로 시작됩니다.
