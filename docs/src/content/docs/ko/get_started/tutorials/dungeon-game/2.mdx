## 모듈 2: 게임 API 구현

먼저 게임 API를 구현해 보겠습니다. 이를 위해 총 4개의 API를 생성해야 합니다:

1. `createGame` - 새로운 게임 인스턴스를 생성합니다.
2. `queryGames` - 이전에 저장된 게임 목록을 페이지네이션 형식으로 반환합니다.
3. `saveAction` - 특정 게임에 대한 액션을 저장합니다.
4. `queryActions` - 게임과 관련된 모든 액션의 페이지네이션 목록을 반환합니다.

### 배포 및 테스트

먼저 코드베이스를 빌드합니다:

<NxCommands commands={['run-many --target build --all']} />

<Aside type="tip">
린트 오류가 발생하면 다음 명령어를 실행하여 자동으로 수정할 수 있습니다.

<NxCommands commands={['run-many --target lint --configuration=fix --all']} />
</Aside>

이제 다음 명령어를 실행하여 애플리케이션을 배포할 수 있습니다:

<NxCommands commands={['run @dungeon-adventure/infra:deploy dungeon-adventure-infra-sandbox']} />

첫 배포는 약 8분 정도 소요됩니다. 이후 배포는 약 2분 정도 걸립니다.

:::tip
람다 함수 코드 변경을 반복하는 경우 코드베이스를 빌드한 후 `--hotswap` 플래그를 사용하여 배포하면 배포 시간이 훨씬 짧아집니다(2-3초).

<NxCommands commands={['run @dungeon-adventure/infra:deploy dungeon-adventure-infra-sandbox --hotswap']} />
:::

<Drawer title="배포 명령어" trigger="모든 스택을 한 번에 배포할 수도 있습니다. 자세한 내용을 보려면 클릭하세요.">

CDK 애플리케이션에 포함된 모든 스택을 배포하려면 다음을 실행합니다:

<NxCommands commands={['run @dungeon-adventure/infra:deploy --all']} />

인프라 프로덕션과 같은 배포 단계를 별도의 스택으로 분리할 수 있으므로 `--all` 플래그는 모든 스택을 배포하려고 시도할 수 있어 원치 않는 배포가 발생할 수 있으므로 **권장하지 않습니다**!

</Drawer>

다음 방법으로 API를 테스트할 수 있습니다:
<ul>
<li>tRPC 백엔드의 로컬 인스턴스를 시작하고 `curl`을 사용하여 API를 호출합니다.</li>
<li>
<Drawer title="Sigv4 활성화된 curl" trigger="배포된 API를 sigv4 활성화된 curl로 호출하기">
`.bashrc` 파일에 다음 스크립트를 추가하거나(그리고 `source`로 적용) 명령어를 실행할 터미널에 직접 붙여넣을 수 있습니다.
```bash
// ~/.bashrc
acurl () {
    REGION=$1
    SERVICE=$2
    shift; shift;
    curl --aws-sigv4 "aws:amz:$REGION:$SERVICE" --user "$(aws configure get aws_access_key_id):$(aws configure get aws_secret_access_key)" -H "X-Amz-Security-Token: $(aws configure get aws_session_token)" "$@"
}
```

다음 예제와 같이 `acurl`을 호출하여 sigv4 인증된 curl 요청을 할 수 있습니다:

###### API Gateway
```bash
acurl ap-southeast-2 execute-api -X GET https://xxx
```

###### 스트리밍 람다 함수 URL
```bash
acurl ap-southeast-2 lambda -N -X POST https://xxx
```
</Drawer>
</li>
</ul>
