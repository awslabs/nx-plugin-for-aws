## 작업 1: Game API 구현

이 섹션에서는 다음 API를 구현합니다:

1. `saveGame` - 게임 생성 또는 업데이트
2. `queryGames` - 저장된 게임 목록을 페이지네이션으로 반환
3. `queryInventory` - 플레이어 인벤토리 아이템 목록을 페이지네이션으로 반환
4. `queryActions` - 특정 게임의 대화 기록 반환

:::note[각 상태가 저장되는 위치]
- `games`와 `inventory`는 **DynamoDB**에 저장됩니다(ElectroDB 사용) — Game API 람다와 Inventory MCP 서버가 읽기/쓰기를 수행합니다.
- `actions`(채팅 기록)는 **S3**에 저장됩니다 — Story Agent의 `S3SessionManager`가 매 턴마다 하나의 JSON blob을 작성하고, Game API의 `queryActions`가 이를 읽어와 UI가 페이지 새로고침 시 기록을 재생할 수 있도록 합니다. `saveAction` 프로시저는 없으며, 에이전트만이 작성자입니다.
:::

### 엔티티 모델링

애플리케이션의 ER 다이어그램은 다음과 같습니다.

```d2
direction: right

game: Game {
  shape: sql_table
  playerName: "string  PK"
  genre: string
  lastUpdated: string
}

item: Item {
  shape: sql_table
  playerName: "string  PK"
  itemName: "string  SK"
  emoji: "string  (optional)"
  lastUpdated: string
  quantity: number
}

game -> item: 1..*
```

`ts#dynamodb` 생성기가 [ElectroDB](https://electrodb.dev/en/core-concepts/introduction/)를 설정했으며, 이를 사용하여 데이터를 모델링합니다. 대화 기록을 S3에 저장할 것이므로 S3 클라이언트에 대한 종속성을 추가합니다:

## 작업 4: Game API 로컬 테스트

API를 시험하기 위해 AWS에 배포할 필요가 없습니다 — `dev` 타겟은 [DynamoDB Local](https://docs.aws.amazon.com/amazondynamodb/latest/developerguide/DynamoDBLocal.html)에 대해 Game API를 실행합니다. 모듈 1에서 Game API를 `DungeonDb` 프로젝트에 연결했기 때문에 이 타겟은 DynamoDB Local도 자동으로 시작합니다.

먼저 린트 문제를 수정합니다:

<PackageManagerShortCommand commands={["lint"]} />

그런 다음 코드베이스를 빌드합니다:

<PackageManagerShortCommand commands={["build"]} />

### 로컬 서버 시작

`dev` 타겟으로 Game API를 로컬에서 시작합니다. DynamoDB Local도 부팅합니다:

<NxCommands commands={["dev game-api"]} />

<Aside type="tip" title="첫 실행">
첫 실행 시 DynamoDB Local 컨테이너 이미지를 가져오므로 시간이 걸릴 수 있습니다. 이 터미널에서 서버를 실행 상태로 두고 아래 `curl` 명령어를 위해 두 번째 터미널을 여세요.
</Aside>

### API 테스트

서버가 실행되면 (빈) 게임 목록을 쿼리합니다:

```bash
curl -X GET 'http://localhost:2022/games.query?input=%7B%7D'
```

:::note[URL 인코딩]
API 테스트 시 전달하는 `%7B%7D`는 URI 인코딩된 빈 JSON 객체(`{}`)입니다.
:::

빈 목록이 표시됩니다:

```json
{"result":{"data":{"items":[],"cursor":null}}}
```

이제 게임을 저장합니다:

```bash
curl -X POST 'http://localhost:2022/games.save' \
  -H 'Content-Type: application/json' \
  -d '{"playerName":"Alice","genre":"zombie"}'
```

저장은 지속된 게임을 반환합니다(엔티티가 설정한 `lastUpdated` 타임스탬프 포함):

```json
{"result":{"data":{"playerName":"Alice","genre":"zombie","lastUpdated":"..."}}}
```

다시 쿼리하여 DynamoDB Local에 지속되었는지 확인합니다:

```bash
curl -X GET 'http://localhost:2022/games.query?input=%7B%7D'
```

이제 응답에 저장된 게임이 포함됩니다:

```json
{"result":{"data":{"items":[{"playerName":"Alice","genre":"zombie","lastUpdated":"..."}],"cursor":null}}}
```

완료되면 로컬 서버를 중지할 수 있습니다(`Ctrl+C`).

## 작업 5: Inventory MCP 서버 로컬 테스트

생성된 `mcp-server-inspect` 타겟을 사용하여 [MCP Inspector](https://github.com/modelcontextprotocol/inspector)로 MCP 서버의 도구를 시험해볼 수 있습니다:

<NxCommands commands={["mcp-server-inspect inventory"]} />

이렇게 하면 MCP 서버가 로컬에서 제공되고(DynamoDB Local도 부팅) `http://localhost:6274`에서 MCP Inspector가 실행되며 연결하도록 미리 구성됩니다. **Connect**를 클릭하고 **Tools** 탭으로 전환한 다음 **List Tools**를 클릭하고 `add-to-inventory`(예: `playerName: Alice`, `itemName: Rusty Sword`, `emoji: ⚔️`)를 시도한 후 `list-inventory-items`로 DynamoDB Local에 지속되었는지 확인합니다. 완료되면 서버를 중지합니다(`Ctrl+C`).
