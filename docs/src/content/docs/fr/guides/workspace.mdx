Lorsque vous créez un nouvel espace de travail avec `@aws/nx-plugin`, le générateur de preset configure un [monorepo Nx](https://nx.dev) avec des valeurs par défaut sensées pour construire sur AWS — incluant TypeScript, Prettier, la structure de l'espace de travail et les outils de sécurité.

## Création d'un espace de travail

<CreateNxWorkspaceCommand workspace="my-project" />

### Options

| Paramètre | Type | Défaut | Description |
|-----------|------|---------|-------------|
| `iacProvider` | `CDK` \| `Terraform` | — | Préférence de fournisseur d'Infrastructure as Code |
| `gitSecrets` | `boolean` | `true` | Configure les hooks pre-commit git-secrets pour empêcher la validation des identifiants AWS |

<Infrastructure>
<Fragment slot="cdk">
<CreateNxWorkspaceCommand workspace="my-project" iacProvider="CDK" />
</Fragment>
<Fragment slot="terraform">
<CreateNxWorkspaceCommand workspace="my-project" iacProvider="Terraform" />
</Fragment>
</Infrastructure>

Pour désactiver l'analyse git-secrets :

```sh
pnpm create @aws/nx-workspace my-project --no-gitSecrets
```

## Structure de l'espace de travail

<FileTree>
  - packages/ Vos projets se trouvent ici
  - .git-secrets/
    - git-secrets Script bash git-secrets intégré pour l'analyse des identifiants
    - setup.sh Enregistre les modèles d'identifiants AWS dans la configuration git locale
  - .husky/
    - pre-commit Exécute l'analyse git-secrets avant chaque commit
  - nx.json Configuration Nx (cibles, générateurs de synchronisation, analytique)
  - package.json Scripts, devDependencies et hook prepare
  - tsconfig.json Configuration TypeScript racine
  - aws-nx-plugin.config.mts Configuration du plugin (fournisseur IaC, licence, etc.)
</FileTree>

## Git Secrets

Les nouveaux espaces de travail incluent des hooks pre-commit [git-secrets](https://github.com/awslabs/git-secrets) qui analysent les fichiers indexés à la recherche de modèles d'identifiants AWS avant chaque commit. Cela empêche la validation accidentelle de clés d'accès, de clés secrètes et d'autres valeurs sensibles.

### Fonctionnement

<Steps>
1. Le script `prepare` s'exécute après l'installation des packages, ce qui initialise les hooks git husky et enregistre les modèles d'identifiants AWS dans votre configuration git locale.

2. À chaque `git commit`, le hook pre-commit husky invoque git-secrets pour analyser les fichiers indexés.

3. Si un modèle correspondant à un identifiant AWS est détecté, le commit est bloqué.
</Steps>

### Modèles détectés

- **AWS Access Key IDs** — `AKIA...`, `ASIA...` et autres préfixes de clés IAM
- **AWS Secret Access Keys** — Chaînes base64 de 40 caractères avec l'étiquette `SECRET_ACCESS_KEY`
- **AWS Account IDs** — Nombres à 12 chiffres avec l'étiquette `ACCOUNT_ID`
- **Bedrock API Keys** — Clés de longue durée (`ABSK...`) et de courte durée
- **Credentials file values** — Valeurs de clés littérales de `~/.aws/credentials`

Les clés d'exemple bien connues (par exemple, `AKIAIOSFODNN7EXAMPLE`) sont automatiquement mises sur liste blanche.

### Suppression des faux positifs

Si git-secrets bloque un commit qui ne contient pas de véritables identifiants, vous pouvez supprimer la détection de plusieurs façons :

**Autoriser un modèle spécifique globalement (pour tous les futurs commits) :**

```sh
git config --add secrets.allowed 'my-pattern-here'
```

**Utiliser un fichier `.gitallowed` (partagé avec votre équipe via le contrôle de version) :**

Créez un fichier `.gitallowed` à la racine du dépôt avec une regex par ligne :

```text title=".gitallowed"
# Allow test fixtures containing fake key patterns
tests/fixtures/.*
# Allow a specific string
my-safe-pattern-here
```

**Ignorer l'analyse pour un seul commit :**

Pour un contournement ponctuel :

```sh
git commit --no-verify
```

### Prérequis

- **Git** doit être initialisé dans l'espace de travail (`git init`)
- **Bash** doit être disponible dans le PATH (fourni avec Git pour Windows sur Windows)
- Les modèles sont enregistrés lorsque `prepare` s'exécute — exécutez la commande d'installation de votre gestionnaire de packages après le clonage pour réenregistrer
