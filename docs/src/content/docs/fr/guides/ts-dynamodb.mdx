Ce générateur crée un nouveau projet TypeScript basé sur [Amazon DynamoDB](https://aws.amazon.com/dynamodb/), utilisant [ElectroDB](https://electrodb.dev/) pour la modélisation d'entités avec sécurité de type. Il génère le code d'application et l'infrastructure nécessaires pour provisionner et gérer une table DynamoDB en utilisant AWS CDK ou Terraform, avec prise en charge de la conception à table unique et développement local intégré via DynamoDB Local.

## Développement local

### Démarrer DynamoDB Local

Le générateur configure une cible `serve-local` qui démarre une instance [DynamoDB Local](https://docs.aws.amazon.com/amazondynamodb/latest/developerguide/DynamoDBLocal.html) et crée la table :

<NxCommands commands={['run <project-name>:serve-local']} />

Cela effectue automatiquement :
1. Télécharge l'image DynamoDB Local (cible `pull-image`)
2. Démarre un conteneur
3. Crée une table locale avec les index définis dans `dynamodb.config.json`

### Modélisation des données

Le projet généré utilise [ElectroDB](https://electrodb.dev/) pour la modélisation d'entités avec sécurité de type sur une seule table DynamoDB. Ajoutez ou mettez à jour les fichiers d'entités sous `src/entities/`, en utilisant l'exemple d'entité généré comme point de départ.

Exemple de définition d'entité :

```ts title="packages/my-table/src/entities/example.ts"
import { Entity } from 'electrodb';
import { getDynamoDBClient, resolveTableName } from '../client.js';

export const createExampleEntity = async () =>
  new Entity(
    {
      model: {
        entity: 'example',
        version: '1',
        service: 'MyTable',
      },
      attributes: {
        id: {
          type: 'string',
          required: true,
        },
        createdAt: {
          type: 'string',
          required: true,
          default: () => new Date().toISOString(),
          readOnly: true,
        },
        updatedAt: {
          type: 'string',
          required: true,
          default: () => new Date().toISOString(),
          watch: '*',
          set: () => new Date().toISOString(),
        },
      },
      indexes: {
        primary: {
          pk: {
            field: 'pk',
            composite: ['id'],
          },
          sk: {
            field: 'sk',
            composite: [],
          },
        },
      },
    },
    { client: getDynamoDBClient(), table: await resolveTableName() },
  );
```

Pour plus de détails, consultez la [documentation des entités ElectroDB](https://electrodb.dev/en/modeling/entities/).

### Utilisation du client DynamoDB

Le fichier généré `src/client.ts` exporte deux utilitaires clés :

- `getDynamoDBClient()` — retourne un singleton `DynamoDBClient` en cache. Lorsque `SERVE_LOCAL=true`, se connecte à l'instance DynamoDB Local locale ; sinon crée un client AWS utilisant la chaîne d'identification par défaut.
- `resolveTableName()` — retourne le nom de la table DynamoDB. Lorsque `SERVE_LOCAL=true`, retourne la constante du nom de table locale ; sinon récupère le nom depuis AWS AppConfig en utilisant la variable d'environnement `RUNTIME_CONFIG_APP_ID` et le met en cache pour les appels suivants.

### Arrêter DynamoDB Local

L'arrêt de `serve-local` (par exemple avec `Ctrl+C`) supprime automatiquement le conteneur DynamoDB Local, mais préserve le volume nommé afin que vos données persistent entre les redémarrages.

:::caution[Windows]
En raison de limitations dans la gestion des signaux sous Windows, le conteneur n'est pas automatiquement supprimé lorsque `serve-local` est arrêté. Vous devrez le supprimer manuellement :

```bash
<engine> rm -f <scope>-dynamodb
```

Remplacez `<engine>` par votre moteur de conteneur (`docker` ou `finch`) et `<scope>` par la portée de votre espace de travail Nx (par exemple `proj`).
:::

## Ajout/Suppression d'index secondaires globaux

Les GSI sont définis dans `dynamodb.config.json` à la racine du projet. Ajoutez une entrée à `globalSecondaryIndexes` pour chaque GSI, en suivant la convention de nommage de [conception à table unique](https://electrodb.dev/en/core-concepts/single-table-relationships/) pour les clés GSI :

```json title="packages/my-table/dynamodb.config.json"
{
  "globalSecondaryIndexes": [
    {
      "indexName": "gsi1pk-gsi1sk-index",
      "partitionKey": "gsi1pk",
      "sortKey": "gsi1sk"
    }
  ]
}
```

Le champ `sortKey` est optionnel pour les GSI avec clé de hachage uniquement.

Ce fichier de configuration est lu directement depuis la racine du projet par tous les consommateurs :
- **Développement local** — `serve-local` lit `dynamodb.config.json` et crée ou met à jour la table locale pour correspondre à la liste GSI
- **CDK** — le construct lit `dynamodb.config.json` au moment de la synthèse, donc les modifications GSI sont reflétées lors du prochain `cdk deploy`
- **Terraform** — le module lit `dynamodb.config.json` au moment du plan/apply

### Un GSI par déploiement

:::caution
DynamoDB [ne permet pas de créer ou supprimer plus d'un GSI dans une seule mise à jour de table](https://docs.aws.amazon.com/amazondynamodb/latest/developerguide/GSI.OnlineOps.html).

Chaque déploiement peut ajouter ou supprimer **au maximum un GSI**. Si vous devez ajouter ou supprimer plusieurs GSI, faites-le un par un — mettez à jour `dynamodb.config.json`, déployez la stack, puis répétez pour le changement suivant.
:::

## Connexion à la table

Dans n'importe quel projet TypeScript, importez les factories d'entités depuis votre package DynamoDB et utilisez-les directement :

```ts
import { createExampleEntity } from ':my-scope/my-table';

const entity = await createExampleEntity();
const result = await entity.query.primary({ id: '123' }).go();
```

:::note[Configuration d'exécution]
Lors de l'exécution dans AWS, `resolveTableName()` récupère le nom de la table depuis AWS AppConfig en utilisant la variable d'environnement `RUNTIME_CONFIG_APP_ID`. Les projets construits avec ce plugin (API tRPC, API Smithy, agents, serveurs MCP) ont déjà cette variable configurée automatiquement. Pour les autres projets TypeScript, assurez-vous que `RUNTIME_CONFIG_APP_ID` est définie dans l'environnement d'exécution avec l'ID d'application AppConfig provisionné par votre infrastructure. Pour plus d'informations, consultez le <Link path="guides/runtime-config">guide de configuration d'exécution</Link>.
:::

## Déploiement de votre table

Le générateur DynamoDB crée une infrastructure CDK ou Terraform en fonction de votre `iac` sélectionné.

<Infrastructure>
<Fragment slot="cdk">
Le construct CDK est créé dans `common/constructs`. Exemple d'utilisation :

```ts title="packages/infra/src/stacks/application-stack.ts"
import { MyTable } from ':my-scope/common-constructs';

export class ApplicationStack extends Stack {
  constructor(scope: Construct, id: string, props?: StackProps) {
    super(scope, id, props);

    const table = new MyTable(this, 'Table');
  }
}
```

Cela provisionne une table DynamoDB avec :
- `pk` (clé de partition) et `sk` (clé de tri), toutes deux de type `String`
- Index Secondaires Globaux tels que définis dans `dynamodb.config.json`
- Facturation à la demande (`PAY_PER_REQUEST`)
- Chiffrement KMS géré par le client avec rotation automatique des clés
- Récupération à un instant dans le temps activée
- Protection contre la suppression activée
- Nom de table enregistré dans Runtime Config sous l'espace de noms `dynamodb` dans AWS AppConfig
</Fragment>

### Protection contre la suppression

La protection contre la suppression est activée par défaut pour empêcher la suppression accidentelle de la table.

#### Désactiver la protection contre la suppression

Désactivez-la pour les environnements où la suppression de table est attendue, comme les stacks de développement ou de prévisualisation éphémères.

<Infrastructure>
<Fragment slot="cdk">

```ts title="packages/infra/src/stacks/application-stack.ts"
import { MyTable } from ':my-scope/common-constructs';

const table = new MyTable(this, 'Table', {
  deletionProtection: false,
});
```
</Fragment>
<Fragment slot="terraform">

```hcl title="packages/infra/src/main.tf"
module "my_table" {
  source                      = "../../common/terraform/src/app/dynamodb/my-table"
  deletion_protection_enabled = false
}
```
</Fragment>
</Infrastructure>

### Mode de facturation

La table utilise par défaut la facturation à la demande (`PAY_PER_REQUEST`). Passez à la capacité provisionnée pour des charges de travail prévisibles à haut débit.

<Infrastructure>
<Fragment slot="cdk">

```ts title="packages/infra/src/stacks/application-stack.ts"
import { BillingMode } from 'aws-cdk-lib/aws-dynamodb';
import { MyTable } from ':my-scope/common-constructs';

const table = new MyTable(this, 'Table', {
  billingMode: BillingMode.PROVISIONED,
  readCapacity: 5,
  writeCapacity: 5,
});
```
</Fragment>
<Fragment slot="terraform">

```hcl title="packages/infra/src/main.tf"
module "my_table" {
  source       = "../../common/terraform/src/app/dynamodb/my-table"
  billing_mode = "PROVISIONED"
}
```
</Fragment>
</Infrastructure>

### Récupération à un instant dans le temps

La [récupération à un instant dans le temps](https://docs.aws.amazon.com/amazondynamodb/latest/developerguide/Point-in-time-recovery.html) est activée par défaut, vous permettant de restaurer la table à n'importe quel moment des 35 derniers jours.

#### Désactiver la récupération à un instant dans le temps

<Infrastructure>
<Fragment slot="cdk">

```ts title="packages/infra/src/stacks/application-stack.ts"
import { MyTable } from ':my-scope/common-constructs';

const table = new MyTable(this, 'Table', {
  pointInTimeRecoverySpecification: { pointInTimeRecoveryEnabled: false },
});
```
</Fragment>
<Fragment slot="terraform">

```hcl title="packages/infra/src/main.tf"
module "my_table" {
  source                         = "../../common/terraform/src/app/dynamodb/my-table"
```
