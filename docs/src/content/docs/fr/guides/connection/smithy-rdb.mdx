Le générateur `connection` relie une <Link path="guides/ts-smithy-api">API Smithy</Link> à un projet de <Link path="guides/ts-rdb">Base de Données Relationnelle</Link>, en injectant un client Prisma dans le contexte du service afin que toutes les implémentations d'opérations puissent accéder à la base de données.

## Prérequis

Avant d'utiliser ce générateur, assurez-vous d'avoir :

1. Un projet <Link path="guides/ts-smithy-api">`ts#smithy-api`</Link> (backend TypeScript)
2. Un projet <Link path="guides/ts-rdb">`ts#rdb`</Link>

## Utilisation

### Exécuter le Générateur

## Sortie du Générateur

Le générateur modifie trois fichiers existants dans votre backend API Smithy :

<FileTree>

- packages/api/src
  - context.ts propriété `db` ajoutée à `ServiceContext`
  - handler.ts client Prisma créé dans `lambdaHandler`, passé à `serviceHandler.handle`
  - local-server.ts client Prisma créé dans le gestionnaire de requêtes, passé à `serviceHandler.handle`

</FileTree>

De plus, il met à jour la cible `serve-local` de l'API pour démarrer automatiquement la base de données.

## Fonctionnement

### ServiceContext

Le générateur ajoute une propriété `db` typée à `ServiceContext` dans `context.ts` :

```ts title="packages/api/src/context.ts" {1,7}
import { getPrisma as getMyDb } from ':my-scope/my-db';

export interface ServiceContext {
  tracer: Tracer;
  logger: Logger;
  metrics: Metrics;
  myDb: Awaited<ReturnType<typeof getMyDb>>;
}
```

### Gestionnaire Lambda

Le client Prisma est instancié dans `lambdaHandler` et transmis via le contexte du service :

```ts title="packages/api/src/handler.ts" {1,10}
import { getPrisma as getMyDb } from ':my-scope/my-db';

export const lambdaHandler = async (event: APIGatewayProxyEvent) => {
  const httpRequest = convertEvent(event);
  const myDb = await getMyDb();
  const httpResponse = await serviceHandler.handle(httpRequest, {
    tracer,
    logger,
    metrics,
    myDb,
  });
  return convertVersion1Response(httpResponse);
};
```

### Utilisation de la Base de Données dans les Opérations

Accédez à `db` depuis le contexte dans vos implémentations d'opérations :

```ts title="packages/api/src/operations/list-users.ts" {8}
import { ListUsersOperationInput, ListUsersOperationOutput } from '../generated/ssdk/index.js';
import { ServiceContext } from '../context.js';

export const listUsers = async (
  input: ListUsersOperationInput,
  ctx: ServiceContext,
): Promise<ListUsersOperationOutput> => {
  const users = await ctx.myDb.user.findMany();
  return { users };
};
```

## Infrastructure

Pour permettre à votre API Smithy de se connecter à la base de données lors de l'exécution, les fonctions Lambda de l'API doivent être déployées dans le même VPC que la base de données et bénéficier d'un accès réseau et IAM.

<Infrastructure>
<Fragment slot="cdk">

Dans votre stack d'application, déployez l'API dans le même VPC que la base de données, puis appelez `allowDefaultPortFrom` et `grantConnect` pour ouvrir le chemin réseau et accorder la permission IAM `rds-db:connect` à chaque gestionnaire Lambda :

```ts title="packages/infra/src/stacks/application-stack.ts"
import { MyDatabase } from ':my-scope/common-constructs';

const db = new MyDatabase(this, 'Db', { vpc, ... });

const api = new MyApi(this, 'Api', {
  integrations: MyApi.defaultIntegrations(this)
    .withDefaultOptions({
      vpc,
      vpcSubnets: { subnetType: SubnetType.PRIVATE_WITH_EGRESS },
    })
    .build(),
});

Object.entries(api.integrations).forEach(([operation, integration]) => {
  db.allowDefaultPortFrom(integration.handler, `Allow ${operation} to connect to the database`);
  db.grantConnect(integration.handler);
});
```

Déployez les fonctions Lambda de l'API dans un **sous-réseau privé avec sortie**, et non dans un sous-réseau privé isolé. Lors de l'exécution, `getPrisma()` récupère les détails de connexion à la base de données depuis AWS AppConfig, ce qui nécessite un accès Internet sortant.

:::note
Cet exemple accorde l'accès à tous les gestionnaires de votre API, mais si seuls certains gestionnaires ont besoin d'accès, il est préférable de les configurer individuellement.
:::

</Fragment>
<Fragment slot="terraform">

Transmettez les sorties du module de base de données dans votre module API afin qu'il puisse atteindre la base de données et lire sa configuration d'exécution :

```hcl title="packages/infra/src/main.tf"
module "my_database" {
  source              = "../../common/terraform/src/app/dbs/my-database"
  vpc_id              = module.vpc.vpc_id
  database_subnet_ids = module.vpc.private_isolated_subnet_ids
  lambda_subnet_ids   = module.vpc.private_subnet_ids
}

module "api" {
  source             = "..."
  vpc_id             = module.vpc.vpc_id
  private_subnet_ids = module.vpc.private_subnet_ids

  appconfig_application_id     = module.my_database.appconfig_application_id
  database_cluster_resource_id = module.my_database.cluster_resource_id
  database_runtime_user        = module.my_database.database_runtime_user
  database_security_group_id   = module.my_database.security_group_id
  database_port                = module.my_database.cluster_port

  environment_variables = {
    RUNTIME_CONFIG_APP_ID = module.my_database.appconfig_application_id
  }
}
```

Déployez les fonctions Lambda de l'API dans des **sous-réseaux privés avec sortie**, et non dans des sous-réseaux privés isolés. Assurez-vous que le rôle Lambda de l'API dispose de la permission `rds-db:connect` et que son groupe de sécurité peut atteindre le groupe de sécurité de la base de données sur le port de la base de données.

## Développement Local

Le générateur applique la même injection du client Prisma dans le gestionnaire de requêtes dans `local-server.ts` :

```ts title="packages/api/src/local-server.ts" {1,5,10}
import { getPrisma as getMyDb } from ':my-scope/my-db';

const server = createServer(async function (req, res) {
  const httpRequest = convertRequest(req);
  const myDb = await getMyDb();
  const httpResponse = await serviceHandler.handle(httpRequest, {
    tracer,
    logger,
    metrics,
    myDb,
  });
  return writeResponse(httpResponse, res);
});
```

<NxCommands commands={["serve-local <api-project-name>"]} />

Cela démarre à la fois l'API et la base de données locale. La variable d'environnement `SERVE_LOCAL=true` est définie automatiquement, de sorte que le client Prisma se connecte à la base de données Docker locale au lieu d'Aurora.
