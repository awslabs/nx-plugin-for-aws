Le générateur `connection` connecte un <Link path="guides/py-agent">Agent Python</Link> à un projet <Link path="guides/py-rdb">Base de Données Relationnelle Python</Link>, rendant la session de base de données disponible dans vos outils d'agent.

## Prérequis

Avant d'utiliser ce générateur, assurez-vous d'avoir :

1. Un projet <Link path="guides/py-agent">`py#agent`</Link>
2. Un projet <Link path="guides/py-rdb">`py#rdb`</Link>

## Sortie du Générateur

Le générateur ajoute le package de base de données comme dépendance de l'espace de travail, met à jour la cible `<agent-name>-dev` de l'agent pour dépendre de la cible `dev` de la base de données afin que les deux démarrent ensemble pendant le développement local, et met à jour le Dockerfile de l'agent généré avec le bundle CA RDS utilisé pour les connexions directes à Aurora.

## Comment ça Fonctionne

Le package de base de données exporte `session_context`, un gestionnaire de contexte qui ouvre une `Session` SQLModel en utilisant le client de base de données généré et la ferme lorsque le bloc `with` se termine.

Pour le développement local, `session_context()` se connecte au conteneur de base de données local. Dans les environnements d'exécution déployés, il lit la configuration de la base de données depuis AWS AppConfig, génère un jeton d'authentification IAM et valide le certificat TLS de la base de données.

## Utiliser la Base de Données dans les Outils d'Agent

Importez `session_context` depuis votre package de base de données et utilisez-le dans vos outils d'agent :

```python title="packages/my_service/my_service/my_agent/agent.py"
from sqlmodel import select
from my_scope.my_db import session_context
from my_scope.my_db.models.example import ExampleModel
from strands import tool

@tool
def list_examples() -> list:
    """List all example records."""
    with session_context() as session:
        return [item.model_dump() for item in session.exec(select(ExampleModel)).all()]
```

## Infrastructure

Le construct d'agent généré implémente `IGrantable` et `IConnectable`, vous pouvez donc accorder l'accès réseau et IAM à la base de données directement sur le construct.

<Infrastructure>
<Fragment slot="cdk">

```ts title="packages/infra/src/stacks/application-stack.ts"
import { MyDatabase } from ':my-scope/common-constructs';

const db = new MyDatabase(this, 'Db', { vpc, ... });
const myAgent = new MyAgent(this, 'MyAgent', { vpc, ... });

db.allowDefaultPortFrom(myAgent);
db.grantConnect(myAgent);
```

`allowDefaultPortFrom` ouvre la règle du groupe de sécurité afin que l'environnement d'exécution de l'agent puisse atteindre le port de la base de données. `grantConnect` accorde la permission IAM `rds-db:connect` au rôle d'exécution de l'agent.

</Fragment>
<Fragment slot="terraform">

Passez les sorties du module de base de données dans votre module d'agent afin qu'il puisse atteindre la base de données et lire sa configuration d'exécution :

```hcl title="packages/infra/src/main.tf"
module "my_database" {
  source              = "../../common/terraform/src/app/dbs/my-database"
  vpc_id              = module.vpc.vpc_id
  database_subnet_ids = module.vpc.private_isolated_subnet_ids
}

module "my_agent" {
  source = "../../common/terraform/src/app/agents/my-agent"

  appconfig_application_id     = module.my_database.appconfig_application_id
  database_cluster_resource_id = module.my_database.cluster_resource_id
  database_runtime_user        = module.my_database.database_runtime_user
  database_security_group_id   = module.my_database.security_group_id
  database_port                = module.my_database.cluster_port
}
```

Assurez-vous que le rôle d'exécution de l'agent a la permission `rds-db:connect` et que son groupe de sécurité peut atteindre le groupe de sécurité de la base de données sur le port de la base de données.

### Exigences SSL lors de la Connexion sans RDS Proxy

Lorsque l'agent se connecte directement au cluster Aurora (sans RDS Proxy), le générateur de connexion met à jour le Dockerfile de l'agent généré pour ajouter le bundle CA Amazon RDS à `/opt/global-bundle.pem` :

```dockerfile
ADD --chmod=0644 https://truststore.pki.rds.amazonaws.com/global/global-bundle.pem /opt/global-bundle.pem
```

Passez ce chemin à `session_context` en utilisant le paramètre optionnel `rds_ca` :

```python title="packages/my_service/my_service/my_agent/agent.py"
from sqlmodel import select
from my_scope.my_db import session_context
from my_scope.my_db.models.example import ExampleModel
from strands import tool

@tool
def list_examples() -> list:
    """List all example records."""
    with session_context(rds_ca='/opt/global-bundle.pem') as session:
        return [item.model_dump() for item in session.exec(select(ExampleModel)).all()]
```

Lors de l'utilisation de RDS Proxy, vous n'avez pas besoin de configurer le bundle CA RDS dans l'environnement d'exécution de l'agent.

## Développement Local

<NxCommands commands={["<agent-name>-dev <project-name>"]} />

Cela démarre l'agent et toutes les bases de données connectées. La variable d'environnement `LOCAL_DEV=true` fait en sorte que le client de base de données se connecte à sa base de données Docker locale au lieu d'Aurora.
