Le générateur `connection` relie un <Link path="guides/py-mcp-server">serveur MCP Python</Link> à un projet de <Link path="guides/py-rdb">base de données relationnelle Python</Link>, rendant la session de base de données disponible pour tous les outils enregistrés dans le serveur.

## Prérequis

Avant d'utiliser ce générateur, assurez-vous d'avoir :

1. Un projet <Link path="guides/py-mcp-server">`py#mcp-server`</Link>
2. Un projet <Link path="guides/py-rdb">`py#rdb`</Link>

## Sortie du générateur

Le générateur ajoute le package de base de données comme dépendance de l'espace de travail, met à jour la cible `<mcp-server-name>-dev` du serveur MCP pour qu'elle dépende de la cible `dev` de la base de données afin que les deux démarrent ensemble pendant le développement local, et met à jour le Dockerfile du serveur MCP généré avec le bundle CA RDS utilisé pour les connexions directes à Aurora.

## Comment ça fonctionne

Le package de base de données exporte `session_context`, un gestionnaire de contexte qui ouvre une `Session` SQLModel en utilisant le client de base de données généré et la ferme lorsque le bloc `with` se termine.

Pour le développement local, `session_context()` se connecte au conteneur de base de données local. Dans les environnements d'exécution déployés, il lit la configuration de la base de données depuis AWS AppConfig, génère un jeton d'authentification IAM et valide le certificat TLS de la base de données.

## Utiliser la base de données dans les outils MCP

Importez `session_context` depuis votre package de base de données et utilisez-le dans vos outils de serveur MCP :

```python title="packages/my_service/my_service/my_mcp_server/server.py"
from sqlmodel import select
from my_scope.my_db import session_context
from my_scope.my_db.models.example import ExampleModel

@mcp.tool()
def list_examples() -> str:
    """List all example records."""
    with session_context() as session:
        items = session.exec(select(ExampleModel)).all()
        return str([item.model_dump() for item in items])
```

## Bases de données multiples

Exécuter à nouveau le générateur avec une cible différente ajoute la deuxième base de données à côté de la première. Les deux contextes de session sont disponibles pour tous les outils :

```python title="packages/my_service/my_service/my_mcp_server/server.py"
from my_scope.my_db import session_context as my_db_session_context
from my_scope.other_db import session_context as other_db_session_context
```

## Infrastructure

Le construct de serveur MCP généré implémente `IGrantable` et `IConnectable`, vous pouvez donc accorder l'accès réseau et IAM à la base de données directement sur le construct.

<Infrastructure>
<Fragment slot="cdk">

```ts title="packages/infra/src/stacks/application-stack.ts"
import { MyDatabase } from ':my-scope/common-constructs';

const db = new MyDatabase(this, 'Db', { vpc, ... });
const myMcpServer = new MyMcpServer(this, 'MyMcpServer', { vpc, ... });

db.allowDefaultPortFrom(myMcpServer);
db.grantConnect(myMcpServer);
```

`allowDefaultPortFrom` ouvre la règle du groupe de sécurité afin que le runtime du serveur MCP puisse atteindre le port de la base de données. `grantConnect` accorde la permission IAM `rds-db:connect` au rôle d'exécution du serveur.

</Fragment>
<Fragment slot="terraform">

Transmettez les sorties du module de base de données dans votre module de serveur MCP afin qu'il puisse atteindre la base de données et lire sa configuration d'exécution :

```hcl title="packages/infra/src/main.tf"
module "my_database" {
  source              = "../../common/terraform/src/app/dbs/my-database"
  vpc_id              = module.vpc.vpc_id
  database_subnet_ids = module.vpc.private_isolated_subnet_ids
}

module "my_mcp_server" {
  source = "../../common/terraform/src/app/mcp-servers/my-mcp-server"

  appconfig_application_id     = module.my_database.appconfig_application_id
  database_cluster_resource_id = module.my_database.cluster_resource_id
  database_runtime_user        = module.my_database.database_runtime_user
  database_security_group_id   = module.my_database.security_group_id
  database_port                = module.my_database.cluster_port
}
```

Assurez-vous que le rôle d'exécution du serveur MCP dispose de la permission `rds-db:connect` et que son groupe de sécurité peut atteindre le groupe de sécurité de la base de données sur le port de la base de données.

### Exigences SSL lors de la connexion sans RDS Proxy

Lorsque le serveur MCP se connecte directement au cluster Aurora (sans RDS Proxy), le générateur de connexion met à jour le Dockerfile du serveur MCP généré pour ajouter le bundle CA Amazon RDS à `/opt/global-bundle.pem` :

```dockerfile
ADD --chmod=0644 https://truststore.pki.rds.amazonaws.com/global/global-bundle.pem /opt/global-bundle.pem
```

Transmettez ce chemin à `session_context` en utilisant le paramètre optionnel `rds_ca` :

```python title="packages/my_service/my_service/my_mcp_server/server.py"
from sqlmodel import select
from my_scope.my_db import session_context
from my_scope.my_db.models.example import ExampleModel

@mcp.tool()
def list_examples() -> str:
    """List all example records."""
    with session_context(rds_ca='/opt/global-bundle.pem') as session:
        items = session.exec(select(ExampleModel)).all()
        return str([item.model_dump() for item in items])
```

Lors de l'utilisation de RDS Proxy, vous n'avez pas besoin de configurer le bundle CA RDS dans le runtime du serveur MCP.

## Développement local

<NxCommands commands={["<mcp-server-name>-dev <project-name>"]} />

Cela démarre le serveur MCP et toutes les bases de données connectées. La variable d'environnement `LOCAL_DEV=true` fait en sorte que le client de base de données se connecte à sa base de données Docker locale au lieu d'Aurora.
