Le générateur DCR Proxy crée un proxy OAuth [Dynamic Client Registration (DCR)](https://datatracker.ietf.org/doc/html/rfc7591) devant un Amazon Cognito User Pool.

Les clients MCP (tels que Claude Code, Kiro CLI ou le MCP Inspector) s'attendent à s'authentifier auprès d'un serveur d'autorisation OAuth qui prend en charge Dynamic Client Registration et la découverte de métadonnées. Amazon Cognito ne prend pas en charge DCR nativement, et son secret App Client ne doit jamais être exposé à un client public. Ce proxy comble cette lacune : il maintient le flux Cognito Hosted UI intact, implémente DCR, injecte le secret App Client côté serveur lors de l'échange de jeton, et transmet le trafic MCP à votre serveur MCP en amont.

L'infrastructure provisionne une API Gateway HTTP API avec six fonctions Lambda câblées aux routes suivantes :

| Route | Description |
| --- | --- |
| `GET /.well-known/oauth-protected-resource` | Métadonnées de ressource protégée |
| `GET /.well-known/oauth-authorization-server` | Métadonnées du serveur d'autorisation |
| `GET /.well-known/openid-configuration` | Configuration OpenID (servie par le gestionnaire de métadonnées du serveur d'autorisation) |
| `POST /register` | Dynamic Client Registration |
| `GET /authorize` | Autorisation (redirige vers le Cognito Hosted UI) |
| `POST /oauth/token` | Échange de jeton (injecte le secret App Client) |
| `ANY /mcp` | Proxy vers le serveur MCP en amont |

Seul le gestionnaire de jeton dispose d'un accès en lecture au secret Cognito App Client dans Secrets Manager.

## Déploiement du proxy DCR

Le proxy ne crée pas vos ressources Cognito ni votre serveur MCP. Au lieu de cela, vous injectez les identifiants de ressources gérées ailleurs (qu'elles soient générées par ce plugin ou provisionnées séparément), gardant le proxy découplé de la manière dont ces ressources sont provisionnées.

Vous fournissez :

- L'id du Cognito User Pool et l'id de l'App Client
- L'ARN d'un secret Secrets Manager contenant le secret App Client. Le gestionnaire de jeton le lit à l'exécution ; la valeur n'est jamais exposée au client.
- L'URL de base du domaine Cognito Hosted UI
- L'URL complète de votre serveur MCP en amont

<Infrastructure>
<Fragment slot="cdk">

import { DcrProxy } from ':my-scope/common-constructs';

:::note
Le proxy `/mcp` utilise les intégrations API Gateway HTTP API, qui ont un délai d'expiration fixe de 29 secondes. Les requêtes en amont de longue durée sont mises en mémoire tampon par le proxy plutôt que diffusées en continu.
:::

### Fronter une AgentCore Gateway

Un amont courant est une [AgentCore Gateway](https://docs.aws.amazon.com/bedrock-agentcore/latest/devguide/gateway.html) authentifiée avec Cognito. Créez un seul User Pool et App Client, autorisez cet App Client sur la gateway, puis passez les **mêmes** identifiants au proxy DCR. L'`upstreamUrl` du proxy est l'URL de la gateway.

La gateway ci-dessous est un construct `aws-cdk-lib/aws-bedrockagentcore` simple configuré pour l'authentification entrante Cognito — et non la gateway authentifiée IAM émise par le générateur `agentcore-gateway`, qui n'est pas compatible avec le flux OAuth du proxy.

<Infrastructure>
<Fragment slot="cdk">
```typescript
import { DcrProxy } from ':my-scope/common-constructs';
import { UserPool } from 'aws-cdk-lib/aws-cognito';
import * as secretsmanager from 'aws-cdk-lib/aws-secretsmanager';
import {
  Gateway,
  GatewayAuthorizer,
  McpProtocolConfiguration,
  MCPProtocolVersion,
} from 'aws-cdk-lib/aws-bedrockagentcore';

// User Pool et App Client uniques partagés par la gateway et le proxy.
// Enregistrez les URL de rappel que vos clients utilisent (voir ci-dessous).
const userPool = new UserPool(this, 'UserPool');
const userPoolClient = userPool.addClient('DcrProxyClient', {
  generateSecret: true,
  oAuth: {
    flows: { authorizationCodeGrant: true },
    callbackUrls: [
      'http://localhost:41100/callback',
      // Callback utilisé par Claude Desktop
      'https://claude.ai/api/mcp/auth_callback',
    ],
  },
});
// Cognito Hosted UI nécessite un domaine — le proxy redirige les flux OAuth ici.
// Réutilisez votre domaine existant si le User Pool en a déjà un.
const userPoolDomain = userPool.addDomain('Domain', {
  cognitoDomain: { domainPrefix: 'my-dcr-proxy' },
});

// Stockez le secret App Client dans Secrets Manager pour que le gestionnaire de jeton le lise
const clientSecret = new secretsmanager.Secret(this, 'ClientSecret', {
  secretStringValue: userPoolClient.userPoolClientSecret,
});

// Une AgentCore Gateway qui autorise les JWT entrants émis pour l'App Client
const gateway = new Gateway(this, 'Gateway', {
  protocolConfiguration: new McpProtocolConfiguration({
    supportedVersions: [MCPProtocolVersion.MCP_2025_06_18],
  }),
  authorizerConfiguration: GatewayAuthorizer.usingCognito({
    userPool,
    allowedClients: [userPoolClient],
  }),
});

new DcrProxy(this, 'DcrProxy', {
  userPoolId: userPool.userPoolId,
  userPoolClientId: userPoolClient.userPoolClientId,
  cognitoClientSecretArn: clientSecret.secretArn,
  cognitoHostedUiBase: userPoolDomain.baseUrl(),
  upstreamUrl: gateway.gatewayUrl ?? '',
});
```
</Fragment>

### Autoriser les URI de redirection client

Parce que le proxy implémente Dynamic Client Registration virtuellement — il n'y a pas d'App Client Cognito par client — chaque client MCP s'authentifie via le **seul App Client** que vous passez au proxy. Pendant le flux OAuth, le proxy transmet le `redirect_uri` du client au Cognito Hosted UI sans modification, donc Cognito effectue la vérification faisant autorité : le callback doit être enregistré en tant qu'**URL de rappel sur cet App Client**, sinon Cognito rejette la connexion.

C'est un effet secondaire de la conception DCR virtuelle. Un client peut enregistrer n'importe quel `redirect_uri` avec le proxy, mais la connexion ne réussit que si cette URL exacte est l'une des URL de rappel de l'App Client. Cognito fait correspondre les URL de rappel exactement, y compris le port, donc les clients qui écoutent sur un port éphémère aléatoire ne peuvent pas être couverts par un caractère générique — vous devez épingler chaque client à une URL de rappel fixe et enregistrer cette URL exacte sur l'App Client.

Ajoutez les URL de rappel que vos clients utilisent lorsque vous créez l'App Client :

<Infrastructure>

```typescript

  generateSecret: true,
  oAuth: {
    flows: { authorizationCodeGrant: true },

      // Clients locaux : épinglez à un port fixe et peu commun plutôt qu'à un port par défaut
      'http://localhost:41100/callback',
      // Callback utilisé par Claude Desktop
      'https://claude.ai/api/mcp/auth_callback',
    ],
  },
});
```

## Consommer un serveur MCP proxié

Le proxy permet aux clients MCP de s'authentifier auprès de votre Cognito User Pool sans aucune configuration spécifique au client au-delà de l'URL du proxy. Lorsqu'un client se connecte au point de terminaison `/mcp`, il découvre les métadonnées OAuth (via `/.well-known/oauth-protected-resource` et `/.well-known/oauth-authorization-server`), s'enregistre dynamiquement et guide l'utilisateur à travers la connexion Cognito Hosted UI. Le proxy injecte le secret App Client lors de l'échange de jeton, donc le client n'en a jamais besoin.

Pour connecter un client, pointez-le vers le `mcpUrl` du proxy (c'est-à-dire `<proxyUrl>/mcp`). Les exemples ci-dessous supposent que votre proxy est déployé sur `https://my-proxy.example.com`.

:::note
Chaque client utilise une URL de rappel pendant le flux OAuth. Cette URL exacte — y compris son port — doit être enregistrée en tant qu'[URL de rappel sur votre Cognito App Client](#autoriser-les-uri-de-redirection-client), sinon la connexion échoue. Lorsqu'un client choisirait autrement un port aléatoire, épinglez-le à une valeur fixe et enregistrez cette valeur.
:::

### Claude Code

Ajoutez le serveur proxié avec la commande [`claude mcp add`](https://code.claude.com/docs/en/mcp), en utilisant le transport HTTP. Par défaut, Claude Code écoute sur un port de rappel aléatoire ; passez `--callback-port` pour l'épingler au port enregistré sur votre App Client (`41100` dans les exemples ci-dessus). Claude Code utilise toujours le chemin `/callback`, donc l'URI de redirection résultant est `http://localhost:41100/callback` :

```bash
claude mcp add --transport http --callback-port 41100 my-proxied-server https://my-proxy.example.com/mcp
```

Lorsque vous invoquez pour la première fois un outil du serveur, Claude Code ouvre le Cognito Hosted UI pour s'authentifier avant que la requête ne soit proxiée en amont.

### Kiro CLI

Ajoutez le serveur à votre configuration MCP [Kiro CLI](https://kiro.dev/docs/cli/mcp/), en utilisant le transport HTTP. Sans `oauth.redirectUri` explicite, Kiro choisit un port de rappel aléatoire ; définissez-le sur l'URL enregistrée sur votre App Client afin que le port et le chemin correspondent exactement :

```json
{
  "mcpServers": {
    "my-proxied-server": {
      "type": "http",
      "url": "https://my-proxy.example.com/mcp",
      "oauth": {
        "redirectUri": "http://localhost:41100/callback"
      }
    }
  }
}
```

Kiro CLI déclenche la connexion Cognito Hosted UI lors de la première utilisation et gère les jetons résultants pour les requêtes ultérieures.

:::tip
Tout client MCP qui prend en charge le transport HTTP Streamable ainsi que l'autorisation OAuth et Dynamic Client Registration peut se connecter au proxy de la même manière — fournissez l'URL `/mcp` et le client gère la découverte, l'enregistrement et la connexion pour vous.
:::
