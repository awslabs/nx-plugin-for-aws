Les nouveaux espaces de travail créés avec `@aws/nx-plugin` incluent des hooks pre-commit [git-secrets](https://github.com/awslabs/git-secrets) configurés par défaut. Cela analyse les fichiers en attente de validation pour détecter les motifs correspondant aux identifiants AWS avant chaque commit, empêchant ainsi les fuites accidentelles d'identifiants.

## Ce qui est inclus

Lorsque vous créez un nouvel espace de travail, les éléments suivants sont configurés automatiquement :

<FileTree>
  - .git-secrets/
    - git-secrets Le script bash git-secrets (vendorisé depuis awslabs/git-secrets)
    - setup.sh Enregistre les motifs d'identifiants AWS dans la configuration git locale
  - .husky/
    - pre-commit Exécute l'analyse git-secrets avant chaque commit
  - package.json Inclut husky et le script prepare pour la configuration automatique
</FileTree>

## Comment ça fonctionne

1. Le script `prepare` s'exécute après `npm install` / `pnpm install`, ce qui initialise les hooks git husky et enregistre les motifs d'identifiants AWS dans votre configuration git locale.
2. À chaque `git commit`, le hook pre-commit husky invoque `git-secrets` pour analyser les fichiers en attente de validation.
3. Si un motif correspondant à un identifiant AWS est détecté, le commit est bloqué avec un message d'erreur.

## Motifs détectés

Les motifs suivants sont analysés :

- **AWS Access Key IDs** — `AKIA...`, `ASIA...`, et autres préfixes de clés IAM suivis de 16 caractères alphanumériques en majuscules
- **AWS Secret Access Keys** — Chaînes base64 de 40 caractères précédées d'une étiquette `SECRET_ACCESS_KEY`
- **AWS Account IDs** — Nombres à 12 chiffres précédés d'une étiquette `ACCOUNT_ID`
- **Bedrock API Keys** — Clés Bedrock à longue durée de vie (`ABSK...`) et à courte durée de vie
- **Valeurs du fichier credentials** — Valeurs de clés littérales trouvées dans `~/.aws/credentials`

Les clés d'exemple bien connues (par exemple, `AKIAIOSFODNN7EXAMPLE`) sont automatiquement mises sur liste blanche.

## Autoriser les faux positifs

Si git-secrets bloque un commit contenant un faux positif :

```sh
# Allow a specific pattern
git config --add secrets.allowed 'my-pattern-here'
```

Ou ajoutez des motifs à un fichier `.gitallowed` à la racine du dépôt (une expression régulière par ligne).

Pour un contournement ponctuel :

```sh
git commit --no-verify
```

## Désactivation

Pour créer un espace de travail sans git-secrets :

```sh
pnpm create @aws/nx-workspace my-project --no-gitSecrets
```

Ou passez `--gitSecrets=false` au générateur de preset.

## Exigences

- **Git** doit être initialisé dans l'espace de travail (`git init`)
- **Bash** doit être disponible dans le PATH (fourni avec Git pour Windows sous Windows)
- Les motifs sont enregistrés lors de l'exécution de `prepare` — si vous clonez une nouvelle copie, exécutez la commande d'installation de votre gestionnaire de paquets pour réenregistrer
