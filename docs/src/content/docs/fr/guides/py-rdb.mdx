Ce générateur crée un nouveau projet de base de données relationnelle Python soutenu par [Amazon Aurora](https://aws.amazon.com/rds/aurora/) (PostgreSQL ou MySQL), [SQLModel](https://sqlmodel.tiangolo.com/) pour la modélisation des données, et [Alembic](https://alembic.sqlalchemy.org/) pour les migrations de schéma. Il génère le code d'application et l'infrastructure nécessaires pour provisionner et gérer une base de données en utilisant AWS CDK ou Terraform, avec une définition de schéma déclarative, un déploiement automatique des migrations et un client de base de données.

## Développement local

### Modélisation des données

Le projet généré utilise [SQLModel](https://sqlmodel.tiangolo.com/) pour définir votre schéma de base de données. Le flux de travail est axé sur le modèle : ajoutez ou mettez à jour les classes de table SQLModel dans le répertoire `<name>/models/` de votre projet de base de données, puis générez une migration à partir de ces modifications de modèle.

Exemple de modèle :

```python title="packages/my_db/my_db/models/example.py"
from sqlalchemy import Column, String
from sqlmodel import Field, SQLModel


class ExampleModel(SQLModel, table=True):
    id: int | None = Field(default=None, primary_key=True)
    name: str = Field(sa_column=Column(String(255), nullable=False))
    description: str | None = Field(default=None, sa_column=Column(String(255), nullable=True))
```

Importez vos modèles dans `<name>/models/__init__.py` afin qu'Alembic puisse les découvrir lors de l'autogénération.

### Création de migrations

Après avoir ajouté ou mis à jour des modèles, utilisez Alembic pour générer et appliquer des scripts de migration. La cible `alembic` générée démarre automatiquement un conteneur de base de données local avant l'exécution :

<NxCommands commands={['run <project>:alembic revision --autogenerate -m "describe your change"']} />

Cela génère un nouveau script de migration dans `migrations/versions/`. Examinez le script généré avant de l'appliquer.

Appliquez la migration à votre base de données locale :

<NxCommands commands={['run <project>:migrate']} />

Lorsque vous déployez la pile AWS, l'infrastructure générée applique automatiquement les migrations générées à la base de données déployée.

### Application de migrations existantes

Lorsque vous récupérez des fichiers de migration créés par d'autres développeurs, appliquez-les à votre base de données locale :

<NxCommands commands={['run <project>:migrate']} />

### Exécution de commandes Alembic

La cible `alembic` générée expose l'interface CLI d'Alembic, vous pouvez donc exécuter n'importe quelle commande Alembic sur la base de données locale. Consultez la [référence des commandes Alembic](https://alembic.sqlalchemy.org/en/latest/ops.html) pour les commandes disponibles.

<NxCommands commands={['run <project>:alembic <alembic-command>']} />

### Arrêt de la base de données locale

L'arrêt de `dev` (par exemple avec `Ctrl+C`) supprime automatiquement le conteneur de base de données local, mais préserve le volume nommé afin que vos données persistent entre les redémarrages.

:::caution[Windows]
En raison de limitations dans la gestion des signaux sous Windows, le conteneur n'est pas automatiquement supprimé lorsque `dev` est arrêté. Vous devrez le supprimer manuellement :

```bash
<engine> rm -f <scope>-<db-name>
```

Remplacez `<engine>` par votre moteur de conteneur (`docker` ou `finch`), `<scope>` par la portée de votre espace de travail Nx (par exemple `proj`), et `<db-name>` par le nom de votre projet de base de données (par exemple `my-db`).
:::

## Connexion à la base de données

Importez `session_context` depuis votre package de base de données et utilisez-le comme gestionnaire de contexte asynchrone pour obtenir une `AsyncSession` :

```python
from sqlmodel import select

from my_scope.my_db import session_context
from my_scope.my_db.models.example import ExampleModel

async def example():
    async with session_context() as session:
        results = (await session.execute(select(ExampleModel))).all()
```

Le client de base de données automatiquement :
- Récupère la configuration de la base de données depuis AWS AppConfig en utilisant la variable d'environnement `RUNTIME_CONFIG_APP_ID`
- Génère des jetons d'authentification temporaires via le signataire RDS `boto3` pour l'authentification IAM
- Établit des connexions TLS en utilisant `ssl.create_default_context()`

#### Exigences SSL lors de la connexion sans RDS Proxy

Le bundle CA Amazon RDS doit être dans le magasin de confiance système de l'environnement d'exécution.

##### Environnements d'exécution de conteneur

<Tabs>
<TabItem label="Amazon Linux">

```dockerfile
ADD https://truststore.pki.rds.amazonaws.com/global/global-bundle.pem /etc/pki/ca-trust/source/anchors/global-bundle.pem
RUN update-ca-trust
```

</TabItem>
<TabItem label="Debian">

```dockerfile
ADD https://truststore.pki.rds.amazonaws.com/global/global-bundle.pem /usr/local/share/ca-certificates/rds-global-bundle.crt
RUN update-ca-certificates
```

</TabItem>
</Tabs>

##### Fonctions Lambda Zip

Pour les fonctions Lambda déployées en zip (telles qu'une <Link path="guides/fastapi">`py#api` FastAPI</Link>), le magasin de confiance CA intégré de l'environnement d'exécution Lambda Amazon Linux 2023 inclut les CA racine Amazon utilisées par RDS.

Lors de l'utilisation de RDS Proxy, vous n'avez pas besoin de configurer le bundle CA RDS dans l'environnement d'exécution qui se connecte à la base de données.
