Ce générateur crée un nouveau projet de base de données relationnelle basé sur [Amazon Aurora](https://aws.amazon.com/rds/aurora/) PostgreSQL. Il génère le code d'application et l'infrastructure nécessaires pour provisionner et gérer une base de données en utilisant AWS CDK ou Terraform.

L'infrastructure générée comprend :

- Construction ou module Aurora réutilisable
- Construction ou module de base de données spécifique à l'application
- Gestionnaire d'utilisateur de base de données d'application authentifié par IAM
- Enregistrement de la configuration d'exécution dans AWS AppConfig
- Gestionnaire de migration pour le déploiement automatique des migrations Prisma

## Implémenter votre base de données

### Modélisation des données

Le projet généré utilise [Prisma ORM](https://www.prisma.io/docs/orm) pour définir votre schéma de base de données et générer un client type-safe. Modélisez vos tables, relations et types de champs dans `prisma/schema.prisma`.

Exemple de modèle `User` :

```ts title="packages/postgres/prisma/schema.prisma"
model User {
  id          Int      @id @default(autoincrement()) 
  firstName   String
  lastName    String
}
```

Pour plus de détails, consultez le guide officiel sur la [modélisation des données avec Prisma](https://www.prisma.io/docs/orm/core-concepts/data-modeling#data-modeling-with-prisma-orm).

### Générer le client de base de données

Le générateur configure automatiquement la cible `generate` pour créer un client Prisma TypeScript type-safe chaque fois que vous construisez le projet. Le client est écrit dans `generated/prisma` (ajouté à `.gitignore`).

Vous pouvez également générer manuellement le client à tout moment :

<Tabs>


  </TabItem>
  <TabItem label="pnpm" icon="seti:pnpm">

```bash

```

  </TabItem>
  <TabItem label="yarn" icon="seti:yarn">

```bash

```

  </TabItem>
  <TabItem label="bun" icon="seti:bun">

```bash

```

  </TabItem>

Le wrapper d'exécution dans `src/prisma.ts` exporte :

- `DB_PACKAGE_NAME` - la clé utilisée sous l'espace de noms de configuration d'exécution `database` dans AWS AppConfig
- `getPrisma()` - charge les paramètres de connexion à la base de données depuis AWS AppConfig et crée un client Prisma utilisant l'authentification IAM

Le client automatiquement :
- Récupère la configuration de la base de données depuis AWS AppConfig en utilisant la variable d'environnement `RUNTIME_CONFIG_APP_ID`
- Génère des jetons d'authentification temporaires via AWS RDS Signer pour l'authentification IAM
- Gère les connexions SSL/TLS avec validation de certificat
- Gère le pooling de connexions via des pools de connexions de base de données persistants

Exemple d'utilisation dans une API tRPC :

```ts title="packages/api/src/procedures/list-user.ts"
import { getPrisma } from ':aws-nx-poc/postgres';
import { publicProcedure } from '../init.js';
import { ListUserOutputSchema } from '../schema/index.js';

export const listUser = publicProcedure
  .output(ListUserOutputSchema)
  .query(async () => {
    const prisma = await getPrisma();
    const users = await prisma.user.findMany({
      orderBy: { id: 'asc' },
    });

    return { users };
  });
```

### Créer des migrations

:::note[Exécuter Prisma depuis le répertoire du package]
Lors de l'utilisation de la CLI Prisma, exécutez les commandes depuis le répertoire de votre package de base de données généré (par exemple, `packages/postgres`) afin que Prisma puisse trouver `prisma/schema.prisma` et `prisma.config.ts`.
:::

Prisma Migrate nécessite une [base de données fantôme](https://www.prisma.io/docs/orm/prisma-migrate/understanding-prisma-migrate/shadow-database) pour comparer votre schéma avec l'état actuel de la base de données et générer les modifications SQL.

Pour générer des migrations localement, pointez Prisma vers une base de données de développement en exportant `DATABASE_URL` :

```bash
export DATABASE_URL="postgresql://username:password@hostname:5432/dbname"
```

Ensuite, créez les fichiers de migration sans les appliquer :


  </TabItem>

## Déployer sur AWS

Le générateur de base de données relationnelle crée une infrastructure CDK ou Terraform en fonction de votre `iacProvider` sélectionné.

<Infrastructure>
<Fragment slot="cdk">
La construction CDK est créée dans `common/constructs`. Exemple d'utilisation :

```ts title="packages/infra/src/stacks/application-stack.ts"
import { Postgres } from ':my-scope/common-constructs';

export class ApplicationStack extends Stack {
  constructor(scope: Construct, id: string, props?: StackProps) {
    super(scope, id, props);

    const vpc = new Vpc(this, 'Vpc', {
      subnetConfiguration: [
        {
          name: 'public',
          subnetType: SubnetType.PUBLIC,
        },
        {
          name: 'private_with_egress',
          subnetType: SubnetType.PRIVATE_WITH_EGRESS,
        },
        {
          name: 'private_isolated',
          subnetType: SubnetType.PRIVATE_ISOLATED,
        },
      ],
    });

    const db = new Postgres(this, 'Db', {
      vpc,
      vpcSubnets: {
        subnetType: SubnetType.PRIVATE_ISOLATED,
      }
    });
  }
}
```

Cela provisionne un cluster Aurora avec RDS Proxy, identifiants administrateur, utilisateur de base de données d'application, enregistrement de configuration d'exécution et gestionnaire de migration. Déployez dans des sous-réseaux privés isolés pour empêcher l'accès direct depuis Internet.

L'infrastructure générée crée deux utilisateurs de base de données :
- **Utilisateur administrateur** - Créé lors du provisionnement du cluster avec les identifiants stockés dans AWS Secrets Manager
- **Utilisateur d'application** - Créé via une ressource personnalisée Lambda avec l'authentification IAM activée et tous les privilèges sur la base de données d'application
</Fragment>

Cela provisionne un cluster Aurora avec RDS Proxy, identifiants administrateur, Lambda create-db-user, enregistrement de configuration d'exécution, Lambda de migration et ressources de registre de conteneurs.

### Connecter les fonctions Lambda

Pour utiliser le client Prisma depuis les fonctions Lambda, assurez-vous que les exigences suivantes sont remplies :

1. Déployez les fonctions dans le même VPC que la base de données
2. Autorisez l'accès réseau au port de la base de données
3. Accordez les permissions IAM `rds-db:connect` pour l'utilisateur d'exécution généré
4. Définissez les variables d'environnement requises :
   - `NODE_EXTRA_CA_CERTS=/var/runtime/ca-cert.pem` (requis pour la validation TLS)
   - `RUNTIME_CONFIG_APP_ID` (défini automatiquement par l'infrastructure générée)
   - `AWS_REGION` (défini automatiquement dans l'environnement d'exécution Lambda)

:::note[CLI Prisma vs Client d'exécution]
Les commandes **Prisma CLI** (`prisma migrate dev`, `prisma migrate deploy`) lisent `DATABASE_URL` depuis les variables d'environnement.

Le **client d'exécution** (`src/prisma.ts`) charge les paramètres de connexion depuis AWS AppConfig et utilise l'authentification IAM.
:::

<Infrastructure>
<Fragment slot="cdk">

```ts title="packages/infra/src/stacks/application-stack.ts"
import { RuntimeConfig } from ':my-scope/common-constructs';

const runtimeConfig = RuntimeConfig.ensure(this);

const api = new Api(this, 'Api', {
  integrations: Api.defaultIntegrations(this)
    .withDefaultOptions({
      vpc,
      vpcSubnets: {
        subnetType: SubnetType.PRIVATE_WITH_EGRESS,
      },
      environment: {
        NODE_EXTRA_CA_CERTS: '/var/runtime/ca-cert.pem',
      },
    })
    .build(),
});

Object.entries(api.integrations).forEach(([operation, integration]) => {
  db.allowDefaultPortFrom(
    integration.handler,
    `Allow ${operation} to connect to the database`,
  );
  db.grantConnect(integration.handler);
});
```

Cela configure chaque fonction Lambda d'API pour qu'elle s'exécute dans le même VPC, lise la configuration d'exécution depuis AppConfig et se connecte à la base de données en utilisant l'authentification IAM.
</Fragment>
<Fragment slot="terraform">
Lors de la connexion des fonctions Lambda à la base de données, utilisez ces sorties de module :

- `appconfig_application_id` - exposer à l'environnement de calcul
- `cluster_port` - autoriser l'accès réseau
- `database_runtime_user` - accorder la permission `rds-db:connect`

Assurez-vous que le calcul s'exécute dans le même VPC et les sous-réseaux privés, et accordez les permissions de lecture AppConfig au rôle de calcul.

### RDS Proxy

L'infrastructure générée inclut un RDS Proxy par défaut, qui se situe entre votre application et le cluster Aurora. RDS Proxy offre plusieurs avantages :

- **Pooling de connexions** - Maintient un pool de connexions de base de données qui peuvent être partagées entre les instances d'application, réduisant la surcharge d'établissement de nouvelles connexions
- **Résilience des connexions** - Gère automatiquement les basculements et les reconnexions lors des remplacements d'instances Aurora ou de la maintenance
- **Authentification IAM** - Prend en charge l'authentification de base de données basée sur IAM, éliminant le besoin de gérer les identifiants de base de données dans votre code d'application
- **Sécurité améliorée** - Impose le chiffrement TLS pour toutes les connexions

<Infrastructure>
<Fragment slot="cdk">

Par défaut, RDS Proxy est activé. Le client d'exécution généré (`getPrisma()`) se connecte automatiquement via le point de terminaison du proxy. Vous pouvez le désactiver si nécessaire :

```ts title="packages/infra/src/stacks/application-stack.ts"
const db = new Postgres(this, 'Db', {
  ...
  enableRdsProxy: false,
});
```

Lorsque RDS Proxy est désactivé, votre application se connecte directement au point de terminaison du cluster Aurora.
</Fragment>
<Fragment slot="terraform">

Par défaut, RDS Proxy est activé. Le client d'exécution généré (`getPrisma()`) se connecte automatiquement via le point de terminaison du proxy. Vous pouvez le désactiver si nécessaire :

```hcl title="packages/infra/src/main.tf"
module "postgres" {
  source = "../../common/terraform/src/app/dbs/postgres"
  ...
  enable_rds_proxy = false
}
```

Lorsque RDS Proxy est désactivé, votre application se connecte directement au point de terminaison du cluster Aurora. Le proxy est configuré pour exiger le chiffrement TLS et utiliser l'authentification IAM.

</Infrastructure>

## Personnaliser l'architecture de votre base de données

L'infrastructure générée peut être personnalisée pour correspondre aux exigences de votre charge de travail. Les exemples suivants démontrent quelques options de personnalisation courantes disponibles.

### Instances du cluster

Configurez les instances writer et reader pour votre cluster Aurora.

<Infrastructure>
<Fragment slot="cdk">

```ts title="packages/infra/src/stacks/application-stack.ts"
const db = new Postgres(this, 'Db', {
  ...
  writer: ClusterInstance.serverlessV2('writer'),
  readers: [ClusterInstance.serverlessV2('reader')],
});
```
</Fragment>
<Fragment slot="terraform">

```hcl title="packages/infra/src/main.tf"
module "postgres" {
  source = "../../common/terraform/src/app/dbs/postgres"
  ...
  instance_count = 2  # 1 writer + 1 reader
}
```
</Fragment>
</Infrastructure>
