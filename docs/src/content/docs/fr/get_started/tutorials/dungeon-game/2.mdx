## Tâche 1 : Implémenter l'API de jeu

Nous allons implémenter les API suivantes dans cette section :

1. `saveGame` - créer ou mettre à jour une partie.
2. `queryGames` - retourner une liste paginée des parties précédemment sauvegardées.
3. `queryInventory` - retourner une liste paginée des objets dans l'inventaire d'un joueur.
4. `queryActions` - retourner l'historique des conversations pour une partie donnée.

:::note[Où réside chaque élément d'état]
- `games` et `inventory` sont dans **DynamoDB** (via ElectroDB) — les lambdas de l'API de jeu et le serveur MCP d'inventaire les lisent/écrivent.
- `actions` (transcription du chat) est dans **S3** — le `S3SessionManager` de l'Agent d'Histoire écrit un blob JSON par tour, et `queryActions` dans l'API de jeu les relit pour que l'interface utilisateur puisse rejouer l'historique lors du rechargement de la page. Il n'y a pas de procédure `saveAction` ; l'agent est le seul rédacteur.
:::

## Tâche 4 : Tester l'API de jeu localement

Il n'est pas nécessaire de déployer sur AWS pour essayer notre API — la cible `serve-local` exécute l'API de jeu contre [DynamoDB Local](https://docs.aws.amazon.com/amazondynamodb/latest/developerguide/DynamoDBLocal.html). Comme nous avons connecté l'API de jeu au projet `DungeonDb` dans le Module 1, cette cible démarre également DynamoDB Local automatiquement.

### Démarrer le serveur local

Démarrez l'API de jeu localement avec la cible `serve-local`. Assurez-vous que votre moteur de conteneur (Docker ou Finch) est en cours d'exécution d'abord, car cela démarre également DynamoDB Local :

<NxCommands commands={["serve-local game-api"]} />

La cible `serve-local` définit `SERVE_LOCAL=true`, donc `resolveTableName()` retourne le nom de la table locale et `getDynamoDBClient()` se connecte à DynamoDB Local au lieu d'AWS — pas de déploiement et pas de `RUNTIME_CONFIG_APP_ID` requis.

<Aside type="tip" title="Première exécution">
La première exécution télécharge l'image du conteneur DynamoDB Local, ce qui peut prendre un moment. Laissez le serveur en cours d'exécution dans ce terminal et ouvrez un second terminal pour les commandes `curl` ci-dessous.
</Aside>

### Tester l'API

Une fois votre serveur démarré, interrogez la liste (vide) des parties :

```bash
curl -X GET 'http://localhost:2022/games.query?input=%7B%7D'
```

:::note[Encodage d'URL]
Le `%7B%7D` que nous passons pour tester l'API est un objet JSON vide encodé en URI (`{}`).
:::

Vous verrez une liste vide :

```json
{"result":{"data":{"items":[],"cursor":null}}}
```

Maintenant, sauvegardez une partie :

```bash
curl -X POST 'http://localhost:2022/games.save' \
  -H 'Content-Type: application/json' \
  -d '{"playerName":"Alice","genre":"zombie"}'
```

La sauvegarde retourne la partie persistée (avec l'horodatage `lastUpdated` que l'entité définit pour vous) :

```json
{"result":{"data":{"playerName":"Alice","genre":"zombie","lastUpdated":"..."}}}
```

Interrogez à nouveau pour confirmer qu'elle est persistée dans DynamoDB Local :

```bash
curl -X GET 'http://localhost:2022/games.query?input=%7B%7D'
```

Cette réponse inclut maintenant la partie sauvegardée :

```json
{"result":{"data":{"items":[{"playerName":"Alice","genre":"zombie","lastUpdated":"..."}],"cursor":null}}}
```

Vous pouvez arrêter le serveur local (`Ctrl+C`) une fois que vous avez terminé.

## Tâche 5 : Tester le serveur MCP d'inventaire localement

Nous pouvons essayer les outils du serveur MCP avec le [MCP Inspector](https://github.com/modelcontextprotocol/inspector) en utilisant la cible `mcp-server-inspect` générée. L'inspecteur se connecte au serveur via STDIO, nous exécutons donc DynamoDB Local séparément et indiquons au serveur de l'utiliser.

Dans un terminal, démarrez DynamoDB Local :

<NxCommands commands={["serve-local dungeon-db"]} />

Dans un second terminal, démarrez l'inspecteur :

<NxCommands commands={["mcp-server-inspect inventory"]} />

Cela démarre le MCP Inspector à `http://localhost:6274`. Dans l'interface utilisateur de l'inspecteur, avant de vous connecter, ajoutez une variable d'environnement `SERVE_LOCAL` avec la valeur `true` pour que les outils du serveur communiquent avec DynamoDB Local, puis cliquez sur **Connect**.

<Aside type="tip" title="Pourquoi définir SERVE_LOCAL ?">
Contrairement à `serve-local`, la cible `mcp-server-inspect` exécute le serveur STDIO simple, elle ne définit donc pas `SERVE_LOCAL` pour vous. Le définir dans l'environnement de l'inspecteur fait que `getDynamoDBClient()` et `resolveTableName()` se connectent à la table locale plutôt qu'à AWS.
</Aside>

Passez à l'onglet **Tools**, cliquez sur **List Tools**, et essayez `add-to-inventory` (par exemple `playerName: Alice`, `itemName: Rusty Sword`, `emoji: ⚔️`) suivi de `list-inventory-items` pour le voir persisté dans DynamoDB Local. Arrêtez les deux terminaux (`Ctrl+C`) lorsque vous avez terminé.

<Aside type="tip" title="Vous préférez le transport HTTP ?">
Si vous préférez piloter le serveur via HTTP (par exemple pour le scripting), la cible `mcp-server-serve-local` exécute le transport HTTP Streamable avec `SERVE_LOCAL=true` défini pour vous, et démarre DynamoDB Local automatiquement :

<NxCommands commands={["run @dungeon-adventure/inventory:mcp-server-serve-local"]} />

Ensuite, initialisez une session contre `http://localhost:8000/mcp` :
```bash
curl -X POST http://localhost:8000/mcp \
  -H 'Content-Type: application/json' \
  -H 'Accept: application/json, text/event-stream' \
  -d '{"jsonrpc":"2.0","id":1,"method":"initialize","params":{"protocolVersion":"2025-03-26","capabilities":{},"clientInfo":{"name":"curl","version":"1.0.0"}}}'
```
</Aside>

Félicitations, vous avez construit et testé votre première API tRPC et serveur MCP contre une table DynamoDB locale ! 🎉🎉🎉
