Le `@aws/nx-plugin` est un plugin [Nx](https://nx.dev/) qui fournit une boîte à outils pour simplifier le développement et le déploiement d'applications full-stack sur AWS. Il propose des modèles préconfigurés pour le code applicatif et d'infrastructure (IaC), réduisant considérablement le temps de configuration initiale. Le plugin gère la complexité de l'intégration des services AWS tout en préservant la flexibilité nécessaire aux personnalisations.

Les utilisateurs peuvent simplement sélectionner les composants souhaités parmi les [Générateurs](#generators) disponibles, fournir les options de configuration requises, et laisser le `@aws/nx-plugin` générer le code de démarrage nécessaire. Plusieurs générateurs sont inclus dans cet outillage pour créer des APIs, sites web, infrastructures cloud, ou même réaliser des opérations complexes comme l'intégration frontend/backend (y compris la mise à jour de fichiers existants via des transformations AST!) avec des clients type-safe.

## Construit sur Nx

[Nx](https://nx.dev) est un framework de build intelligent spécialisé dans la gestion de projets polyglottes complexes grâce à deux fonctionnalités clés : un système de build avancé et des générateurs de code.

Le système de build utilise un cache sophistiqué et l'analyse de graphes de dépendances pour optimiser l'exécution des tâches. Il ne reconstruit que les éléments impactés par les modifications, réduisant radicalement les temps de compilation. Les tâches s'exécutent en parallèle selon un ordre optimal déterminé automatiquement par Nx, rendant les opérations comme les tests, le linting ou le build extrêmement efficaces, surtout sur les gros projets.

Les générateurs servent d'outils de scaffolding automatisés pour créer des structures de code cohérentes. Ils peuvent produire aussi bien des composants individuels que des applications complètes, en suivant des modèles prédéfinis. Les équipes peuvent personnaliser ces générateurs pour respecter leurs standards internes.

Ces fonctionnalités font de Nx un outil idéal pour gérer des monorepos et maintenir des pratiques de développement cohérentes à grande échelle, tout en conservant une vélocité élevée.

## Générateurs

Tous les composants de ce plugin sont implémentés sous forme de [Générateurs](https://nx.dev/features/generate-code) qui automatisent les tâches répétitives et permettent un scaffolding cohérent pour les projets nouveaux ou existants.

Les générateurs peuvent être invoqués via la [CLI Nx](https://nx.dev/features/generate-code#invoke-generators) ou le [plugin IDE Nx Console](https://nx.dev/getting-started/editor-setup).

## Modifiable à volonté

Tout le code généré est votre propriété et peut être modifié librement. Cela simplifie radicalement l'expérience développeur : si un élément généré ne vous convient pas, vous avez toute latitude pour l'adapter.

## Dépendances minimales

Le `@aws/nx-plugin` minimise volontairement les dépendances globales. Les prérequits se limitent strictement aux outils nécessaires pour les générateurs utilisés.

Par exemple, les générateurs TypeScript nécessitent uniquement [Node](https://nodejs.org/en/download). Pour les projets Python, seul [UV](https://docs.astral.sh/uv/) est requis.

<Aside>
  UV a été choisi comme gestionnaire de paquets Python pour plusieurs raisons :
  - Outil unique remplaçant `pip`, `pip-tools`, `pipx`, `poetry`, `pyenv`, `twine`, `virtualenv`, etc.
  - 10 à 100x plus rapide que `pip`
  - Support natif des [workspaces/monorepos](https://docs.astral.sh/uv/concepts/projects/workspaces/)
</Aside>

## Politique de version unique

La configuration par défaut du monorepo applique une [politique de version unique](https://nx.dev/concepts/decisions/dependency-management#single-version-policy) pour les dépendances Node et Python.

Cela signifie que tous les projets du monorepo utilisent par défaut les mêmes versions de dépendances, réduisant les risques d'incompatibilités entre packages.

Pour Node, cela se traduit par un seul fichier de verrouillage (`lockfile`) à la racine et un seul `node_modules` contenant toutes les dépendances. Les nouvelles dépendances s'ajoutent dans le `package.json` racine.

Pour Python, cela implique un seul `.venv` à la racine du monorepo. Chaque projet Python possède son propre `pyproject.toml`, mais les versions des dépendances sont gérées par le workspace UV et synchronisées dans le fichier `uv.lock` racine.

## Sûreté de type

Le `@aws/nx-plugin` utilise la sûreté de type pour simplifier l'expérience développeur via l'autocomplétion IDE, tout en éliminant les erreurs runtime potentielles. Tous les composants fournis sont donc type-safe par défaut.
