Le `@aws/nx-plugin` est un plugin [Nx](https://nx.dev/) qui fournit une boîte à outils pour construire et déployer des applications full-stack sur AWS. Il vous offre une collection de [Générateurs](#generators) qui génèrent du code applicatif prêt pour la production _et_ l'infrastructure pour le déployer — type-safe, exécutable localement et prêt à être livré.

Plutôt que de partir d'une page blanche, vous (ou votre agent IA) choisissez les composants dont vous avez besoin — APIs, sites web, authentification, agents IA, infrastructure — fournissez quelques options de configuration, et le plugin génère du code de démarrage suivant les meilleures pratiques. Il connecte même les projets entre eux pour vous (y compris la mise à jour de fichiers existants via des transformations AST) pour produire des clients type-safe entre votre frontend et votre backend.

## Construit sur Nx et ses générateurs

[Nx](https://nx.dev) est un framework de build intelligent pour gérer les monorepos. Il est agnostique au langage, permettant à TypeScript, Python, l'infrastructure et plus encore de cohabiter et de se construire ensemble dans un seul workspace. Son système de build utilise le caching et l'analyse du graphe de dépendances pour ne reconstruire que ce qui a changé et exécuter les tâches en parallèle, gardant les tests, le linting et le build rapides à mesure que votre workspace grandit.

Chaque fonctionnalité du `@aws/nx-plugin` est fournie sous forme de [Générateur](https://nx.dev/features/generate-code) Nx — un scaffolding automatisé qui produit des structures de code cohérentes à partir de templates prédéfinis. Les générateurs peuvent être invoqués via la [CLI Nx](https://nx.dev/features/generate-code#invoke-generators), le [plugin IDE Nx Console](https://nx.dev/getting-started/editor-setup), ou par votre assistant IA via le <Link path="/get_started/building-with-ai">Nx Plugin for AWS MCP Server</Link>.

Parce que chaque composant est un générateur, vous n'ajoutez que ce dont vous avez besoin, quand vous en avez besoin. Commencez avec une API, ajoutez un site web plus tard, connectez-les quand vous êtes prêt — chaque étape est une seule commande.

## Construire avec des frameworks éprouvés

Le `@aws/nx-plugin` évite délibérément de construire des abstractions personnalisées par-dessus les outils que vous connaissez déjà. Au lieu d'inventer des frameworks, runtimes ou wrappers sur mesure, il génère du code idiomatique en utilisant des projets open-source établis et largement adoptés. Cela signifie que vos compétences existantes se transfèrent directement, vous pouvez vous appuyer sur la documentation officielle de chaque framework, et vous n'êtes jamais enfermé dans une couche que seul ce plugin comprend.

## Modifiable à volonté

Tout le code généré est _votre_ code. Le `@aws/nx-plugin` est un outil de build-time, pas une dépendance runtime — une fois qu'un générateur a été exécuté, le plugin n'est plus dans l'équation. Vous pouvez lire chaque fichier qu'il a produit, le commiter et le modifier comme bon vous semble.

Cela signifie :

- **Pas de trappes de sortie.** Il n'y a pas de surface de configuration propriétaire à apprendre ou à combattre lorsque vous devez faire quelque chose que le générateur n'a pas anticipé. Vous éditez le code généré directement, de la même manière que vous éditeriez n'importe quel code que vous avez écrit vous-même.
- **Pas de nouvelles couches d'abstraction.** Les générateurs émettent du code simple et idiomatique pour les frameworks sous-jacents. Il n'y a pas d'API wrapper ou d'indirection magique entre vous et React, tRPC, FastAPI, CDK ou n'importe quel autre outil — ce que vous voyez est ce qui s'exécute.
- **Le plugin n'est pas une dépendance.** Il génère du code et s'efface. Votre application n'importe pas et ne dépend pas de `@aws/nx-plugin` au runtime, donc vous n'êtes jamais bloqué par lui et jamais enfermé.

Si vous n'aimez pas quelque chose dans le code généré, vous êtes libre de le modifier. Les générateurs vous donnent une longueur d'avance, pas une cage.

## Dépendances minimales

Le `@aws/nx-plugin` s'efforce de maintenir le nombre de dépendances globales au minimum. Ce dont vous avez besoin pour commencer se résume aux générateurs que vous invoquez.

Par exemple, tout générateur basé sur TypeScript ne nécessitera que l'installation de [Node](https://nodejs.org/en/download). Pour les projets basés sur Python, [UV](https://docs.astral.sh/uv/) est la seule exigence.

## Sûreté de typage

Le `@aws/nx-plugin` utilise la sûreté de typage pour simplifier l'expérience développeur via l'autocomplétion IDE, tout en éliminant les erreurs runtime qui ne se manifesteraient autrement que dans une implémentation non type-safe. Ainsi, tous les composants fournis sont type-safe par défaut.

La sûreté de typage traverse les frontières des projets : lorsque vous connectez un site web à une API, le client généré partage les types avec le backend, de sorte qu'un changement dans un contrat d'API apparaît comme une erreur de compilation dans votre frontend plutôt qu'un bug en production. Refactorisez en toute confiance.
