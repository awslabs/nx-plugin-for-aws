Le PDK prend en charge l'infrastructure CDK écrite en Python et Java. Nous ne prenons pas en charge cela dans le plugin Nx pour AWS au moment de la rédaction de ce document.

L'approche recommandée serait soit de migrer votre infrastructure CDK vers TypeScript, soit d'utiliser nos générateurs et de migrer le package common constructs vers votre langage cible. Vous pouvez utiliser l'IA générative pour accélérer ce type de migrations, par exemple avec [Amazon Q CLI](https://aws.amazon.com/developer/learning/q-developer-cli/). Un agent d'IA peut itérer sur la migration jusqu'à ce que les modèles CloudFormation synthétisés soient identiques.

Le même principe s'applique à l'infrastructure générée par les API Type Safe en Python ou Java - vous pouvez traduire le construct générique `rest-api.ts` du package common constructs et implémenter votre propre générateur de métadonnées pour votre langage cible (reportez-vous à la section [APIs Modelled with OpenAPI](#apis-modelled-with-openapi)).

Vous pouvez utiliser le <Link path="/guides/python-project">générateur `py#project`</Link> pour un projet Python de base dans votre code CDK (et transférer votre fichier `cdk.json` en ajoutant les cibles pertinentes). Pour les projets Java, vous pouvez utiliser le plugin [`@nx/gradle`](https://nx.dev/technologies/java/introduction) de Nx, ou [`@jnxplus/nx-maven`](https://github.com/gridatek/jnxplus/tree/main/packages/nx-maven) pour Maven.
