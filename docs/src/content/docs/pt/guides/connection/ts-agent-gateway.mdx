O gerador `connection` pode conectar seu <Link path="guides/ts-agent">Agente TypeScript</Link> a um <Link path="guides/agentcore-gateway">AgentCore Gateway</Link>.

O gerador configura o agente para que ele se autentique no Gateway com IAM SigV4 quando implantado, e multiplexe diretamente para os servidores MCP anexados quando executado localmente.

## Pré-requisitos

Antes de usar este gerador, certifique-se de ter:

1. Um projeto TypeScript com um componente <Link path="guides/ts-agent">Agent</Link> (`infra: agentcore`)
2. Um projeto <Link path="guides/agentcore-gateway">`cedar#agentcore-gateway`</Link>

## Saída do Gerador

O gerador emite arquivos de cliente principais compartilhados em seu pacote `agent-connection`, além de um wrapper por Gateway, e modifica seu agente:

<FileTree>

- packages/common/agent-connection
  - src
    - core/
      - agentcore-gateway-mcp-client.ts Cliente MCP SigV4 para o Gateway implantado
      - agentcore-gateway-mcp-local-client.ts Cliente multiplex local que distribui para servidores MCP anexados
    - app/
      - \<gateway-kebab>-client.ts Wrapper por Gateway com lista `ATTACHED_MCP_SERVERS`
    - index.ts Re-exporta o cliente do Gateway

</FileTree>

Além disso, o gerador:

- Modifica o `agent.ts` do seu agente para importar a classe do cliente do Gateway, chamar `<Gateway>Client.create()` e registrar o cliente retornado no array `tools`
- Conecta o target `<agent>-serve-local` do agente para depender do agregador `<gateway>-serve-local` do Gateway
- Instala as dependências SigV4 / MCP necessárias

## Usando o Gateway conectado

O gerador transforma o `agent.ts` do seu agente para usar o cliente do Gateway:

```ts title="packages/example/src/my-agent/agent.ts" {2,5,8}
import { Agent } from '@strands-agents/sdk';
import { MyGatewayClient } from ':my-scope/agent-connection';

export const getAgent = async () => {
  const myGateway = await MyGatewayClient.create();
  return new Agent({
    systemPrompt: '...',
    tools: [myGateway],
  });
};
```

Quando implantado (`SERVE_LOCAL` não definido), `create()` retorna um único `McpClient` apontado para o endpoint MCP do Gateway e autenticado com SigV4. Quando `SERVE_LOCAL=true`, ele retorna um array de `McpClient`s — um por servidor MCP na lista `ATTACHED_MCP_SERVERS` do cliente. O `Agent.tools` do Strands aceita qualquer formato (ele achata transparentemente arrays de ferramentas aninhados), então o mesmo `agent.ts` funciona uniformemente em ambos os modos.

O ID da sessão é propagado automaticamente para os servidores MCP downstream: o transporte subjacente lê a sessão com escopo de solicitação através do [contexto de sessão TS do Strands](https://docs.aws.amazon.com/bedrock-agentcore/latest/devguide/observability.html) e o encaminha para o Gateway como o cabeçalho `X-Amzn-Bedrock-AgentCore-Runtime-Session-Id`.

## Infraestrutura

Após executar o gerador, você deve conceder ao agente permissão para invocar o Gateway.

<Infrastructure>
<Fragment slot="cdk">
```ts title="packages/infra/src/stacks/application-stack.ts" {5}
const gateway = new MyGateway(this, 'MyGateway');
const myAgent = new MyAgent(this, 'MyAgent');

// Grant the agent permissions to invoke the Gateway
gateway.grantInvokeAccess(myAgent);
```

A URL do Gateway é automaticamente registrada no namespace `agentcore.gateways.<ClassName>` da <Link path="guides/runtime-config">Configuração de Runtime</Link> pelo construto CDK gerado, para que o agente possa descobri-la em tempo de execução.

## Desenvolvimento Local

O gerador configura o target `serve-local` do agente para:

1. Iniciar todos os servidores MCP anexados ao Gateway conectado
2. Definir `SERVE_LOCAL=true` para que o cliente gerado distribua para esses servidores MCP locais em vez de chamar o Gateway implantado

Execute o agente localmente com:

<NxCommands commands={["<agent-name>-serve-local <project-name>"]} />

### `GATEWAY_URL_OVERRIDE`

Para executar o agente localmente **contra um Gateway implantado** (por exemplo, para exercitar políticas Cedar), defina:

```bash
export GATEWAY_URL_OVERRIDE=https://<gateway-id>.gateway.bedrock-agentcore.<region>.amazonaws.com/mcp
```

Quando definido, o cliente usa a URL do Gateway implantado e assina solicitações SigV4 com suas credenciais AWS locais em vez de recorrer à multiplexação local.

### Fidelidade local

O modo multiplex local ignora o próprio Gateway, então:

- **Sem avaliação de política Cedar.** Todas as ferramentas são visíveis para o agente, independentemente das políticas. Use `GATEWAY_URL_OVERRIDE` para exercitar políticas contra um Gateway implantado.
- **A prefixação de nomes de ferramentas é preservada.** As ferramentas de cada servidor MCP local são encapsuladas para expor nomes no formato `<target-name>___<tool-name>`, correspondendo ao que o Gateway implantado emite. Isso mantém o prompt do sistema de um agente e os nomes de ação Cedar que você referencia consistentes entre execuções locais e implantadas.
