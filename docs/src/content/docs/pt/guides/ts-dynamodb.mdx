Este gerador cria um novo projeto TypeScript apoiado pelo [Amazon DynamoDB](https://aws.amazon.com/dynamodb/), usando [ElectroDB](https://electrodb.dev/) para modelagem de entidades com segurança de tipos. Ele gera o código da aplicação e a infraestrutura necessária para provisionar e gerenciar uma tabela DynamoDB usando AWS CDK ou Terraform, com suporte a design de tabela única e desenvolvimento local integrado via DynamoDB Local.

## Desenvolvimento Local

### Iniciando o DynamoDB Local

O gerador configura um alvo `serve-local` que inicia uma instância do [DynamoDB Local](https://docs.aws.amazon.com/amazondynamodb/latest/developerguide/DynamoDBLocal.html) e cria a tabela:

<NxCommands commands={['run <project-name>:serve-local']} />

Isso automaticamente:
1. Baixa a imagem do DynamoDB Local (alvo `pull-image`)
2. Inicia um contêiner
3. Cria uma tabela local com índices pré-definidos

### Modelagem de Dados

O projeto gerado usa [ElectroDB](https://electrodb.dev/) para modelagem de entidades com segurança de tipos em uma única tabela DynamoDB. Adicione ou atualize arquivos de entidade em `src/entities/`, usando a entidade de exemplo gerada como ponto de partida.

Exemplo de definição de entidade:

```ts title="packages/my-table/src/entities/example.ts"
import { Entity } from 'electrodb';
import { getDynamoDBClient, resolveTableName } from '../client.js';

export const createExampleEntity = async () =>
  new Entity(
    {
      model: {
        entity: 'example',
        version: '1',
        service: 'MyTable',
      },
      attributes: {
        id: {
          type: 'string',
          required: true,
        },
        createdAt: {
          type: 'string',
          required: true,
          default: () => new Date().toISOString(),
          readOnly: true,
        },
        updatedAt: {
          type: 'string',
          required: true,
          default: () => new Date().toISOString(),
          watch: '*',
          set: () => new Date().toISOString(),
        },
      },
      indexes: {
        primary: {
          pk: {
            field: 'pk',
            composite: ['id'],
          },
          sk: {
            field: 'sk',
            composite: [],
          },
        },
      },
    },
    { client: getDynamoDBClient(), table: await resolveTableName() },
  );
```

Para mais detalhes, consulte a [documentação de entidades do ElectroDB](https://electrodb.dev/en/modeling/entities/).

### Usando o Cliente DynamoDB

O arquivo `src/client.ts` gerado exporta dois utilitários principais:

- `getDynamoDBClient()` — retorna um singleton `DynamoDBClient` em cache. Quando `SERVE_LOCAL=true`, conecta-se à instância local do DynamoDB Local; caso contrário, cria um cliente AWS usando a cadeia de credenciais padrão.
- `resolveTableName()` — retorna o nome da tabela DynamoDB. Quando `SERVE_LOCAL=true`, retorna a constante do nome da tabela local; caso contrário, busca o nome do AWS AppConfig usando a variável de ambiente `RUNTIME_CONFIG_APP_ID` e o armazena em cache para chamadas subsequentes.

### Parando o DynamoDB Local

Parar o `serve-local` (por exemplo, com `Ctrl+C`) remove automaticamente o contêiner do DynamoDB Local, mas preserva o volume nomeado para que seus dados persistam entre reinicializações.

:::caution[Windows]
Devido a limitações no tratamento de sinais no Windows, o contêiner não é removido automaticamente quando o `serve-local` é parado. Você precisará removê-lo manualmente:

```bash
<engine> rm -f <scope>-dynamodb
```

Substitua `<engine>` pelo seu mecanismo de contêiner (`docker` ou `finch`) e `<scope>` pelo escopo do seu workspace Nx (por exemplo, `proj`).
:::

## Adicionar/Remover Índices Secundários Globais

Comece atualizando `GLOBAL_SECONDARY_INDEXES` em `src/gsi.ts` — esta é a fonte de verdade para suas definições de GSI, e na próxima execução do `serve-local` o script adicionará ou removerá automaticamente índices na tabela local para refletir a nova lista. Ao usar CDK, essas alterações também são refletidas automaticamente em sua tabela AWS na próxima implantação.

<OptionFilter when={{ iacProvider: 'Terraform' }}>
  Diferentemente do CDK, as alterações em `src/gsi.ts` não são refletidas automaticamente em seu módulo Terraform. Você também deve atualizar manualmente seu módulo Terraform para adicionar ou remover as definições de GSI correspondentes antes de implantar.
</OptionFilter>

## Conectando à Tabela

Em qualquer projeto TypeScript, importe as fábricas de entidades do seu pacote DynamoDB e use-as diretamente:

```ts
import { createExampleEntity } from ':my-scope/my-table';

const entity = await createExampleEntity();
const result = await entity.query.primary({ id: '123' }).go();
```

:::note[Configuração de runtime]
Ao executar na AWS, `resolveTableName()` busca o nome da tabela do AWS AppConfig usando a variável de ambiente `RUNTIME_CONFIG_APP_ID`. Projetos construídos com este plugin (APIs tRPC, APIs Smithy, agentes, servidores MCP) já têm essa variável configurada automaticamente. Para outros projetos TypeScript, certifique-se de que `RUNTIME_CONFIG_APP_ID` esteja definida no ambiente de runtime com o ID da aplicação AppConfig provisionado pela sua infraestrutura.
:::

## Implantando sua Tabela

O gerador DynamoDB cria infraestrutura CDK ou Terraform com base no `iacProvider` selecionado.

<Infrastructure>
<Fragment slot="cdk">
O construto CDK é criado em `common/constructs`. Exemplo de uso:

```ts title="packages/infra/src/stacks/application-stack.ts"
import { MyTable } from ':my-scope/common-constructs';

export class ApplicationStack extends Stack {
  constructor(scope: Construct, id: string, props?: StackProps) {
    super(scope, id, props);

    const table = new MyTable(this, 'Table');
  }
}
```

Isso provisiona uma tabela DynamoDB com:
- `pk` (chave de partição) e `sk` (chave de ordenação), ambas do tipo `String`
- 2 Índices Secundários Globais por padrão, conforme definido em `src/gsi.ts`
- Cobrança sob demanda (`PAY_PER_REQUEST`)
- Criptografia KMS gerenciada pelo cliente com rotação automática de chave
- Recuperação point-in-time habilitada
- Proteção contra exclusão habilitada
- Nome da tabela registrado no Runtime Config sob o namespace `dynamodb` no AWS AppConfig
</Fragment>
<Fragment slot="terraform">


### Proteção contra Exclusão

A proteção contra exclusão está habilitada por padrão para evitar a exclusão acidental da tabela.

#### Desabilitar Proteção contra Exclusão

Desabilite-a para ambientes onde a exclusão da tabela é esperada, como stacks de desenvolvimento ou preview de curta duração.

<Infrastructure>
<Fragment slot="cdk">

```ts title="packages/infra/src/stacks/application-stack.ts"
import { MyTable } from ':my-scope/common-constructs';

const table = new MyTable(this, 'Table', {
  deletionProtection: false,
});
```
</Fragment>
<Fragment slot="terraform">

```hcl title="packages/infra/src/main.tf"
module "my_table" {
  source                      = "../../common/terraform/src/app/dynamodb/my-table"
  deletion_protection_enabled = false
}
```
</Fragment>
</Infrastructure>

### Modo de Cobrança

A tabela usa cobrança sob demanda (`PAY_PER_REQUEST`) por padrão. Mude para capacidade provisionada para cargas de trabalho previsíveis e de alto throughput.

<Infrastructure>
<Fragment slot="cdk">

```ts title="packages/infra/src/stacks/application-stack.ts"
import { BillingMode } from 'aws-cdk-lib/aws-dynamodb';
import { MyTable } from ':my-scope/common-constructs';

const table = new MyTable(this, 'Table', {
  billingMode: BillingMode.PROVISIONED,
  readCapacity: 5,
  writeCapacity: 5,
});
```
</Fragment>
<Fragment slot="terraform">

```hcl title="packages/infra/src/main.tf"
module "my_table" {
  source       = "../../common/terraform/src/app/dynamodb/my-table"
  billing_mode = "PROVISIONED"
}
```
</Fragment>
</Infrastructure>

A recuperação point-in-time está habilitada por padrão, permitindo que você restaure a tabela para qualquer ponto nos últimos 35 dias.
