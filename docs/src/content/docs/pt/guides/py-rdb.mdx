Este gerador cria um novo projeto de banco de dados relacional Python apoiado pelo [Amazon Aurora](https://aws.amazon.com/rds/aurora/) (PostgreSQL ou MySQL), [SQLModel](https://sqlmodel.tiangolo.com/) para modelagem de dados e [Alembic](https://alembic.sqlalchemy.org/) para migrações de esquema. Ele gera o código da aplicação e a infraestrutura necessária para provisionar e gerenciar um banco de dados usando AWS CDK ou Terraform, com definição de esquema declarativa, implantação automática de migração e um cliente de banco de dados.

## Desenvolvimento Local

### Modelagem de Dados

O projeto gerado usa [SQLModel](https://sqlmodel.tiangolo.com/) para definir o esquema do seu banco de dados. O fluxo de trabalho é model-first: adicione ou atualize classes de tabela SQLModel no diretório `<name>/models/` do seu projeto de banco de dados e, em seguida, gere uma migração a partir dessas alterações de modelo.

Exemplo de modelo:

```python title="packages/my_db/my_db/models/example.py"
from sqlalchemy import Column, String
from sqlmodel import Field, SQLModel


class ExampleModel(SQLModel, table=True):
    id: int | None = Field(default=None, primary_key=True)
    name: str = Field(sa_column=Column(String(255), nullable=False))
    description: str | None = Field(default=None, sa_column=Column(String(255), nullable=True))
```

Importe seus modelos em `<name>/models/__init__.py` para que o Alembic possa descobri-los durante a autogeração.

### Criando Migrações

Após adicionar ou atualizar modelos, use o Alembic para gerar e aplicar scripts de migração. O alvo `alembic` gerado inicia automaticamente um contêiner de banco de dados local antes de executar:

<NxCommands commands={['run <project>:alembic revision --autogenerate -m "describe your change"']} />

Isso gera um novo script de migração em `migrations/versions/`. Revise o script gerado antes de aplicá-lo.

Aplique a migração ao seu banco de dados local:

<NxCommands commands={['run <project>:migrate']} />

Quando você implantar a pilha AWS, a infraestrutura gerada aplica automaticamente as migrações geradas ao banco de dados implantado.

### Aplicando Migrações Existentes

Quando você baixar arquivos de migração criados por outros desenvolvedores, aplique-os ao seu banco de dados local:

<NxCommands commands={['run <project>:migrate']} />

### Executando Comandos Alembic

O alvo `alembic` gerado expõe a CLI do Alembic, para que você possa executar qualquer comando Alembic no banco de dados local. Consulte a [referência de comandos do Alembic](https://alembic.sqlalchemy.org/en/latest/ops.html) para comandos disponíveis.

<NxCommands commands={['run <project>:alembic <alembic-command>']} />

### Parando o Banco de Dados Local

Parar `dev` (por exemplo, com `Ctrl+C`) remove automaticamente o contêiner do banco de dados local, mas preserva o volume nomeado para que seus dados persistam entre reinicializações.

:::caution[Windows]
Devido a limitações no tratamento de sinais no Windows, o contêiner não é removido automaticamente quando `dev` é parado. Você precisará removê-lo manualmente:

```bash
<engine> rm -f <scope>-<db-name>
```

Substitua `<engine>` pelo seu motor de contêiner (`docker` ou `finch`), `<scope>` pelo escopo do seu workspace Nx (por exemplo, `proj`) e `<db-name>` pelo nome do seu projeto de banco de dados (por exemplo, `my-db`).
:::

## Conectando ao Banco de Dados

Importe `session_context` do seu pacote de banco de dados e use-o como um gerenciador de contexto para obter uma `Session` do SQLModel:

```python
from sqlmodel import select

from my_scope.my_db import session_context

with session_context() as session:
    results = session.exec(select(ExampleModel)).all()
```

O cliente de banco de dados automaticamente:
- Recupera a configuração do banco de dados do AWS AppConfig usando a variável de ambiente `RUNTIME_CONFIG_APP_ID`
- Gera tokens de autenticação temporários via `boto3` RDS Signer para autenticação IAM
- Gerencia conexões SSL/TLS com validação de certificado

### Requisitos SSL ao Conectar Sem RDS Proxy

Ao conectar diretamente ao cluster Aurora (sem RDS Proxy), passe o caminho do pacote CA do Amazon RDS usando o parâmetro opcional `rds_ca`:

```python
from my_scope.my_db import session_context

with session_context(rds_ca='/opt/global-bundle.pem') as session:
    ...
```

O valor `rds_ca` deve apontar para um pacote de certificado CA disponível no sistema de arquivos do seu tempo de execução. Para o Amazon RDS, use o pacote CA global de:

```text
https://truststore.pki.rds.amazonaws.com/global/global-bundle.pem
```

Como você disponibiliza esse arquivo depende do tempo de execução que se conecta ao banco de dados.

#### Imagens de Contêiner de Tempo de Execução

Se você preparar sua própria imagem de contêiner para o tempo de execução, adicione o pacote CA do RDS no seu Dockerfile e passe esse caminho para `session_context`:

```dockerfile
ADD --chmod=0644 https://truststore.pki.rds.amazonaws.com/global/global-bundle.pem /opt/global-bundle.pem
```

```python
with session_context(rds_ca='/opt/global-bundle.pem') as session:
    ...
```

#### Funções Lambda Compactadas

Se você implantar uma função Lambda compactada, anexe a camada Lambda `RdsCaLayer` (CDK) ou o módulo Terraform `ca-layer`. A camada CDK usa empacotamento Docker para adicionar o pacote CA, e a camada implantada o fornece em `/opt/global-bundle.pem`, que você pode passar para `session_context`:

```python
with session_context(rds_ca='/opt/global-bundle.pem') as session:
    ...
```
