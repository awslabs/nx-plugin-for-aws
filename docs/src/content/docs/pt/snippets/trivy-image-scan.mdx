A imagem Docker construída para este projeto pode ser verificada em busca de vulnerabilidades usando [Trivy](https://trivy.dev/), executando a partir da [imagem Trivy hospedada no ECR](https://gallery.ecr.aws/aquasecurity/trivy).

Um target `trivy` é adicionado ao seu projeto que verifica a imagem construída e **sai com código diferente de zero** se qualquer vulnerabilidade de severidade `HIGH` ou `CRITICAL` for encontrada. O `Dockerfile` gerado usa uma imagem base sem vulnerabilidades corrigíveis conhecidas dessas severidades no momento da geração, e atualiza ferramentas incluídas (como `npm`) para mantê-la assim.

A verificação usa o mesmo mecanismo de contêiner que a construção da sua imagem (`docker` ou `finch`), portanto, nenhuma ferramenta adicional é necessária. Como a verificação só é executada novamente quando a imagem muda, uma imagem inalterada não é verificada novamente. O script raiz `trivy` fornecido verifica todas as imagens no workspace:

:::tip[Execute o Trivy no CI]
A verificação intencionalmente não está conectada ao `build`, pois isso pode introduzir atrito desnecessário ao iterar durante o desenvolvimento, já que o banco de dados de vulnerabilidades do Trivy é continuamente atualizado com novos CVEs.

Em vez disso, recomendamos executar o comando acima como uma etapa dedicada no seu pipeline de CI antes da implantação em ambientes de produção.
:::

:::caution[Vulnerabilidades não corrigíveis são ignoradas]
A verificação é executada com `--ignore-unfixed`, então vulnerabilidades sem uma correção disponível não falham a construção, já que não podem ser corrigidas atualizando ferramentas no seu `Dockerfile` — elas não são relatadas na saída de verificação da construção. À medida que novas vulnerabilidades são divulgadas, o resultado das verificações pode mudar ao longo do tempo, então revise a imagem periodicamente executando `trivy image <sua-imagem>` sem a flag, e reconstrua contra uma imagem base mais recente (ou adicione etapas de contêiner para aplicar patches disponíveis) assim que uma correção for publicada.
:::

#### Suprimindo Descobertas do Trivy

Pode haver casos em que você deseja suprimir uma vulnerabilidade específica, por exemplo, quando nenhuma correção está disponível ainda e você avaliou o risco como aceitável.

Adicione o ID da vulnerabilidade (um por linha) ao arquivo `.trivyignore` na raiz do seu projeto (ou seja, ao lado do seu `project.json`):

```gitignore
# .trivyignore
# node-tar arbitrary file write - not exploitable in our usage
CVE-2024-XXXXX
```

Para mais detalhes sobre filtragem de descobertas, consulte a [documentação de filtragem do Trivy](https://trivy.dev/latest/docs/configuration/filtering/#by-finding-ids).
