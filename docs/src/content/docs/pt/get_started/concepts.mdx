O `@aws/nx-plugin` é um plugin do [Nx](https://nx.dev/) que fornece um conjunto de ferramentas para construir e implantar aplicações full-stack na AWS. Ele oferece uma coleção de [Geradores](#generators) que criam código de aplicação pronto para produção _e_ a infraestrutura para implantá-lo — type-safe, executável localmente e pronto para entrega.

Em vez de começar de uma página em branco, você (ou seu agente de IA) escolhe os componentes necessários — APIs, websites, autenticação, agentes de IA, infraestrutura — fornece algumas opções de configuração, e o plugin gera código inicial seguindo as melhores práticas. Ele até conecta projetos entre si (incluindo atualização de arquivos existentes via transformações AST) para produzir clientes type-safe entre seu frontend e backend.

## Construído sobre Nx e seus geradores

O [Nx](https://nx.dev) é um framework de build inteligente para gerenciar monorepos. Ele é agnóstico de linguagem, permitindo que TypeScript, Python, infraestrutura e mais coexistam e sejam construídos juntos em um único workspace. Seu sistema de build usa cache e análise de grafo de dependências para reconstruir apenas o que mudou e executar tarefas em paralelo, mantendo testes, linting e builds rápidos conforme seu workspace cresce.

Cada funcionalidade do `@aws/nx-plugin` é entregue como um [Gerador](https://nx.dev/features/generate-code) do Nx — scaffolding automatizado que produz estruturas de código consistentes a partir de templates predefinidos. Geradores podem ser invocados via [CLI do Nx](https://nx.dev/features/generate-code#invoke-generators), o [plugin IDE Nx Console](https://nx.dev/getting-started/editor-setup), ou pelo seu assistente de IA através do <Link path="/get_started/building-with-ai">Nx Plugin for AWS MCP Server</Link>.

Como cada componente é um gerador, você adiciona apenas o que precisa, quando precisa. Comece com uma API, adicione um website depois, conecte-os quando estiver pronto — cada passo é um único comando.

## Construa com frameworks comprovados

O `@aws/nx-plugin` deliberadamente evita construir abstrações customizadas sobre as ferramentas que você já conhece. Em vez de inventar frameworks, runtimes ou wrappers personalizados, ele cria código idiomático usando projetos open-source estabelecidos e amplamente adotados. Isso significa que suas habilidades existentes se transferem diretamente, você pode confiar na documentação oficial de cada framework, e você nunca fica preso em uma camada que apenas este plugin entende.

## Aberto para modificações

Todo código gerado é _seu_ código. O `@aws/nx-plugin` é uma ferramenta de build-time, não uma dependência de runtime — uma vez que um gerador foi executado, o plugin não está mais na jogada. Você pode ler cada arquivo que ele produziu, fazer commit e alterá-lo como quiser.

Isso significa:

- **Sem escape hatches.** Não há superfície de configuração proprietária para aprender ou lutar quando você precisa fazer algo que o gerador não antecipou. Você edita o código gerado diretamente, da mesma forma que editaria qualquer código que você mesmo escreveu.
- **Sem novas camadas de abstração.** Geradores emitem código simples e idiomático para os frameworks subjacentes. Não há API wrapper ou indireção mágica entre você e React, tRPC, FastAPI, CDK ou qualquer outra ferramenta — o que você vê é o que executa.
- **O plugin não é uma dependência.** Ele gera código e sai do caminho. Sua aplicação não importa ou depende de `@aws/nx-plugin` em runtime, então você nunca é bloqueado por ele e nunca fica preso.

Se você não gostar de algo no código gerado, você é livre para modificá-lo. Geradores dão a você um ponto de partida, não uma gaiola.

## Dependências mínimas

O `@aws/nx-plugin` busca manter o número de dependências globais no mínimo. O que você precisa para começar se resume a quais geradores você invoca.

Por exemplo, qualquer gerador baseado em TypeScript exigirá apenas [Node](https://nodejs.org/en/download) instalado. Para projetos baseados em Python, [UV](https://docs.astral.sh/uv/) é o único requisito.

## Type safety

O `@aws/nx-plugin` emprega type-safety para simplificar a experiência do desenvolvedor via autocompletar do IDE, ao mesmo tempo que elimina erros de runtime que de outra forma só surgiriam em uma implementação não type-safe. Como tal, todos os componentes fornecidos são type-safe por padrão.

Type safety flui através das fronteiras do projeto: quando você conecta um website a uma API, o cliente gerado compartilha tipos com o backend, então uma mudança em um contrato de API aparece como um erro de tempo de compilação no seu frontend em vez de um bug em produção. Refatore com confiança.
