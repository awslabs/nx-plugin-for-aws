## Tarefa 1: Executar tudo localmente

Inicie a pilha local completa — o servidor de desenvolvimento game-ui junto com uma Game API local e um AG-UI Story Agent local (que por sua vez inicializa o servidor Inventory MCP) — com um único comando:

<NxCommands commands={["serve-local game-ui"]} />

`game-ui:serve-local` tem `dependsOn` em `game-api:serve-local` e `dungeon_adventure.story:agent-serve-local`, então o Nx inicia o servidor local de cada projeto em paralelo. Através das conexões que configuramos no Módulo 1, elas por sua vez inicializam o servidor Inventory MCP e o DynamoDB Local. Certifique-se de que seu mecanismo de contêiner está em execução e então abra o servidor de desenvolvimento em um navegador.

<Aside type="tip" title="Sem login localmente">
No modo `serve-local`, o componente `CognitoAuth` pula o fluxo de login do Cognito (não há pool de usuários rodando localmente), então você chegará direto ao jogo sem fazer login. Você criará uma conta real quando implantarmos na AWS no final deste módulo.
</Aside>

<Aside type="caution" title="Manter o servidor de dev ativo">
Deixe o servidor de desenvolvimento em execução durante o restante deste módulo — ele recarrega automaticamente cada alteração abaixo.
</Aside>

## Tarefa 2: Onde o CopilotKit já está configurado

Quando você executou o gerador `connection` para `game-ui → story` no <Link path="get_started/tutorials/dungeon-game/1">Módulo 1</Link>, a integração AG-UI do site Shadcn foi gerada para você. Vale a pena dar uma olhada rápida:

## Tarefa 4: Criar as rotas do jogo

Precisamos de duas rotas — uma para escolher um herói, outra para jogar. Ambas usam componentes shadcn e o chat do CopilotKit; não há UI de chat feita manualmente.

<TabItem label="routes/index.tsx">

Este é o seletor de jogo: formulário de novo jogo (shadcn `Input` + `Button` + `Card`) mais uma lista "Continuar" alimentada por `useGameApi().games.query` com `useInfiniteQuery` — um `<div>` sentinela na parte inferior observado por um `IntersectionObserver` chama automaticamente `fetchNextPage()` quando rolado para a visualização, e spinners ao lado do cabeçalho e abaixo da lista exibem o estado de carregamento. Iniciar um jogo executa `saveGame` com o par `(playerName, genre)` (para que apareça na próxima vez) e navega para a rota de jogo.
</TabItem>
<TabItem label="routes/game/$playerName.tsx">
<E2ECode path="dungeon-adventure/4/routes/game/$playerName.tsx.template" lang="tsx" />

Esta é a rota de jogo. Ela constrói o `threadId` determinístico (``{player}-{genre}`` preenchido para 33 caracteres — o hook AG-UI o enviará literalmente como o id de sessão do AgentCore), renderiza `<CopilotChat agentId="agent" threadId={threadId} />` e sobrepõe o inventário de `useGameApi().inventory.query` por cima. Na montagem, `useGameApi().actions.query({ sessionId: threadId })` lê o histórico de conversação que o agente armazenou no S3 e — se houver algum — chama `agent.setMessages(...)` para reidratar o chat; caso contrário, envia uma mensagem de usuário de preparação para iniciar a história. `agent.messages` é assinado via `useAgent({ updates: [OnMessagesChanged] })` para que cada novo turno também refaça a consulta do inventário (as chamadas de ferramenta MCP modificam o DynamoDB diretamente).

<Aside type="tip" title="Aguarde `isFetching`, não `isLoading`">
O cache padrão do React Query retém o primeiro resultado por 5 minutos, então uma visita de retorno inicialmente vê a lista vazia obsoleta da primeira carga. Definimos `staleTime: 0` + `refetchOnMount: 'always'` e protegemos o `useEffect` de hidratação em `!isFetching && isSuccess` para que o caminho de re-preparação só dispare em um thread genuinamente vazio.
</Aside>

<Aside type="tip" title="threadIds determinísticos tornam /game/$playerName?genre=... compartilhável">
Revisitar a mesma URL — seja após um reload, um redeploy ou em outro dispositivo — resolve para o mesmo `threadId`, então o agente continua contando a mesma história.
</Aside>

<Aside type="tip" title="Parâmetros de caminho">
A sintaxe `$playerName` diz ao `@tanstack/react-router` para tratar `playerName` como um [parâmetro de caminho](https://tanstack.com/router/v1/docs/framework/react/guide/path-params). `validateSearch` torna `genre` fortemente tipado contra nosso enum.
</Aside>
</TabItem>

## Tarefa 5: Implantar na AWS

Seu jogo está completo e você testou cada parte localmente. Agora vamos implantá-lo na AWS para que você possa jogá-lo de qualquer lugar.

## Tarefa 6: Misturando componentes locais e implantados

Você viu os dois extremos do espectro: tudo local (`serve-local`) e tudo implantado. Durante o desenvolvimento do dia a dia, muitas vezes é útil misturar os dois — por exemplo, iterar no código do site contra a API e agente *reais* implantados, ou executar a API localmente contra a tabela DynamoDB *real*.

A chave é a variável de ambiente `RUNTIME_CONFIG_APP_ID`. Quando um projeto é executado **sem** `SERVE_LOCAL=true`, o `resolveTableName()` gerado (e as buscas de configuração de runtime do agente / API) buscam sua configuração do AWS AppConfig usando este id de aplicação — o valor `RuntimeConfigApplicationId` das suas saídas do CDK.

Cada site tem um target `load:runtime-config` que baixa o `runtime-config.json` implantado (pool do Cognito, endpoints da API, ARN do agente) para o servidor de desenvolvimento local. Algumas combinações úteis:

- **Site local → backend implantado.** Baixe a configuração implantada uma vez e então execute o target `serve` simples (não `serve-local`) para que a UI se comunique com a API e agente implantados:

  <NxCommands commands={["run game-ui:load:runtime-config"]} />
  <NxCommands commands={["serve game-ui"]} />

- **API local → tabela DynamoDB real.** Execute o target `serve` simples com `RUNTIME_CONFIG_APP_ID` definido para o id de aplicação implantado, para que `resolveTableName()` resolva a tabela implantada em vez do DynamoDB Local:

  <NxCommands env={{RUNTIME_CONFIG_APP_ID:"<RuntimeConfigApplicationId from CDK outputs>"}} commands={["serve game-api"]} />

Essa flexibilidade permite que você teste exatamente a parte em que está trabalhando, contra quaisquer componentes — locais ou implantados — que façam sentido.
