## Tarefa 1: Implementar a API do Jogo

Vamos implementar as seguintes APIs nesta seção:

1. `saveGame` - criar ou atualizar um jogo.
2. `queryGames` - retornar uma lista paginada de jogos salvos anteriormente.
3. `queryInventory` - retornar uma lista paginada de itens no inventário de um jogador.
4. `queryActions` - retornar o histórico de conversação para um determinado jogo.

:::note[Onde cada parte do estado reside]
- `games` e `inventory` estão no **DynamoDB** (via ElectroDB) — as lambdas da Game API e o servidor MCP de Inventário os leem/escrevem.
- `actions` (transcrição do chat) está no **S3** — o `S3SessionManager` do Story Agent escreve um blob JSON por turno, e `queryActions` na Game API os lê de volta para que a UI possa reproduzir o histórico ao recarregar a página. Não há procedimento `saveAction`; o agente é o único escritor.
:::

### Modelagem de entidades

Este é o diagrama ER para nossa aplicação.

```d2
direction: right

game: Game {
  shape: sql_table
  playerName: "string  PK"
  genre: string
  lastUpdated: string
}

item: Item {
  shape: sql_table
  playerName: "string  PK"
  itemName: "string  SK"
  emoji: "string  (optional)"
  lastUpdated: string
  quantity: number
}

game -> item: 1..*
```

Vamos modelar nossos dados com [ElectroDB](https://electrodb.dev/en/core-concepts/introduction/) sobre a tabela `DungeonDb` que criamos com o gerador `ts#dynamodb` no <Link path="get_started/tutorials/dungeon-game/1">Módulo 1</Link>. Esse gerador já adicionou `electrodb` e `@aws-sdk/client-dynamodb` ao workspace, então a única dependência extra que precisamos é o cliente S3 usado pelo procedimento `queryActions` da Game API para ler os turnos de conversação que o agente escreveu:

Observe que as entidades usam os helpers `getDynamoDBClient()` e `resolveTableName()` do `client.ts` gerado do projeto — isso é o que permite que o mesmo código de entidade se comunique com o DynamoDB Local durante o desenvolvimento e com a tabela implantada na AWS.

:::note[Procedimentos isolados]
Observe que como cada procedimento é atendido por uma função lambda individual, podemos seguir o princípio de menor privilégio e atribuir apenas as permissões de leitura/escrita necessárias com base na implementação do procedimento.

:::tip[Configuração de Runtime]
O construto `DungeonDb` registra o nome da tabela implantada na <Link path="guides/runtime-config">Configuração de Runtime</Link> sob o namespace `dynamodb`, armazenado no AWS AppConfig. O helper `resolveTableName()` gerado o recupera em tempo de execução usando Lambda Powertools, em vez de precisar ser passado como uma variável de ambiente. Um benefício dessa abordagem é que, à medida que o número de tabelas ou outros identificadores de infraestrutura cresce, você precisa apenas de uma única variável de ambiente `RUNTIME_CONFIG_APP_ID` ao executar contra a infraestrutura implantada.
:::

## Tarefa 4: Testar a Game API localmente

Não há necessidade de implantar na AWS para testar nossa API — o target `serve-local` executa a Game API contra o [DynamoDB Local](https://docs.aws.amazon.com/amazondynamodb/latest/developerguide/DynamoDBLocal.html). Como conectamos a Game API ao projeto `DungeonDb` no Módulo 1, este target também inicia o DynamoDB Local automaticamente.

Primeiro, corrija quaisquer problemas de lint:

<PackageManagerShortCommand commands={["lint"]} />

Depois construa a base de código:

<PackageManagerShortCommand commands={["build"]} />

<Aside type="note" title="Workspace fora de sincronia">
Adicionar procedimentos e as novas referências de projeto significa que o Nx pode novamente avisar que "O workspace está fora de sincronia" (como no <Link path="get_started/tutorials/dungeon-game/1">Módulo 1</Link>). Selecione **Sim, sincronizar as alterações e executar as tarefas** para continuar. Você verá este aviso sempre que as referências de projeto mudarem.
</Aside>

### Iniciar o servidor local

Inicie a Game API localmente com o target `serve-local`. Certifique-se de que seu mecanismo de contêiner (Docker ou Finch) esteja em execução primeiro, pois isso também inicializa o DynamoDB Local:

<NxCommands commands={["serve-local game-api"]} />

O target `serve-local` define `SERVE_LOCAL=true`, então `resolveTableName()` retorna o nome da tabela local e `getDynamoDBClient()` se conecta ao DynamoDB Local em vez da AWS — sem implantação e sem `RUNTIME_CONFIG_APP_ID` necessário.

<Aside type="tip" title="Primeira execução">
A primeira execução baixa a imagem de contêiner do DynamoDB Local, o que pode levar um momento. Deixe o servidor em execução neste terminal e abra um segundo terminal para os comandos `curl` abaixo.
</Aside>

### Testar a API

Assim que seu servidor estiver em execução, consulte a lista (vazia) de jogos:

```bash
curl -X GET 'http://localhost:2022/games.query?input=%7B%7D'
```

:::note[Codificação de URL]
O `%7B%7D` que passamos para testar a API é um objeto JSON vazio codificado em URI (`{}`).
:::

Você verá uma lista vazia:

```json
{"result":{"data":{"items":[],"cursor":null}}}
```

Agora salve um jogo:

```bash
curl -X POST 'http://localhost:2022/games.save' \
  -H 'Content-Type: application/json' \
  -d '{"playerName":"Alice","genre":"zombie"}'
```

O save retorna o jogo persistido (com o timestamp `lastUpdated` que a entidade define para você):

```json
{"result":{"data":{"playerName":"Alice","genre":"zombie","lastUpdated":"..."}}}
```

Consulte novamente para confirmar que está persistido no DynamoDB Local:

```bash
curl -X GET 'http://localhost:2022/games.query?input=%7B%7D'
```

Esta resposta agora inclui o jogo salvo:

```json
{"result":{"data":{"items":[{"playerName":"Alice","genre":"zombie","lastUpdated":"..."}],"cursor":null}}}
```

Você pode parar o servidor local (`Ctrl+C`) quando terminar.

## Tarefa 5: Testar o servidor MCP de Inventário localmente

Podemos testar as ferramentas do servidor MCP com o [MCP Inspector](https://github.com/modelcontextprotocol/inspector) usando o target `mcp-server-inspect` gerado. O inspector se conecta ao servidor via STDIO, então executamos o DynamoDB Local separadamente e dizemos ao servidor para usá-lo.

Em um terminal, inicie o DynamoDB Local:

<NxCommands commands={["serve-local dungeon-db"]} />

Em um segundo terminal, inicie o inspector:

<NxCommands commands={["mcp-server-inspect inventory"]} />

Isso inicia o MCP Inspector em `http://localhost:6274`. Na interface do inspector, antes de conectar, adicione uma variável de ambiente `SERVE_LOCAL` com o valor `true` para que as ferramentas do servidor se comuniquem com o DynamoDB Local, depois clique em **Connect**.

<Aside type="tip" title="Por que definir SERVE_LOCAL?">
Diferente do `serve-local`, o target `mcp-server-inspect` executa o servidor STDIO simples, então ele não define `SERVE_LOCAL` para você. Defini-lo no ambiente do inspector faz com que `getDynamoDBClient()` e `resolveTableName()` se conectem à tabela local em vez da AWS.
</Aside>

Mude para a aba **Tools**, clique em **List Tools**, e experimente `add-to-inventory` (por exemplo, `playerName: Alice`, `itemName: Rusty Sword`, `emoji: ⚔️`) seguido de `list-inventory-items` para vê-lo persistido no DynamoDB Local. Pare ambos os terminais (`Ctrl+C`) quando terminar.

<Aside type="tip" title="Prefere o transporte HTTP?">
Se você preferir controlar o servidor via HTTP (por exemplo, para scripts), o target `mcp-server-serve-local` executa o transporte HTTP Streamable com `SERVE_LOCAL=true` definido para você, e inicia o DynamoDB Local automaticamente:

<NxCommands commands={["run @dungeon-adventure/inventory:mcp-server-serve-local"]} />

Então inicialize uma sessão contra `http://localhost:8000/mcp`:
```bash
curl -X POST http://localhost:8000/mcp \
  -H 'Content-Type: application/json' \
  -H 'Accept: application/json, text/event-stream' \
  -d '{"jsonrpc":"2.0","id":1,"method":"initialize","params":{"protocolVersion":"2025-03-26","capabilities":{},"clientInfo":{"name":"curl","version":"1.0.0"}}}'
```
</Aside>

Parabéns, você construiu e testou sua primeira API tRPC e servidor MCP contra uma tabela DynamoDB local!  🎉🎉🎉
